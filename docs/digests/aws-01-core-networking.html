Core networking: VPCs, subnets, route tables, and how AWS decides who gets to the internet

The topic that finally made "public subnet" stop being a vague AWS buzzword for me.

In one line

Every EC2 instance (and most of the compute you run) on AWS lives inside a VPC you design yourself, and whether any piece of it can be reached from (or reach out to) the internet comes down to one thing: what a route table says.

The big idea

A VPC (Virtual Private Cloud) is your own private slice of AWS network space, and inside it you carve out subnets to hold your servers, databases, and everything else. The part that tripped me up at first is that a subnet isn't "public" or "private" because you flip a setting on it. It's public because its route table happens to contain one specific rule that points general internet traffic at an internet gateway. Change that one rule and the exact same subnet, with the exact same servers in it, becomes private. Once that clicks, an internet gateway and a NAT gateway stop being mysterious AWS jargon and just become two different answers to "where does this route rule point."

Key terms

  • VPC (Virtual Private Cloud) — your own logically isolated network inside AWS that you define. Think of it as an empty office floor before you've built any internal walls.
  • CIDR block — the range of IP addresses assigned to a VPC or subnet, written like 10.0.0.0/16. The number after the slash is the size of the range; a smaller number means more addresses.
  • Subnet — a smaller range of IP addresses carved out of a VPC's CIDR block. A subnet always sits entirely inside one Availability Zone; it can't span zones.
  • Availability Zone (AZ) — one of several physically separate data center clusters inside an AWS Region.
  • Route table — the set of rules attached to a subnet (or a gateway) that decides where outbound traffic from that subnet is allowed to go.
  • Route — one rule inside a route table: a destination (which IP addresses this rule matches) paired with a target (where matching traffic gets sent).
  • Local route — the automatic route every route table gets for free, letting resources inside the VPC talk to each other without you writing anything.
  • Internet gateway (IGW) — the AWS component you attach to a VPC to give it a doorway to the public internet.
  • NAT gateway — a managed AWS service that lets resources without a public IP address start connections out to the internet, while stopping anything on the internet from starting a connection back in.
  • Elastic IP — a static public IPv4 address you allocate and attach to a resource, for example a NAT gateway.
  • Public IP / private IP — a public IP is reachable from the internet; a private IP only works inside the VPC (or a network you've explicitly connected it to).

How it works

Here's the grounding piece: AWS runs an enormous shared pool of physical servers and network hardware underneath everyone's accounts. A VPC is the logical fence AWS draws around a slice of that shared infrastructure so it behaves, from your point of view, like your own private network with your own address scheme, even though physically it's sitting on the same racks as thousands of other customers. Once you accept that a VPC is just "your own address space," almost every other networking decision in AWS reduces to two questions for any resource you launch: which private network is this thing part of, and can it reach (or be reached from) the internet? The rest of this section is really just answering those two questions in more and more detail.

VPCs and CIDR blocks

When you create a VPC, the first thing you choose is its IPv4 CIDR block, something like 10.0.0.0/16. That block is the total pool of private IP addresses available inside that VPC. AWS also gives every account a default VPC in each Region so you can start launching servers immediately without designing anything yourself. According to AWS's own documentation, a default VPC is set up with a /16 block of 172.31.0.0/16, which provides up to 65,536 private IPv4 addresses, plus a /20 subnet in each Availability Zone (up to 4,096 addresses each, a few of which AWS reserves for its own use), an internet gateway already attached, and a route already in place sending general internet traffic to it.

Simplification: I'm not going deep on CIDR math here. For a beginner, just treat the number after the slash as roughly "how many addresses you get": a smaller number after the slash means a bigger range. Real subnetting has more rules than that, but this gets you far enough to read a VPC diagram.

The default VPC is genuinely convenient for getting started or for a quick public website, but it's also a shared starting template, not something you should build a real production system inside without understanding what's already configured in it (a public internet gateway route reaching every subnet, for one).

Subnets, and why "public" and "private" aren't a checkbox

A subnet is a smaller range of addresses carved out of the VPC's CIDR block, and it always lives entirely inside one Availability Zone. This is where the idea that finally made this topic click for me lives: AWS documentation is explicit that the subnet's "type" (public, private, and a couple of others) is determined entirely by how you've configured routing for it, not by any property you set on the subnet itself. A public subnet is one whose route table has a direct route to an internet gateway. A private subnet is one whose route table doesn't have that route (it needs a NAT device to reach the internet at all). AWS also documents a couple of other subnet types you'll see less often: a VPN-only subnet, which only routes out through a Site-to-Site VPN connection, and an isolated subnet, which has no route to anything outside the VPC at all.

Here's the detail that really drove the point home for me. AWS's own internet gateway documentation walks through an example with two subnets, one in each of two Availability Zones. Both subnets have instances with public IP addresses. But only one of the two route tables actually points internet-bound traffic at the internet gateway. The subnet whose route table is missing that route is private, and its instances can't reach the internet even though they have public IP addresses, because nothing routes their traffic there. Having a public IP is necessary but not sufficient. The route table is what actually decides it.

Every subnet you create is automatically associated with the VPC's main route table until you explicitly change that association, and you can always swap which route table a subnet uses.

Route tables: the core mental model

A route table is a list of rules. Each rule (a "route") is a destination, a range of IP addresses, paired with a target, the gateway or device that traffic to that destination should be sent through. Every VPC gets a main route table automatically, and you can create additional custom route tables for more granular control, then associate specific subnets with them.

Every route table also gets a local route for free: a rule that lets anything inside the VPC's own CIDR block reach anything else inside it, without you ever writing that rule yourself. You'll never see a subnet that can't talk to other things in the same VPC unless someone has deliberately removed that route (which AWS allows, but it isn't the default).

When more than one route in a table could match the same destination, AWS uses what it calls the longest prefix match: it picks the most specific route available. For example, if a table has both a route for 172.31.0.0/16 and a broader catch-all route for 0.0.0.0/0, traffic headed into that 172.31.0.0/16 range uses the more specific route, and everything else falls through to the catch-all.

Internet gateways: the doorway

An internet gateway is a VPC component you attach to a VPC to let it talk to the internet. AWS describes it as horizontally scaled, redundant, and highly available, and states it doesn't introduce availability risk or bandwidth constraints on your traffic. There's no charge for the internet gateway itself, though normal EC2 data transfer charges still apply to traffic passing through it.

Attaching an internet gateway to a VPC isn't enough on its own. You also need a route in a subnet's route table pointing at it, usually 0.0.0.0/0 (meaning "everything not covered by a more specific route"), though AWS also allows scoping that route to a narrower range instead of the whole internet. That one route is the entire difference between a public and a private subnet.

One detail I hadn't appreciated: for IPv4 traffic, the internet gateway itself is quietly doing a one-to-one network address translation on your behalf. Your instance only ever knows its own private IP address; the internet gateway swaps in the instance's public IPv4 or Elastic IP address on the way out, and swaps the destination back to the private address on the way in. You never configure this, it's just part of what "attach an internet gateway and add the route" gets you.

NAT gateways: how a private subnet still reaches out

A NAT gateway solves a specific problem: you want instances in a private subnet to be able to start outbound connections (download a security patch, call an external API) without ever being reachable from the internet themselves. AWS's default NAT gateway type is a public NAT gateway: you create it inside a public subnet, you must attach an Elastic IP address to it, and you route it to the internet gateway for that VPC. There's also a private NAT gateway type for routing to other VPCs or on-premises networks through a transit gateway or virtual private gateway instead of the internet; it can't have an Elastic IP attached, and traffic routed to an internet gateway through a private NAT gateway gets dropped.

So the chain for a private subnet's outbound internet access looks like this: the private subnet's route table sends 0.0.0.0/0 to the NAT gateway. The NAT gateway itself lives inside a public subnet, and that public subnet's own route table sends 0.0.0.0/0 to the internet gateway. Two hops, two route tables, same underlying idea both times.

A few concrete numbers AWS documents about NAT gateways, useful for sanity-checking whether one is enough for your workload: each NAT gateway supports 5 Gbps of bandwidth and automatically scales up to 100 Gbps, and it can process 1 million packets per second, scaling up to 10 million (beyond that it starts dropping packets). Each IPv4 address on a NAT gateway supports up to 55,000 simultaneous connections to any single destination (identified by destination IP, port, and protocol); you can raise that ceiling by attaching more addresses (up to 8 total: 1 primary plus 7 secondary), and a public NAT gateway is limited to 2 associated Elastic IP addresses by default, which is a quota you can request to have increased. NAT gateways also only support TCP, UDP, and ICMP traffic, and use ports 1024 to 65535.

One operational catch worth knowing early: a NAT gateway lives in one specific Availability Zone. If resources across multiple AZs share a single NAT gateway and that AZ has a problem, everything relying on it loses internet access, even though their own AZ is fine. AWS's own guidance for resilience is to run a separate NAT gateway per AZ and route each AZ's private subnets to the one in its own zone.

Why it matters

This is the layer everything else in AWS sits on top of. A typical pattern you'll see constantly: load balancers and bastion hosts sit in public subnets because they need to be reachable, while application servers and databases sit in private subnets so nothing on the open internet can connect to them directly, they only get outbound access (for patches, API calls, and so on) through a NAT gateway. Getting this wrong in either direction is a real security and reliability issue, not just a style preference: an accidentally public database subnet is a genuine exposure, and a private subnet with no NAT path at all just means your servers silently can't reach anything outside the VPC.

It's also worth remembering that AWS's own infrastructure runs on the same kind of networking layer underneath, at a much bigger scale. On December 7, 2021, AWS had a major multi-hour outage in its US-EAST-1 Region, caused (per AWS's own public post-event summary) by an automated scaling activity that triggered unexpected behavior in internal network clients, which overwhelmed the networking devices connecting AWS's internal network to its main network. It's a different layer than a customer VPC's route tables, but it's a good reminder that "boring" networking plumbing, not the application code on top of it, is often exactly where things actually break.

Remember this

  • A subnet is public or private because of its route table, not because of a setting you flip on the subnet itself.
  • The one line that matters: destination 0.0.0.0/0, target an internet gateway.
  • Local routes let everything inside a VPC's CIDR block talk to each other automatically; you never write that route yourself.
  • A NAT gateway lives in a public subnet and lets private resources call out, without letting anything call in.
  • When more than one route could match, AWS uses the most specific one (longest prefix match).

Try it yourself

  1. Build the two-subnet pattern by hand. In the AWS console (or via CLI/CloudFormation if you're comfortable), create a VPC with one public and one private subnet. Before you launch anything, write down what you expect each route table to contain, then check it against reality.
  2. Prove the public path. Launch a small EC2 instance in the public subnet with a public IP and connect to it directly. Find the exact route in its subnet's route table that made that possible.
  3. Prove the private path. Launch an instance in the private subnet with no public IP, add a NAT gateway in the public subnet, and update the private subnet's route table to send 0.0.0.0/0 to it. Confirm the private instance can reach out (run a package update) while nothing from the internet can reach it.
  4. Break it on purpose. Remove the 0.0.0.0/0 route from the public subnet's route table and watch your "public" instance lose internet access, even though you didn't touch the instance itself. This is the whole lesson in one experiment.
  5. Map your own home network. Sketch your home Wi-Fi as if it were a VPC: your router doing NAT is roughly playing the role of a NAT gateway, and your ISP connection is roughly your internet gateway. Label what your "local route" would be.
  6. Re-associate a subnet. Create a second, custom route table in your practice VPC, then re-associate just one subnet with it instead of the main route table. Trace exactly how that changes what that subnet can reach.
