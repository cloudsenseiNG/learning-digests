Core networking: VPCs, subnets, route tables, and NAT gateways

In one line: the one route table entry that decides whether a subnet can reach the internet, and the two gateways involved in getting traffic in and out of your own private slice of AWS.

The big idea

Every AWS resource that needs to talk over a network has to live inside one, and a VPC (Virtual Private Cloud) is that network: a slice of AWS's infrastructure that's logically isolated and that you get to define yourself. You pick the address range, split it into subnets that live in different physical locations, and then decide, subnet by subnet, whether it can reach the public internet. That last decision isn't a checkbox anywhere. It comes down to a single line in a routing table.

Key terms

  • VPC — a logically isolated virtual network you define inside AWS. AWS's own description: it "closely resembles a traditional network that you'd operate in your own data center."
  • CIDR block — a compact way of writing a range of IP addresses, like 10.0.0.0/16. It's the address space your VPC (or subnet) owns.
  • Availability Zone (AZ) — one of several physically separate, isolated locations inside an AWS Region. A VPC spans every AZ in its Region; a subnet does not.
  • Subnet — a smaller range of IP addresses carved out of a VPC's CIDR block. A subnet has to live entirely inside one Availability Zone.
  • Route table — a set of rules ("routes") attached to a subnet. Each route says: traffic headed to this destination should go to this target.
  • Internet gateway (IGW) — the component you attach to a VPC so it can talk to the public internet at all.
  • NAT gateway — lets a resource that has no public IP start outbound connections to the internet, without letting anything on the internet start a connection back in.
  • Public subnet / private subnet — not a property you set on the subnet itself. It's a label that describes what the subnet's route table points at.

How it works

A VPC is your own address space

When you create a VPC, the first thing you give it is a CIDR block, something like 10.0.0.0/16. That's the entire pool of private IP addresses your VPC owns. Every AWS account also gets a default VPC in each Region, already wired up with an internet gateway and public subnets, which is why you can launch an EC2 instance and reach it on day one without touching any networking config. A custom VPC starts empty: just an address range, waiting for you to add subnets, route tables, and gateways.

Simplification: CIDR math (why /16 means a bigger range than /24) is general networking convention, not something specific to AWS, so I'm not walking through the bit arithmetic here. AWS's docs page for creating a VPC didn't spell out the exact smallest/largest block size it allows when I checked it, so if that number matters for what you're building, check the current console or docs rather than assuming.

Subnets and why they're tied to one Availability Zone

A subnet is a slice of the VPC's CIDR block, and AWS is strict about one thing: "each subnet must reside entirely within one Availability Zone and cannot span zones." That's not an arbitrary rule, it's what makes multi-AZ resilience possible. If you put resources in subnets across two or three AZs, the failure of one AZ (a whole physical location going down) doesn't take your whole app with it. AWS's own guidance is to spread production subnets across at least two AZs for exactly this reason.

Route tables: the traffic controller

AWS describes a route table plainly: it "serves as the traffic controller for your VPC." Every route table holds a list of routes, and each route is just a pair: a destination (a CIDR block, like 0.0.0.0/0 for "anywhere") and a target (where to send traffic headed there, like an internet gateway, a NAT gateway, or a VPC peering connection). Every subnet is associated with a route table (if you don't pick one, it uses the VPC's main route table), and that table's rows are the only thing that decides where a packet leaving the subnet actually goes.

Simplification: AWS's docs also mention resources inside a VPC being able to reach each other automatically, which usually shows up as an implicit route covering the VPC's own address range. I'm not detailing exactly how that's represented here since I didn't confirm the specifics on the pages I read; look at a real route table in the console to see it for yourself.

The internet gateway: the door to the internet

An internet gateway is, in AWS's words, "a horizontally scaled, redundant, and highly available VPC component that allows communication between your VPC and the internet." On its own it does nothing for a subnet. You have to attach it to the VPC and add a route in a subnet's route table that points at it (usually 0.0.0.0/0 → igw-xxxxxxxx for "send anything not otherwise matched to the internet gateway"). A default VPC already has an internet gateway attached and that route in place, which is why default VPCs "just work." A VPC you create yourself starts without either.

What actually makes a subnet "public"

This is the part that took me a minute to really get: AWS states it directly. "If a subnet is associated with a route table that has a route to an internet gateway, it's known as a public subnet. If a subnet is associated with a route table that does not have a route to an internet gateway, it's known as a private subnet." That's it. There's no separate "make this subnet public" switch. It's entirely about whether the route table attached to that subnet has a row pointing at an internet gateway.

One detail that surprised me: AWS explicitly calls out that an instance in a private subnet can't reach the internet "even if they have public IP addresses," because there's still no route to the internet gateway. Having a public IP and having a route to the internet gateway are two separate things, and you need both.

NAT gateways: outbound-only for private subnets

So how does something in a private subnet get software updates or call an external API if it has no route to the internet gateway? That's the job of a NAT gateway. AWS's definition: "You can use a NAT gateway so that instances in a private subnet can connect to services outside your VPC but external services can't initiate a connection with those instances." Traffic only flows one way: out, and back as a reply. Nothing on the internet can open a fresh connection to that private instance through it.

A public NAT gateway has to live in a public subnet (the docs are explicit: "You create a public NAT gateway in a public subnet and must associate an Elastic IP address with the NAT gateway at creation"), and traffic reaches it from the private subnet, then the NAT gateway forwards it to the internet gateway for the actual trip out. The private subnet's route table gets its own 0.0.0.0/0 row, but pointed at the NAT gateway, not the IGW, which is exactly why it stays classified as private.

Why it matters

This is the layer everything else in AWS sits on top of. Whether you're launching an EC2 instance, pointing a Lambda function at a VPC, or standing up a database, one of the first questions is which subnet it lands in and what that subnet can reach. The common real-world pattern (a load balancer or bastion host in a public subnet, application servers and databases in private subnets behind a NAT gateway) exists because of exactly the mechanics in this digest: route tables deciding what's reachable, and a NAT gateway giving private resources a controlled way out without opening a way in. Once this model clicks, the next topics in this track (Transit Gateway for connecting multiple VPCs, then EC2 placement) build directly on it.

Remember this

  • A VPC is your own isolated slice of AWS network space, defined by a CIDR block (a compact way of writing an IP address range).
  • A subnet lives inside exactly one Availability Zone and is carved out of the VPC's CIDR block.
  • "Public" and "private" aren't settings on the subnet. They're just a description of what that subnet's route table points its 0.0.0.0/0 row at.
  • An internet gateway has to be attached to the VPC and referenced by a route before any subnet can use it.
  • A NAT gateway lives in a public subnet and lets private-subnet resources start outbound connections, without accepting unsolicited inbound ones.

Try it yourself

  • Build the standard layout in the console. In the VPC console, choose "Create VPC" → "VPC and more," pick 1 Availability Zone, 1 public subnet, 1 private subnet, and 0 NAT gateways to start. Note the CIDR blocks AWS assigns automatically, and open the resource map to see how the pieces connect.
  • Rebuild it with the AWS CLI. Follow the documented sequence: aws ec2 create-vpc --cidr-block 10.0.0.0/24, then two aws ec2 create-subnet calls in different Availability Zones. Compare the subnet IDs and CIDR blocks you get back to what the console gave you.
  • Draw a route table by hand first. Before opening the console's Route Tables tab, sketch what you expect the public and private route tables to contain (destination + target rows). Then check your guess against the real thing.
  • Prove a private subnet can't reach out. Launch a small instance in your private subnet (no NAT gateway yet) and try to reach the internet from it. Confirm it fails, and connect that back to the route table you sketched.
  • Add the NAT gateway and watch it flip. Create a NAT gateway in the public subnet (console, or aws ec2 create-nat-gateway --subnet-id ... --allocation-id ... after allocating an Elastic IP), add the 0.0.0.0/0 → nat-... route to the private route table, and retry the same outbound test.
  • Break it back on purpose. Remove the internet gateway route from your public subnet's route table and confirm resources there lose internet access too, even with public IPs assigned. This is the cleanest proof that "public" is a route table property, not a fixed label.
