Core Networking: VPCs, Subnets, Route Tables & NAT

The mental model that makes the rest of AWS networking click: nothing is "public" or "private" by nature, it's just where a route table sends the traffic.

In one line

Every AWS resource you launch lives inside a VPC, a private network you carve into subnets, and whether a subnet can reach the internet comes down to a single row in its route table, not some special setting on the subnet itself.

The big idea

A VPC (Virtual Private Cloud) is a network you define for yourself inside an AWS region. You slice it into subnets, and each subnet lives entirely within one Availability Zone (a data center, more or less). What makes a subnet "public" (reachable from the internet) or "private" (not directly reachable) isn't a checkbox anywhere. It's whether that subnet's route table has a route sending internet-bound traffic to an internet gateway. Once that clicks, the rest of VPC networking (public web tier, private database tier, NAT for outbound-only access) is really just variations on that one idea.

Key terms

  • Region — a physical AWS location (e.g. us-east-1) made up of multiple, isolated data centers.
  • Availability Zone (AZ) — one or more discrete data centers inside a region. A VPC spans a whole region; a subnet lives in exactly one AZ.
  • VPC — Amazon Virtual Private Cloud: a logically isolated virtual network you define inside a region, where you launch resources like EC2 instances and databases.
  • CIDR block — a compact way of writing an IP address range and its size, like 10.0.0.0/16. The number after the slash says how many addresses are in the range (a smaller number means a bigger range).
  • Subnet — a range of IP addresses carved out of a VPC's CIDR block. You launch resources into a subnet, and each subnet must sit inside exactly one AZ.
  • Route table — a set of rules (routes) attached to a subnet that decide where outbound traffic from that subnet is allowed to go, based on its destination address.
  • Local route — a route every VPC route table gets automatically, letting resources anywhere inside the VPC's CIDR block reach each other.
  • Main route table — the route table AWS creates automatically with your VPC. Any subnet you don't explicitly associate with a different route table uses this one.
  • Internet gateway (IGW) — a VPC component you attach to your VPC that lets resources with a public IP address talk to the public internet.
  • NAT gateway — a managed service that lets resources in a private subnet start outbound connections to the internet (or other networks), without allowing anything on the internet to start a connection back to them.
  • Public subnet — a subnet whose route table has a route sending internet-bound traffic (0.0.0.0/0) to an internet gateway. This is a description of the routing, not a special subnet type.
  • Private subnet — a subnet whose route table has no such route to an internet gateway.
  • Public IPv4 address / Elastic IP — an IP address that's routable from the public internet. A resource needs one of these (plus a route to an IGW) to be reachable from outside AWS.

How it works

Here's what a VPC actually is and why everything in AWS networking starts there: it's your own logically isolated slice of an AWS region, the network you'd otherwise have had to build in your own data center. There's no charge for the VPC itself (some of the pieces you attach to it, like NAT gateways, do cost money). Your AWS account already comes with a default VPC in every region, pre-wired with an internet gateway and public subnets so you can launch an EC2 instance and reach it immediately. A VPC you create yourself (a "nondefault" VPC) starts with none of that: no internet gateway, no route to one, until you add it.

1. CIDR blocks and IP addressing

When you create a VPC, you have to give it an IPv4 CIDR block. AWS documents allow sizes between a /16 netmask (65,536 addresses) and a /28 netmask (16 addresses), and recommends picking a block from the private ranges defined in RFC 1918. AWS's own docs give these as example ranges:

  • 10.0.0.0/16 (from the 10.0.0.0–10.255.255.255 range)
  • 172.31.0.0/16 (from the 172.16.0.0–172.31.255.255 range)
  • 192.168.0.0/20 (from the 192.168.0.0–192.168.255.255 range)

A few ranges are off-limits for a VPC entirely: 0.0.0.0/8, 127.0.0.0/8 (loopback), 169.254.0.0/16 (link-local), and 224.0.0.0/4 (multicast). Once you've created the VPC with a primary CIDR block, you can't resize it, but you can associate additional (secondary) CIDR blocks later, each between /28 and /16 and non-overlapping with what's already there.

2. Subnets and Availability Zones

A subnet is just a slice of your VPC's CIDR block, and the one hard rule is that it has to live entirely inside a single Availability Zone; it can't span AZs. That's the whole reason multi-AZ design exists in AWS: if you want your app to survive one data center having a bad day, you put subnets (and the resources in them) in at least two AZs.

The docs describe a few subnet "types," but every one of them is really just a description of the subnet's route table, not a separate mode you switch on:

  • Public subnet — has a route to an internet gateway.
  • Private subnet — has no direct route to an internet gateway (it might still reach the internet through a NAT gateway).
  • VPN-only subnet — routes to a Site-to-Site VPN connection via a virtual private gateway, with no route to an internet gateway.
  • Isolated subnet — no routes to anything outside the VPC at all.

Every subnet also gets associated with a network ACL (an extra, subnet-level firewall) and a route table. If you don't explicitly pick either, the subnet uses the VPC's default network ACL (which allows all traffic by default) and the main route table.

3. Route tables — the actual crux

This is the piece that ties it all together. A route table is a set of routes, and each route is just a pair: a destination (a CIDR block, like 0.0.0.0/0 for "anywhere") and a target (where to send traffic headed there: an internet gateway, a NAT gateway, a peering connection, and so on). Every VPC route table gets a local route automatically, which is what lets resources anywhere inside the VPC's CIDR block reach each other without any extra setup. AWS's own example of a VPC with a primary and secondary CIDR block shows exactly this: both ranges get their own local route in the table (10.0.0.0/16 → local and 10.2.0.0/16 → local).

Every VPC also gets a main route table automatically, and any subnet you don't explicitly associate with a different route table just uses that one. You can create additional (custom) route tables and associate specific subnets with them for finer control.

Here's the part that made this click for me: AWS's own docs define public vs. private subnet purely in terms of this table. Quoting the internet gateway docs directly: "If a subnet is associated with a route table that has a route to an internet gateway, it's known as a public subnet. If a subnet is associated with a route table that does not have a route to an internet gateway, it's known as a private subnet." That's the entire rule. Nothing about the subnet itself changes; it's the route table entry.

4. Internet gateways

An internet gateway is a VPC component (AWS describes it as horizontally scaled, redundant, and highly available, with no bandwidth constraint of its own) that you attach to a VPC to allow communication with the public internet. Attaching it isn't enough by itself: you also need a route in a subnet's route table pointing at it, and the resource itself needs a public IPv4 address (or IPv6 address) to actually be reachable.

For IPv4 traffic, the internet gateway also does the network address translation: your instance only ever knows its private IP address, and the internet gateway swaps in the public/Elastic IP address on the way out (and swaps it back on the way in). One useful fact from the docs: a default VPC already comes with an internet gateway and a route table with the 0.0.0.0/0 → igw route in place; a VPC you create yourself starts with neither.

5. NAT gateways — outbound access without an open door

Private subnets often still need to reach the internet. A database server pulling a patch is the classic example, or an application server calling out to a third-party API. But you don't want the internet to be able to start a connection into that database. That's exactly the gap a NAT gateway fills: instances in a private subnet can connect out, but nothing on the internet can connect in to them through the NAT gateway.

A NAT gateway has to live in a public subnet (this is a common source of confusion since it "serves" the private subnet). If you create a public NAT gateway (the default connectivity type), you attach an Elastic IP address to it, and you route traffic from the NAT gateway on to the internet gateway for the VPC. So the full outbound path for a private-subnet instance looks like: instance → NAT gateway (in the public subnet) → internet gateway → internet. AWS also offers a private NAT gateway type for routing to other VPCs or on-premises networks (no internet gateway or Elastic IP involved), but the public type is the one that solves "private subnet needs internet access."

One detail worth remembering: connections through a NAT gateway must always be initiated from inside the VPC. That's the mechanism that makes it outbound-only.

Why it matters

This one row of routing config is the difference between "reachable from the internet" and "not." Most real AWS architectures put the pieces that need to face the public internet (a load balancer, a bastion host) in public subnets, and keep everything else (application servers, databases) in private subnets that reach out through a NAT gateway when they need to. Getting this wrong in either direction is a real, common failure mode: put a database in a subnet with a route to an internet gateway plus a public IP, and it's exposed; forget the NAT gateway route on a private subnet, and your app server can't pull dependencies or call external APIs and just times out. Once you can read a route table, most "why can't this reach the internet" and "why is this reachable when it shouldn't be" questions become fast to answer.

Remember this

  • A subnet is public only because its route table sends 0.0.0.0/0 to an internet gateway. There's no separate "public" flag.
  • A subnet lives in exactly one Availability Zone. A VPC spans the whole region.
  • Every route table gets a free local route so resources inside the VPC's CIDR block can always reach each other.
  • NAT gateways live in a public subnet and let a private subnet start outbound connections without accepting inbound ones.
  • A VPC's CIDR block size is fixed between /16 (65,536 addresses) and /28 (16 addresses), and picking too small a block early is a common regret.

Try it yourself

  1. Sketch it first. On paper (or any diagramming tool), draw a VPC with CIDR 10.0.0.0/16, two Availability Zones, and one public + one private subnet in each (four subnets total), picking non-overlapping CIDR blocks for each. Done when: you have four labeled subnet boxes with CIDR blocks that don't overlap and a note on which AZ each sits in.
  2. Build the VPC for real. Using the AWS console or CLI (free tier), create the VPC and the four subnets from your sketch, tagging each "public" or "private." Done when: aws ec2 describe-subnets (or the console subnet list) shows all four with the CIDR blocks and AZs you planned.
  3. Wire up the internet gateway. Create and attach an internet gateway to your VPC, then add a 0.0.0.0/0 → igw route to a route table associated with only the two public subnets. Done when: a public-subnet EC2 instance with a public IP can reach the internet (e.g. respond to a ping or serve a page), and you can explain why the private subnets can't yet.
  4. Prove the private subnet has no direct inbound path. Launch a small EC2 instance in one private subnet with no public IP assigned. Try to reach it directly from your laptop (SSH, curl, whatever its security group would normally allow from the internet). Done when: the connection simply can't be made from outside the VPC, because there's neither a public IP nor a route in for it.
  5. Add a NAT gateway and verify outbound-only. Create a NAT gateway in one of your public subnets (remember it needs an Elastic IP and isn't free to run, so tear it down after), then add a 0.0.0.0/0 → nat-xxxx route to the private route table. Done when: your private instance can now reach the internet outbound (e.g. update its packages), but an external host still can't initiate a connection to it.
  6. Read a route table cold and narrate it. Pick any subnet's route table (yours or a screenshot from the docs) and, without looking anything up, explain out loud what each destination/target row means and whether the subnet is public or private. Done when: you could explain it to someone who's never touched AWS.

Go deeper

Amazon VPC also supports VPC endpoints, which let resources reach certain AWS services privately, without going through an internet gateway or a NAT gateway at all. If a private subnet's whole reason for needing outbound internet access is just "talking to S3 or another AWS service," a VPC endpoint is worth exploring as an alternative to a NAT gateway for that specific traffic.
