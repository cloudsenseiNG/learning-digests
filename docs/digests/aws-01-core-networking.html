Core networking: VPC, subnets, route tables, NAT, and the IGW

In one line: how AWS traffic actually decides where to go, and why one row in a route table is the entire difference between a subnet being reachable from the internet or not.

The big idea

Every AWS resource you launch (a server, a database) needs a network to live in, and that network is a VPC: Amazon's term for Virtual Private Cloud, which is really just your own private slice of network inside AWS that you define yourself. Inside a VPC you carve out subnets, smaller IP ranges, and each subnet ends up being "public" or "private" not because of a switch you flip on the subnet, but because of exactly one row in a routing table attached to it. Once you can read that one row, most of the VPC picture (what's exposed to the internet, what isn't, how private things still manage to reach out) clicks into place.

Key terms

  • VPC: your own isolated network inside AWS, defined by a range of IP addresses. e.g. a VPC with the range 10.0.0.0/16 has 65,536 addresses to hand out to everything inside it.
  • CIDR block: a compact way to write "a range of IP addresses" as address/prefix-length, like 10.0.0.0/16. The smaller the number after the slash, the bigger the range (AWS allows VPC CIDR blocks from /16 down to /28).
  • Subnet: a smaller slice of a VPC's IP range. A subnet has to sit entirely inside one Availability Zone (roughly, one physical cluster of AWS data centers). You launch resources into subnets, never directly into the VPC.
  • Route table: a set of rows attached to a subnet, each one saying "traffic headed to this destination, send it to this target." Every VPC gets a default one (the main route table) automatically.
  • Internet gateway (IGW): the door between your VPC and the public internet. You attach one to the VPC as a whole, then a route table decides which subnets actually get to use it.
  • NAT gateway: a one-way door for a private subnet. It lets things inside start conversations with the internet (grab a software update, call an external API) without letting anything on the internet start a conversation with them.

How it works

A VPC is a fenced-off piece of network you own

When you create a VPC, the main thing you decide upfront is its IP address range. AWS allows that range to be as big as a /16 (65,536 addresses) or as small as a /28 (16 addresses); a common starting choice looks like 10.0.0.0/16. Nothing runs directly "in the VPC" though, everything you launch goes into a subnet carved out of that range.

A subnet is public or private purely because of its route table

This is the part that took me longer than it should have to internalize: the AWS docs are explicit that a subnet's type "is determined by how you configure routing," not by any label on the subnet itself. A public subnet is one whose route table has a route pointing at an internet gateway. A private subnet is one whose route table doesn't. Same kind of subnet object either way, the only difference is one row in a table.

The route table is the actual decision-maker

Every route table is a list of rows, each one a destination (a CIDR range) and a target (where to send matching traffic). Every route table comes with one row for free: the local route, which just means "anything addressed inside my own VPC, keep it inside my own VPC." Every subnet you create is automatically attached to the VPC's main route table unless you associate a custom one instead. Add a row that says destination 0.0.0.0/0 (in plain English: "anything not covered by a more specific row") with a target of this VPC's internet gateway, and every instance in that subnet now has a path out. Leave that row out and there is no path out, no matter what else is configured on the subnet or its instances, even if those instances have public IP addresses.

The internet gateway is the door, but it's inert until something routes to it

An internet gateway is a component you attach to your VPC as a whole. AWS describes it as "horizontally scaled, redundant, and highly available," so there's no capacity you manage for it, you just attach it. It also quietly performs one-to-one NAT for IPv4 traffic (translating your instance's private IP to a public one on the way out, and back again on the way in), so from the instance's own point of view it's always just talking in private IPs. Attaching an IGW to a VPC does nothing by itself: it only becomes useful once a route table points a subnet's traffic at it. One exception worth knowing: the default VPC that every AWS account starts with in each Region already comes pre-wired with an IGW and the 0.0.0.0/0 route in its main route table. A VPC you create yourself starts with neither.

A NAT gateway lets a private subnet reach out without letting anything reach in

Say you've got a database sitting in a private subnet (no IGW route, so nothing on the internet can reach it directly), but it still needs to call out to download a patch or hit an external API. That's what a NAT gateway is for. You create it inside a public subnet (it needs an IGW path itself to work), attach an Elastic IP (a fixed public IP) to it, and then in the private subnet's route table you add a 0.0.0.0/0 row whose target is the NAT gateway instead of the IGW. Traffic then flows: private instance → NAT gateway → internet gateway → internet. Nothing can start a connection in the opposite direction; every connection has to originate from inside the VPC. For production setups, AWS's own guidance is to put a NAT gateway in every Availability Zone you use, because if multiple zones share one NAT gateway and that zone has a problem, every other zone loses its outbound path too.

Why it matters

Once you can read a route table, a lot of "why can't my instance reach the internet" debugging (or the scarier version, "why can anyone reach my database") collapses into one question: what does this subnet's route table actually say. It's also the mental model behind nearly every AWS architecture diagram you'll see: a couple of public subnets for load balancers, a couple of private subnets for the application and database, and a route table quietly making that separation real instead of just a label on a diagram.

It's not just a small-scale concern either. AWS's own post-event report for the December 7, 2021 outage in its Northern Virginia (US-EAST-1) Region named the failed component as "networking devices... that provide additional routing and network address translation" between AWS's internal network and its main network. An automated capacity-scaling activity triggered unexpected behavior that overwhelmed those devices starting at 7:30 AM PST; congestion didn't meaningfully improve until 1:34 PM PST, and the affected network devices weren't fully recovered until 2:22 PM PST, roughly seven hours. Along the way, access to Amazon S3 buckets and DynamoDB tables through VPC Endpoints was impaired, and EC2, Route 53, and the AWS Console itself all saw errors and delays. (Worth being precise here: this was a fault in AWS's own internal routing and NAT infrastructure, not a customer VPC route table like the one this digest walks through, they're different systems built for different purposes. The parallel is the idea, not the exact mechanism: routing and address translation aren't a background detail, they're the substrate everything else depends on, and a fault there has outsized blast radius.)

Remember this

  • A subnet is public or private only because of its route table, not because of any label you set on the subnet itself.
  • 0.0.0.0/0 pointed at an internet gateway is the entire definition of a "public subnet."
  • The local route, built into every route table for free, is what lets things inside your VPC talk to each other regardless of public or private.
  • A NAT gateway has to live in a public subnet itself, and it only lets traffic go out, never in.
  • Your account's default VPC already has an IGW and a public route wired up; a VPC you create yourself starts with neither.

Try it yourself

  1. Build a public + private subnet pair by hand. In the VPC console, create a VPC, then a public subnet and a private subnet inside it (or use the console's "VPC and more" wizard, which offers this directly). Complete when you can see two subnets and can point at exactly which route table row makes one public and not the other.
  2. Trace a route table like a lawyer. Take any route table (yours, or one from a diagram you find) and, without looking anything up, write one plain sentence per row explaining what traffic it catches and where it sends it. Complete when every row has a sentence.
  3. Break the internet path on purpose. In a test VPC, remove the 0.0.0.0/0 route from a public subnet's route table (console: Route tables → Edit routes), then try to reach an instance in it. Complete when you've confirmed that a wide-open security group isn't enough without the route, and can explain why.
  4. Stand up a NAT gateway and prove the one-way door. Create a NAT gateway in a public subnet with an Elastic IP, point your private subnet's 0.0.0.0/0 route at it, then from an instance inside curl an external endpoint (should work). Complete when you've also confirmed nothing external can initiate a connection back to that instance.
  5. Draw the diagram before you check it. On paper or in any drawing tool, sketch a VPC with 2 Availability Zones, a public and private subnet in each, one IGW, and a NAT gateway per zone, before rereading the docs. Complete when your sketch correctly places each NAT gateway inside a public subnet and the IGW attached to the VPC itself (not a subnet).
  6. Build a tiny CIDR slicer. Write a script (any language) that takes a block like 10.0.0.0/16 and a target prefix like /24, and prints the resulting subnet ranges. Complete when it correctly slices a /16 into at least 4 non-overlapping /24s.
