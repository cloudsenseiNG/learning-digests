Core Networking: VPC, Subnets, Route Tables, NAT

Public or private isn't a checkbox anywhere in AWS. It's decided entirely by one line in a route table, and once that clicks, the rest of VPC networking falls into place.

The big idea

Every AWS resource you launch lives inside a VPC (Virtual Private Cloud), your own logically isolated slice of network. A VPC is carved into subnets, and each subnet's fate, whether it can reach the internet or not, is decided entirely by what its route table points to. There's no "public subnet" setting to flip. It's just routing.

Key terms

  • VPC (Virtual Private Cloud): a virtual network dedicated to your AWS account, logically isolated from everyone else's. You define its IP range and add subnets and gateways inside it.
  • Subnet: a range of IP addresses inside a VPC where you launch resources. A subnet always lives entirely within one Availability Zone, it can't span two.
  • Route table: a set of rules (routes) that decide where traffic from a subnet gets sent. Each route has a destination (an IP range) and a target (where to send matching traffic).
  • Main route table: the route table a VPC gets automatically. Any subnet you haven't explicitly attached to a different route table uses this one by default.
  • Local route: an automatic route every route table gets, matching the VPC's own IP range, so resources inside the VPC can always reach each other.
  • Internet gateway (IGW): the AWS-managed component that lets a VPC talk to the public internet.
  • NAT gateway: an AWS-managed service that lets instances with no public IP make outbound-only connections to the internet (or another network), without accepting unsolicited inbound ones.

How it works

A VPC is your slice, subnets are its rooms

A VPC is a virtual network dedicated entirely to your account. Inside it, you carve out subnets, ranges of IP addresses where you actually launch things like EC2 instances. One rule that trips people up at first: a subnet must sit entirely inside a single Availability Zone (AWS's phrase for an isolated data center location within a region), it cannot stretch across two. If you want redundancy across Availability Zones, you create a separate subnet in each one.

Every new AWS account gets a head start here: if your account was created after December 4, 2013, AWS gives you a default VPC in every Region automatically, already wired up with a default subnet in every Availability Zone, an internet gateway attached, and a route in the main route table sending all traffic to that gateway. That's why you can launch an EC2 instance in a brand-new account and have it reach the internet immediately, without configuring any networking yourself. A VPC you build yourself (a "nondefault" VPC) starts with none of that: no internet gateway and no route to one, until you add both.

Route tables decide public vs private, nothing else does

Every subnet is associated with exactly one route table: either one you explicitly attach, or, if you haven't, the VPC's main route table by default. Every route table automatically includes a local route matching the VPC's own CIDR range, so anything inside the VPC can always reach anything else inside it, that part is free and automatic.

What makes a subnet "public" is nothing more than this: its route table has a route sending 0.0.0.0/0 (meaning "everything else, whatever isn't matched by a more specific route") to an internet gateway. No route like that, no internet access, even if the instance inside has a public IP address assigned. That's the entire distinction. There's no separate "make this subnet public" toggle anywhere in the console, just that one row in a table.

Same VPC, same account. The only difference is one row in a route table.

NAT gateways: outbound access without being reachable

Sometimes you want a private instance (a database, an internal service) to reach out to the internet, say, to download a package update, without ever being reachable from the internet. That's what a NAT gateway is for: it's an AWS-managed NAT (Network Address Translation) service that lets instances in a private subnet initiate outbound connections, while blocking any unsolicited inbound ones.

AWS documents two connectivity types:

  • Public NAT gateway (the default): lives in a public subnet, requires an Elastic IP address at creation, and routes traffic to the VPC's internet gateway. This is the one you'd use so a private-subnet instance can reach the open internet.
  • Private NAT gateway: no Elastic IP allowed. Used to reach other VPCs or an on-premises network through a transit gateway or virtual private gateway, not the open internet. If you try to route a private NAT gateway's traffic to an internet gateway, AWS's documentation is explicit that the internet gateway just drops it.

Either way, per AWS's own documentation, connections must always be initiated from inside the VPC that contains the NAT gateway, nothing on the outside can open a connection in.

A NAT gateway is a one-way door: out is fine, in is not.

Simplification: subnets also get an automatic default network ACL (a separate, stateless firewall layer that allows all traffic by default), and security groups sit on top of that at the instance level. Both matter for real security posture, but they're a different mechanism from routing and out of scope for this networking-basics digest.

Why it matters

This isn't just trivia. When something in your VPC can't reach the internet (or, worse, is reachable when it shouldn't be), the very first thing worth checking is the route table attached to that specific subnet, not security groups, not the instance itself.

AWS's own infrastructure runs on the same principle at a much larger scale, and it's exactly this kind of internal networking layer that caused one of AWS's most disruptive outages. On December 7, 2021, an automated scaling activity triggered unexpected behavior in AWS's internal network clients, which caused a surge of connection attempts that overwhelmed the network devices connecting AWS's internal network to its main network. That single internal-networking failure rippled outward for hours, degrading or breaking EC2 API calls, the AWS Management Console, Route 53 APIs, and several other services that had nothing obviously to do with networking on the surface. The lesson scales down too: the "boring" routing layer underneath your app is very often exactly where a bad day starts.

Remember this

  • A subnet lives in exactly one Availability Zone. It never spans two.
  • "Public" and "private" aren't settings, they're just whether the subnet's route table sends 0.0.0.0/0 to an internet gateway.
  • A default VPC (accounts created after Dec 4, 2013) comes pre-wired with a public default subnet in every AZ, an internet gateway, and the route to it. A VPC you build yourself starts with none of that.
  • A NAT gateway lets private instances reach out to the internet, but nothing from the internet can reach in through it.
  • Public NAT gateways need an Elastic IP and route to an internet gateway; private NAT gateways can't have an Elastic IP and are for reaching other VPCs or on-prem, not the open internet.

Try it yourself

  • Build a VPC from scratch. In the AWS free tier, create a nondefault VPC with one subnet, no internet gateway attached yet. Confirm an EC2 instance you launch there truly has no internet access.
  • Flip it public. Attach an internet gateway to that VPC, add a 0.0.0.0/0 route pointing to it in the subnet's route table, and confirm the same instance now reaches the internet, without ever touching the instance itself.
  • Two subnets, two fates. Create a second subnet in the same VPC, leave its route table on the main one (no route to the internet gateway), and compare: same VPC, two different outcomes, purely from routing.
  • Stand up a NAT gateway. Put an instance in a private subnet, add a public NAT gateway in a public subnet with an Elastic IP, route the private subnet's 0.0.0.0/0 to the NAT gateway, and confirm the private instance can reach out but can't be reached from outside.
  • Break it on purpose. Remove the internet gateway route from a public subnet's route table and watch previously-working outbound traffic stop immediately, then put it back. It's a fast way to feel how much routing (and nothing else) is doing the work.
  • Map two Availability Zones. Create one subnet in each of two Availability Zones in the same VPC and try to create a single subnet spanning both, confirm for yourself that the console won't let you.
