VPC Core Networking: The Route Table Row That Decides Public vs Private

Every AWS resource lives inside a fenced-off network you draw yourself, called a VPC. Whether something in it can be reached from the internet almost never comes down to a firewall rule you'd expect. It comes down to one row in a routing table.

The big idea

A VPC (Virtual Private Cloud) is a private network you define inside AWS, the same way you might design a network of rooms in a building you own. Inside it you carve out subnets, which are just smaller ranges of IP addresses, and you decide how traffic is allowed to leave each one using route tables. The thing that took me the longest to really get: "public subnet" and "private subnet" are not special AWS settings you flip on. They are just a description of what's in the subnet's route table. If its table has a route pointing at an internet gateway, it's public. If it doesn't, it's private. That's the whole rule.

Key terms

  • VPC (Virtual Private Cloud) — a logically isolated virtual network you define inside your AWS account, similar to a network you'd run in your own data center. e.g. one VPC per environment: one for staging, one for production.
  • CIDR block — the range of IP addresses assigned to a VPC or subnet, written like 10.0.0.0/16. The number after the slash tells you how big the range is: a smaller number means more addresses. e.g. 10.0.0.0/16 = 65,536 addresses. 10.0.0.0/28 = 16 addresses (the smallest a VPC or subnet is allowed to be).
  • Subnet — a slice of your VPC's IP address range, pinned to exactly one Availability Zone (AWS's term for an isolated data center cluster within a region). You launch things like EC2 instances into subnets, not directly into a VPC.
  • Availability Zone (AZ) — one of several isolated locations within an AWS region. Spreading subnets across AZs is how you protect an application from one data center going down.
  • Route table — a set of rules attached to a subnet (or gateway) that says, for a given destination IP range, which "target" traffic should go through to get there.
  • Route — a single rule inside a route table: a destination CIDR block, plus a target (a gateway, network interface, or connection) to send matching traffic to.
  • Main route table — the route table AWS automatically creates with every VPC. Any subnet you don't explicitly attach to a different table uses this one by default.
  • Local route — the route AWS adds automatically to every route table so resources inside the VPC can always reach each other, regardless of subnet.
  • Internet gateway (IGW) — the VPC component that connects your network to the public internet. A subnet only counts as "public" if its route table has a route pointing at one.
  • Public subnet — a subnet whose route table has a direct route to an internet gateway.
  • Private subnet — a subnet whose route table has no direct route to an internet gateway.
  • NAT gateway — a managed device you place in a public subnet that lets resources in a private subnet start outbound connections to the internet, without ever letting the internet start a connection back to them.
  • Elastic IP address — a static, public IPv4 address you can attach to a resource (an instance, a NAT gateway) so it keeps the same public address over time.

How it works

Here's the mental model that made this click for me: everything you run on AWS, an EC2 instance, a load balancer, a database, has to live inside a VPC, because a VPC is what actually defines the private IP address space those resources get assigned from. AWS gives every account a default VPC in every region, already wired up so you can launch something and reach the internet immediately, but understanding what's happening underneath that default setup is what makes the rest of AWS networking make sense. So let's build one up from scratch, piece by piece.

The VPC and its CIDR range

When you create a VPC, you have to give it an IPv4 CIDR block: the range of private IP addresses everything inside it will draw from. AWS documents the allowed size as anywhere from a /16 netmask (65,536 addresses) down to a /28 netmask (16 addresses). AWS recommends picking a range from the private address space defined in RFC 1918, things like 10.0.0.0/16, 172.16.0.0/12, or 192.168.0.0/16, since those ranges are set aside for private networks and won't collide with real internet addresses. A few specific ranges are blocked outright (like 0.0.0.0/8 and the multicast range 224.0.0.0/4), and you can later attach additional secondary CIDR blocks to the same VPC if you outgrow the first one.

Why it exists: without this boundary, AWS would have no way to hand your resources private IP addresses that are guaranteed not to clash with another customer's resources sitting on the same underlying hardware. The VPC is what makes "my own private network, logically separated from everyone else's" actually true.

Subnets and Availability Zones

A subnet is just a smaller slice carved out of the VPC's CIDR range, and AWS is strict about one thing: a subnet has to sit entirely inside a single Availability Zone, it can never span two. So if your VPC is 10.0.0.0/16, you might carve out 10.0.1.0/24 as a subnet in AZ "a" and 10.0.2.0/24 as a separate subnet in AZ "b". This is exactly why the standard setup you'll see everywhere is "one public and one private subnet per AZ": it lets an application keep running even if a whole Availability Zone has a bad day.

One detail that surprised me: subnets aren't as roomy as the CIDR math suggests. AWS reserves the first four addresses and the last address in every subnet, five addresses total, that you can never assign to your own resources. In a 10.0.0.0/24 subnet (256 addresses) those are 10.0.0.0 (the network address itself), 10.0.0.1 (reserved for the VPC's router), 10.0.0.2 (reserved for AWS's DNS server), 10.0.0.3 (reserved for AWS's future use), and 10.0.0.255 (the broadcast address, which AWS doesn't actually support in a VPC but still reserves). So a /28 subnet, the smallest allowed at 16 addresses, only actually gives you 11 usable ones.

Route tables and how routing decides public vs private

This is the piece that reframed everything for me. A route table is attached to a subnet (or, if you never attach one explicitly, the subnet just uses the VPC's main route table by default). Every route inside that table is a simple pair: a destination CIDR block ("where is this traffic trying to go") and a target ("what device or connection should handle it"). Every route table gets one route for free, the local route, which just says "anything addressed inside this VPC, keep it inside the VPC." That's what lets two subnets in the same VPC talk to each other by default.

The public-vs-private label isn't a checkbox anywhere in the console. It's derived entirely from whether the subnet's route table happens to contain a route whose target is an internet gateway. Add that one row, destination 0.0.0.0/0 (meaning "anything not covered by a more specific route"), target your internet gateway's ID, and the subnet is now public. Leave it out, and it's private, full stop, even if every instance in it happens to have a public IP address sitting unused.

Internet gateways

An internet gateway is the AWS-managed component that actually bridges your VPC to the public internet. It's described as horizontally scaled, redundant, and highly available, so it isn't a single box you have to worry about failing, and there's no charge for the gateway itself (though normal data transfer charges still apply). You attach one IGW to a VPC, then add the 0.0.0.0/0 → igw-xxxx route to whichever route table(s) should be public. For IPv4 traffic, the IGW also quietly does one-to-one network address translation: your instance only ever knows its own private IP address, and the IGW is what maps that to the instance's public or Elastic IP address on the way out and back.

One useful default to know: a default VPC (the one AWS gives every account per region automatically) already comes with an internet gateway attached and a route table with the 0.0.0.0/0 route already in place. A VPC you create yourself gets neither by default, you have to add both yourself.

NAT gateways, and why private subnets aren't stuck

Having no route to an internet gateway is exactly the point of a private subnet: nothing outside the VPC can start a connection into it. But that raises an obvious problem. What if something inside a private subnet, an application server, say, needs to reach out to download a package update or call an external API? That's what a NAT gateway is for. You place it in a public subnet (it needs its own route to the internet gateway to work), attach an Elastic IP address to it, and then in your private subnet's route table you add a 0.0.0.0/0 route whose target is the NAT gateway instead of the internet gateway.

Traffic then flows: private instance → NAT gateway → internet gateway → internet, and the response follows the same path back. The NAT gateway swaps the private instance's address for its own on the way out, so from the internet's point of view, the traffic is coming from the NAT gateway, not from inside your private subnet. Crucially, this only works one direction: the docs are explicit that connections must always be initiated from inside the VPC. Nothing on the internet can use the NAT gateway to start a new connection into your private subnet. AWS also documents a "NAT instance" option, running your own NAT on a regular EC2 instance instead, but explicitly recommends the managed NAT gateway over it for better availability and bandwidth with less to administer yourself.

A private instance can start a conversation outbound through the NAT gateway, but nothing can start one coming in.

Why it matters

Almost every AWS architecture you'll ever look at, a simple two-tier web app, a Kubernetes cluster, a data pipeline, is built on exactly this pattern: public subnets for the things that need to face the internet (load balancers, bastion hosts), private subnets for everything that shouldn't (application servers, databases), and a route table quietly deciding which is which. Once this clicks, reading any AWS network diagram gets a lot faster, because you stop looking for a "public" label and start looking for the route table row instead. It's also the first thing worth checking when something "should be reachable but isn't," or worse, "shouldn't be reachable but is."

Remember this

  • A VPC is your own private IP address space in AWS; you choose its CIDR range, between a /16 and a /28.
  • Every subnet lives in exactly one Availability Zone, and AWS reserves 5 IP addresses in every subnet you can't use.
  • "Public" and "private" aren't settings. They're a description of one thing: does the subnet's route table have a route to an internet gateway or not.
  • An internet gateway lets traffic in and out; a NAT gateway lets private-subnet traffic out only, never in.
  • The route table is the single source of truth for where traffic goes. When in doubt, read the route table.

Try it yourself

Micro-projects
  • Build a two-subnet VPC. In the AWS console, create a VPC with a custom CIDR (e.g. 10.0.0.0/16), then add one public and one private subnet in the same Availability Zone with non-overlapping CIDR blocks (e.g. 10.0.1.0/24 and 10.0.2.0/24).
  • Wire up the internet gateway. Create and attach an internet gateway to your VPC, then add a 0.0.0.0/0 route pointing at it in the route table associated with your public subnet. Confirm your private subnet's route table has no such route.
  • Prove the difference. Launch a small EC2 instance in the public subnet with a public IPv4 address auto-assigned, and one in the private subnet without one. (You'll also need a security group that allows inbound traffic for this to actually work end to end; that's a separate topic from routing, so just note it for now.)
  • Add a NAT gateway. Create a NAT gateway in your public subnet with an Elastic IP, then point your private subnet's 0.0.0.0/0 route at it instead of the internet gateway. From the private instance, test that outbound requests (like a package update or a curl to an external endpoint) succeed.
  • Inspect it from the CLI. Run aws ec2 describe-vpcs and aws ec2 describe-route-tables and compare what comes back against what you configured in the console, so the CLI output and the visual model line up in your head.
  • Size a tiny subnet. Create a subnet with a /28 CIDR block, the smallest AWS allows, and work out (or check in the console) how many usable IP addresses you actually get once the 5 reserved addresses are subtracted.
