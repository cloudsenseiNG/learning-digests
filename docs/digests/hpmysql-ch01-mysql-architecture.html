MySQL Architecture: The Layers Underneath Every Query

In one line: how MySQL is actually put together, from the moment a query arrives to the moment a storage engine hands back a row, and why almost every tuning decision later traces back to this shape.

The big idea

MySQL isn't one big program that does everything itself. It's built as stacked layers: a layer that handles connections, a "brain" layer that parses and optimizes your SQL, and a swappable storage engine layer underneath that's the only part that actually touches disk. Once you can picture those layers, and how MySQL lets many people read and write the same data at once without corrupting it (locks, transactions, MVCC, replication), the rest of learning MySQL starts to feel like "which layer does this happen in, and how do I make that layer's job easier" instead of a pile of unrelated facts.

Key terms

Storage engine: the swappable component that actually stores and retrieves rows on disk. e.g. InnoDB is the one you'll use almost always.

Storage engine API: the internal interface the MySQL server uses to talk to whichever storage engine is plugged in, so the server doesn't need to know the engine's internal details.

Thread: one line of execution inside the server process. MySQL gives each client connection its own thread, and keeps a pool of them ready so it doesn't have to create one from scratch every time.

Parse tree: the internal, structured representation MySQL builds from your raw SQL text before it can plan or run anything.

Query optimizer: the part of the server that decides how a query will actually run: which table to read first, which index to use, whether to rewrite the query.

Query cache: an older, now fully removed (as of MySQL 8.0) internal feature that tried to serve repeated queries straight from memory. If you're using a current version, this doesn't exist anymore.

Concurrency control: the rules that stop multiple people writing to the same data at the same time from corrupting it.

Lock: a flag on a piece of data that says "someone's using this, wait your turn."

Shared (read) lock: many clients can hold this on the same data at once, since reading doesn't change anything.

Exclusive (write) lock: only one client can hold this, and it blocks every other read and write on that data until it's released.

Table lock: a lock on an entire table. Cheap to manage, but it blocks everyone else working on that table, even rows they don't care about.

Row lock: a lock on just the specific rows being changed. Lets more people work at once, at the cost of more bookkeeping.

Transaction: a group of SQL statements treated as one all-or-nothing unit of work.

ACID: atomicity, consistency, isolation, durability: the four properties a transaction system needs to actually be safe.

Isolation level: a setting that controls how much of other people's unfinished transactions you're allowed to see.

Dirty read: seeing a change from another transaction that hasn't even committed yet (and might be rolled back).

Nonrepeatable read: running the exact same SELECT twice inside one transaction and getting a different answer, because someone else committed a change in between.

Phantom read: running the same range query twice and seeing a new row appear the second time, because someone inserted into that range in between.

Deadlock: two transactions each waiting on a lock the other one is holding, forever, until something forces the issue.

Write-ahead logging: writing a durable log record of a change to disk before (or instead of) immediately updating the real table on disk.

MVCC (multiversion concurrency control): letting a transaction read a consistent snapshot of the data without needing a lock, by keeping multiple versions of a row around behind the scenes.

AUTOCOMMIT: MySQL's default mode, where every single statement is automatically its own tiny transaction that commits instantly.

Two-phase locking: InnoDB's approach of acquiring locks as needed throughout a transaction, but releasing all of them together only at commit or rollback, never one at a time mid-transaction.

Replication: automatically copying writes from one MySQL server out to one or more other servers.

Source / replica: the source is the one server writes land on first; a replica is a server that copies those writes after the fact.

Tablespace: the set of files InnoDB uses on disk to store its data; InnoDB manages what's inside it.

Clustered index: InnoDB's way of physically storing a table's rows in primary-key order, inside the index structure itself, instead of as a separate list.

How it works

"MySQL architecture" just means: what are the pieces MySQL is made of, and how do they hand work to each other. It's worth sitting with this before you tune anything, because nearly every performance knob you'll eventually turn (which index to add, which isolation level to pick, whether to worry about a lock wait) only makes sense once you know which layer it lives in and why that layer exists. Skipping this and jumping straight to "how do I make this query faster" is a bit like trying to fix a car noise without knowing whether the sound is coming from the engine, the brakes, or the exhaust.

1. The three-layer view

Picture MySQL as three layers stacked on top of each other. The top layer handles things that aren't really MySQL-specific at all: accepting a network connection, checking a username and password, and confirming the client is allowed to run the query it's asking to run. The middle layer is where most of the "smarts" live: it parses your SQL text into an internal structure, figures out the fastest way to actually run it (which index to use, what order to read tables in), and handles anything that isn't tied to one particular storage engine, like stored procedures, triggers, and views. The bottom layer is the storage engine, or engines, that actually store and fetch the rows on disk.

Concretely: when you run SELECT * FROM orders WHERE customer_id = 42, the top layer already knows who you are and that you're allowed to query orders. The middle layer parses that SQL, decides an index on customer_id is the fastest path, and hands the storage engine a much simpler instruction: roughly "fetch me the rows matching this key." The storage engine is the only layer that actually touches a data file.

The "why" here matters: the middle layer talks to whatever storage engine is underneath through one shared internal interface, called the storage engine API. That API hides the differences between engines, so the parsing and optimizing code doesn't need a special case for every engine. It's also, notably, a couple of dozen fairly low-level operations, things like "begin a transaction" or "fetch the row with this primary key," not SQL itself. Storage engines never see or parse SQL and never talk to each other directly; they only answer requests the server sends them.

2. Connection management and security

Before any SQL runs, the top layer's job is just to get you connected safely. Every client connection gets its own thread inside the server process, and that thread is what your queries actually run on. Creating a brand-new thread for every connection would be wasteful, so the server keeps a cache of ready-to-use threads instead of building and tearing one down each time.

Once you're connected, MySQL checks who you are using your username, the host you're connecting from, and your password (X.509 certificates over a TLS connection are also supported as an alternative). After that, for every single query you send, the server checks whether your account actually has permission for that specific action, for instance, whether you're allowed to run a SELECT against a particular table at all.

The "why": authentication and authorization happen before parsing and optimizing even start, so the server never wastes work planning a query it's going to reject anyway. It also means privilege checks aren't a storage-engine concern; they're handled once, centrally, regardless of which engine a table underneath happens to use.

3. Optimization and execution: the parser, the optimizer, and the vanished query cache

Once you're authenticated, your SQL text has to become something the server can actually act on. MySQL parses your query into an internal structure called a parse tree, then runs a variety of optimizations on it: it can rewrite the query, decide what order to read tables in, and choose which indexes to use. You can nudge this process with special keywords (hints) inside the query itself, and you can ask the server to explain its own decisions, which gives you a concrete starting point for reworking a slow query, schema, or setting.

The optimizer itself doesn't care which storage engine a table uses, but the engine still shapes its decisions: the optimizer asks each engine about its capabilities, the cost of certain operations, and statistics about the table's data, since some engines support index types that make particular queries much cheaper.

Worth flagging since it trips people up if they've read older MySQL material: earlier MySQL versions had a built-in query cache that could serve a repeated query straight from memory. As concurrency grew, it became a well-known bottleneck, and it was officially deprecated in MySQL 5.7.20 and fully removed in 8.0. It's genuinely gone, not just discouraged. Caching frequently-served results is still a good idea, but today that means reaching for an external cache like memcached or Redis in front of MySQL, not a built-in feature of the server itself.

4. Storage engines: the swappable third layer

The clean split between "the brain" and "the storage" is what lets MySQL swap out how data is physically stored without touching how SQL is parsed or optimized. That's what people mean when they call MySQL's storage layer pluggable: different tables in the same database, even the same query, can involve different storage engines, each with its own tradeoffs, similar in spirit to how a Linux machine can offer several different filesystems.

In practice today, that choice is simple: use InnoDB. It's the default engine, it supports transactions and crash recovery, and the book calls it the "gold standard" you should reach for as your default for basically every table. Older engines like MyISAM still exist in MySQL's history and don't support transactions, which is why mixing them into transactional logic is risky (more on that below).

The "why" behind pluggability: the query optimizer genuinely doesn't care what engine a table uses for most decisions, but it does ask each engine about its capabilities and the cost of certain operations, since some engines support index types or locking strategies that others don't. That's a deliberate design tradeoff: instead of forcing one general-purpose storage strategy on every workload, MySQL lets the storage layer specialize.

5. Concurrency control: locks, and how much to lock

Any time more than one query might change the same data at the same time, something has to arbitrate. A simple way to picture the problem: imagine a spreadsheet on a shared drive that your whole team can edit. If two people try to change the same cell at once, you need rules, or someone's edit silently disappears.

MySQL solves this the way most databases do, with two kinds of lock. A shared (read) lock lets many clients read the same data at once, since reading doesn't change anything and nothing conflicts. An exclusive (write) lock is different: only one client can hold it, and while it's held, it blocks every other read and write on that data. The logic is simple once you see it: it's only ever safe to have one writer touching a piece of data at a time, and no one reading it mid-change.

Say two people try to update the same customer's email address at the same second. Without locking, you could end up with a row that's a mash-up of both writes, or one write silently overwritten. With write locks, the second write simply waits until the first one finishes.

Lock granularity is the tradeoff of how much you lock at once. A table lock locks an entire table: it's the cheapest to manage (little bookkeeping) but the most blocking, since even readers or writers who don't care about the same rows have to wait. A row lock locks only the exact rows being touched, so unrelated rows can be written concurrently, but every lock, its owner, how long it's been held, has to be tracked, which costs more overhead. MySQL doesn't force one answer here: because storage engines implement their own locking, different engines pick different granularities to suit different workloads.

6. Transactions and the ACID guarantee

A transaction groups several SQL statements into one all-or-nothing unit: either every statement in it applies, or none of them do. A classic reason this matters is moving money between two accounts. Say you're moving $150 from Priya's checking account to her savings account. That takes at least three steps: check the checking balance is at least $150, subtract $150 from checking, add $150 to savings.

1START TRANSACTION; 2SELECT balance FROM checking WHERE customer_id = 88214; 3UPDATE checking SET balance = balance - 150.00 WHERE customer_id = 88214; 4UPDATE savings SET balance = balance + 150.00 WHERE customer_id = 88214; 5COMMIT;

If the server crashed right after line 3 but before line 4, and there were no transaction wrapping this, Priya's $150 would just vanish. Wrapping it in a transaction means if anything goes wrong before COMMIT, MySQL can ROLLBACK and undo the partial work, so the money is never "in transit and lost."

For a transaction system to actually be trustworthy, it needs to satisfy ACID:

  • Atomicity: the whole transaction happens, or none of it does. No such thing as a half-applied transaction.
  • Consistency: the database moves from one valid state straight to the next valid state; a crash mid-transaction can't leave money missing from both accounts.
  • Isolation: other transactions generally can't see this one's changes until it commits (with nuance, covered next).
  • Durability: once committed, the change survives a crash. (Durability is described in the source material as "slightly fuzzy," since no system is ever 100% durable, that's part of why backups exist at all.)

This is one of InnoDB's strongest features, and it's a big part of why it's the recommended default: getting ACID guarantees from the storage engine means you don't have to hand-roll that safety logic in your application code.

7. Isolation levels: how much unfinished work you're allowed to see

Isolation sounds simple ("don't let transactions see each other's mess") but the ANSI SQL standard actually defines four levels, each trading off how much concurrency you get against how much you might see of another transaction's in-progress work.

LevelDirty reads?Nonrepeatable reads?Phantom reads?
READ UNCOMMITTEDYesYesYes
READ COMMITTEDNoYesYes
REPEATABLE READ (MySQL's default)NoNoYes*
SERIALIZABLENoNoNo

*In InnoDB specifically, REPEATABLE READ also avoids phantom reads in practice, because InnoDB uses MVCC (and gap locking on writes) to close that hole. That's an InnoDB-specific strengthening, not something the ANSI standard requires at that level.

Walking the levels: READ UNCOMMITTED lets you see other transactions' changes before they've even committed (a "dirty read"), which is rarely worth the risk. READ COMMITTED (the default in most other databases, but not MySQL) only shows you committed changes, but the same query run twice in one transaction can return different rows if someone else committed in between, a nonrepeatable read. REPEATABLE READ, MySQL's default, guarantees rows you've already read will "look the same" if you read them again in the same transaction. SERIALIZABLE is the strictest: it locks every row a transaction reads, which is safe but can create a lot of waiting and lock contention, so it's rarely used in practice.

8. Deadlocks: when two transactions wait on each other forever

A deadlock happens when two transactions each hold a lock the other one needs, and each is waiting for the other to let go. It's a direct consequence of row-level locking: it can only happen once you've got more than one lock in play at a time.

Picture a stock_price table keyed by (stock_id, date). Transaction A updates the price for stock 4, then tries to update the price for stock 3. Transaction B, running at the same moment, updates the price for stock 3 first, then tries to update stock 4. Each transaction now holds a lock the other one wants next, and neither will let go until it commits, which it can't do until it gets the lock the other is holding. That's the deadlock.

MySQL doesn't just let this hang forever. InnoDB actively watches for these circular waits and breaks them by rolling back one of the two transactions immediately (specifically, the one estimated to have the fewest exclusive row locks, since that's the cheapest one to undo), returning an error to that client instead of leaving it stuck. Because deadlocks are a normal, expected part of a system with row-level locking, well-behaved applications are built to catch that error and simply retry the transaction from the start.

9. Transaction logging: why the log gets written first

Updating a big on-disk table for every single change would be slow, since that means random reads and writes scattered across a data file. Instead, most transactional storage engines (InnoDB included) update an in-memory copy of the data first, which is fast, and separately append a record of the change to a transaction log on disk. Appending to a log is comparatively cheap because it's sequential I/O in one small area of the disk, not random I/O scattered everywhere. This pattern is called write-ahead logging: the log entry lands on durable disk before, or as part of, making the change durable, and the actual table on disk gets updated by a separate process at some later point.

The payoff: if the server crashes after the log entry is written but before the on-disk table itself is updated, the storage engine can replay the log on restart and recover the change. The exact recovery mechanics differ per engine, but the core idea (log it fast and sequentially, apply it to the real table later) is shared.

10. Running transactions day to day: AUTOCOMMIT, mixing engines, and explicit locks

By default MySQL runs in AUTOCOMMIT mode: every single INSERT, UPDATE, or DELETE is implicitly its own tiny transaction, committed the instant it runs. Turning AUTOCOMMIT off is what lets you group several statements into one transaction with START TRANSACTION (or BEGIN) and finish with COMMIT or ROLLBACK. You can also set which isolation level a session (or the whole server) uses with a SET command. One easy-to-miss detail: certain commands, especially structural ones like ALTER TABLE, will silently commit whatever transaction is currently open before they run, so it's worth checking your MySQL version's docs for the full list if you rely on this.

Because MySQL doesn't manage transactions at the server level at all, each storage engine implements its own transaction support, and this has a real consequence worth calling out plainly: you can't reliably mix storage engines inside one transaction. If you write to both a transactional table (InnoDB) and a nontransactional table (like older MyISAM tables) in the same transaction and everything goes fine, no problem. But if that transaction ever needs to roll back, the nontransactional table's changes can't be undone, which leaves your data in an inconsistent state, and MySQL often won't even warn you clearly that this happened. The practical rule: pick one engine, InnoDB, and don't mix engines inside application logic that depends on transactions.

InnoDB's own locking, described earlier, is mostly automatic and implicit, it just happens based on your isolation level using a two-phase locking approach (locks can be acquired throughout the transaction, but are all released together at COMMIT or ROLLBACK). InnoDB also supports grabbing a lock explicitly with SELECT ... FOR SHARE or SELECT ... FOR UPDATE when you specifically need to hold a row while you decide what to do with it. Separately, MySQL supports server-level LOCK TABLES / UNLOCK TABLES commands, but the source material is emphatic that you shouldn't reach for these with InnoDB: if you need transactional safety, use a transaction, since InnoDB's row-level locking already covers what most people reach for LOCK TABLES to solve, and mixing it with transactions has surprising, version-dependent behavior.

11. MVCC: how readers avoid waiting at all

Multiversion concurrency control (MVCC) is how most of MySQL's transactional storage engines, InnoDB included, get much higher concurrency than plain row locking alone would allow. The core idea: instead of a reader having to wait for a writer to finish, the engine keeps multiple versions of a row around and gives each transaction a consistent "snapshot" of the data as it looked at some point in time.

InnoDB does this by assigning every transaction an ID the first time it reads data. When a row is changed, InnoDB doesn't overwrite history, it writes an undo record (a note on how to reverse the change) to an undo log, and points the row at that record. When another session reads the row, InnoDB compares the row's transaction ID to what that session is allowed to see. If the current version isn't visible yet (its transaction hasn't committed), InnoDB walks backward through the undo log until it finds a version that is visible to that session. This is also why two transactions running at the same moment can legitimately see different data in the very same table at the same time, which can be genuinely confusing the first time you encounter it.

The "why" is concurrency: because most reads can be satisfied from a snapshot instead of a lock, most SELECT queries in InnoDB never acquire a lock at all, they just read as fast as they can. The tradeoff is overhead: extra bookkeeping per row, more work to walk undo history, and general housekeeping. MVCC only works cleanly under REPEATABLE READ and READ COMMITTED; READ UNCOMMITTED always reads the newest version regardless of snapshot, and SERIALIZABLE locks every row it reads, so neither one is really compatible with the "read a snapshot without locking" idea.

12. Replication: copying writes to other servers

MySQL is built to accept writes on exactly one server at a time, called the source. Every other server, called a replica, receives a copy of those writes automatically, a process called replication. Concretely, the source keeps a dedicated thread per replica, logged in as a replication client, which wakes up whenever a write happens and streams the new data out.

The "why" is consistency: accepting writes on only one server at a time avoids the much harder problem of reconciling two servers that both think they're the authority for the same row. The tradeoff is that replicas aren't a live mirror, they're a copy that's a little bit behind, because getting the write to them happens after the write is already done on the source. The source material is explicit that for anything running in production, you should run replication with at least three replicas, ideally spread across different physical locations, specifically for disaster recovery.

13. InnoDB: MySQL's default engine, and what's new in 8.0

Since InnoDB is what you'll actually use almost all the time, it's worth knowing a few of its specifics. It stores data in a set of files collectively called a tablespace, which InnoDB manages internally. It defaults to REPEATABLE READ and supports all four ANSI isolation levels. Its tables are built on a clustered index, meaning rows are physically stored in primary key order as part of the index itself, which makes primary-key lookups fast, but means every secondary index also has to carry a copy of the primary key columns, so a large primary key makes every other index on that table larger too.

A few notable things changed in MySQL 8.0 specifically: table metadata moved out of separate .frm files into a transactional data dictionary stored inside InnoDB itself, which means even structural changes to a table (DDL, like ALTER TABLE) can now be atomic, either fully applied or fully rolled back, the same all-or-nothing guarantee transactions give your data. Separately, InnoDB gained native JSON document storage with automatic validation back in the 5.7 release, and MySQL 8.0.7 added the ability to index values inside JSON arrays directly.

One thing worth flagging plainly: the source material describes several InnoDB internals (the adaptive hash index, insert buffer, predictive read-ahead) only by name, pointing to a later, dedicated chapter for depth. This digest doesn't cover those internals for that reason, don't take their absence here as "they don't matter," they're just out of scope for this pass.

Why it matters

Nearly everything you'll do to make a MySQL-backed app faster or safer traces back to one of these layers. Slow query? You're probably fighting the optimizer's choices in the middle layer, or an indexing decision that lives in the storage engine. Random errors under load that seem to "come and go"? That's very often lock contention or a deadlock, both concurrency-control problems. An app that reads its own just-written data and doesn't see it? That's very likely reading from a replica that hasn't caught up yet, an asynchronous-replication problem, not a bug in your code. Knowing which layer you're dealing with narrows down where to actually look.

Remember this

  • MySQL is layered: connection handling first, then a "brain" that parses and optimizes SQL, then a swappable storage engine that's the only part that touches disk.
  • Storage engines are what actually store your data, and MySQL lets you pick per table, but InnoDB is the sane default for almost everything now.
  • Locking is a tradeoff: table locks are cheap but block everyone; row locks let more work happen at once but cost more bookkeeping.
  • REPEATABLE READ is MySQL's default isolation level, and InnoDB's MVCC means most reads never need a lock at all.
  • Replication is asynchronous by default. A replica is a copy that's a little behind, not a live mirror, and that gap is exactly what can bite you, especially right after a failover.

Try it yourself

  • Open two sessions against a MySQL instance (a free local install or a small container works fine). In one, start a transaction and update a row without committing. In the other, try to read, then try to write, that same row, and watch exactly when it blocks and when it doesn't.
  • Deliberately reproduce a deadlock: with two sessions, have each start a transaction, update two of the same rows but in opposite order, and watch InnoDB detect the cycle and roll one of them back automatically.
  • Compare isolation levels yourself: set one session to READ COMMITTED, run the same SELECT twice inside a transaction while a second session commits a change to that row in between, and watch the answer change. Then repeat under REPEATABLE READ and confirm it doesn't.
  • Build a tiny "lock timer": in a script, open a transaction, update a row, and hold it open for a fixed number of seconds before committing. From a second connection, time how long a write to that same row has to wait.
  • If you can spin up two MySQL instances (or containers), configure one as source and one as replica, write a batch of rows quickly to the source, and measure how long it takes before they show up on the replica. Try it a second time under heavier write load and see if the lag grows.
  • Pick one small table and read up on what happens when you roll back a transaction that touched both an InnoDB table and a non-transactional table in the same statement sequence. Even without running it, walking through why that's unsafe cements the "don't mix engines in one transaction" rule.

Go deeper (optional)

If MVCC's undo-log mechanics feel fuzzy on a first pass, that's completely normal, it's the most conceptually dense part of this chapter. Come back to that section right after you've run the two-session locking experiment above; watching a transaction actually wait (or not wait) makes the "snapshot instead of a lock" idea click in a way reading alone usually doesn't.
