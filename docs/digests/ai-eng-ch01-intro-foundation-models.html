Introduction to Building AI Applications with Foundation Models

What a foundation model actually is, why one training trick let these models scale past anything before them, and what changes about the job of building AI applications now that you're adapting a model instead of building one from scratch.

In one line

You used to have to train a model to build an AI application. Now you mostly adapt one that already exists, and that single shift is what created the discipline people are calling AI engineering.

The big idea

For most of AI's history, if you wanted a model to do something, you built and trained a model for that exact thing. Foundation models broke that pattern. They're trained once, at huge scale, on huge amounts of ordinary data, and then anyone can adapt the same model to a wide range of different jobs. That's possible because of a training trick called self-supervision, which let models learn from raw text and images instead of needing every example hand-labeled by a person. The result: building an AI application today usually means prompting, retrieving context for, or lightly finetuning an existing model, rather than training one yourself. That's a genuinely different job from traditional machine learning (ML) engineering, and this chapter is about what that job looks like.

Key terms

  • Language model — a model that encodes statistical information about a language, so it can predict how likely a word is to appear in a given context. Example: given "My favorite color is __", it should predict "blue" more often than "car".
  • Token / tokenization / vocabulary — a token is the basic unit a language model works with (a character, a word, or a piece of a word). Tokenization is the process of splitting text into tokens, and the vocabulary is the full set of tokens a model knows. GPT-4 splits "I can't wait to build AI applications" into 9 tokens, and its vocabulary has 100,256 of them.
  • Autoregressive language model — predicts the next token using only the tokens that came before it. This is what almost every text-generating model today is (including the models behind ChatGPT-style tools).
  • Masked language model — predicts a missing token using context from both before and after it (fill-in-the-blank). BERT is the well-known example. Used more for classification-style tasks than for generating open-ended text.
  • Generative AI — a model that produces open-ended output by combining a fixed vocabulary in new ways, instead of picking from a fixed list of answers.
  • Supervision — training a model on data that a human has labeled (for example, transactions tagged "fraud" or "not fraud").
  • Self-supervision — training where the labels come from the data itself instead of a human. A sentence can supply its own "next word" labels. Not the same as unsupervised learning, where there's no label at all.
  • Parameter — a variable inside a model that gets adjusted during training. Roughly, a model's "size" is measured by how many of these it has.
  • LLM (large language model) — a language model that's gotten big by training on huge, mostly unlabeled text datasets. "Large" is a moving target: 117 million parameters was considered large in 2018 and small a year later.
  • Multimodal model — a model that works with more than one kind of data (for example text and images) instead of only one.
  • Foundation model — a large language or large multimodal model that's general-purpose enough, and meant, to be built on top of for many different applications. "Foundation" points at both its importance and the fact that it's a base layer, not a finished product.
  • Embedding — a vector (list of numbers) that's meant to capture the meaning of a piece of data, so that similar things end up as similar vectors.
  • Model as a service — the practice of a few organizations training expensive foundation models and then exposing them to everyone else through an API, so you don't have to train your own model to use one.
  • AI engineering — the process of building applications on top of an existing foundation model, as opposed to training a model yourself.
  • Model adaptation — the general term for techniques that make a general-purpose model behave the way you need for your specific task. The three common ones: prompt engineering, retrieval-augmented generation (RAG), and finetuning.
  • Prompt engineering — adapting a model's behavior through instructions and context in the input, without changing its weights.
  • Retrieval-augmented generation (RAG) — connecting a model to an external source of information (like a database) that it can pull from when generating a response.
  • Finetuning — continuing to train an already-trained model on your own data, which does change its weights.
  • Pre-training — training a model from scratch, starting from randomly initialized weights. This is the most resource-intensive stage by far.
  • Inference / inference optimization — inference is the process of a trained model computing an output for a given input. Inference optimization is the work of making that faster and cheaper.
  • Agent — an AI system that can plan a sequence of steps and use external tools to carry them out, rather than just answering a single prompt.

How it works

Before getting into the specific ideas, it helps to have one grounding picture in your head: a foundation model is a single, very large model that was trained once, on a very broad slice of the world's data, and then handed off (often through an API) for anyone to build on. That's different from the older pattern, where a team trained a smaller model that was only good at the one job they built it for. The rest of this section walks through how we got from "a model that predicts the next word" to "a general-purpose base layer that a whole new engineering discipline gets built on top of."

From language models to LLMs

A language model, at its core, is just something that has learned the statistics of a language well enough to guess what word is likely to come next. This isn't a new idea: Sherlock Holmes used letter-frequency statistics to crack a substitution cipher in an 1905 story, and Claude Shannon formalized how to model English statistically in a 1951 paper. What's changed is scale.

Models don't work with whole words directly, they work with tokens, pieces of text that can be a whole word, a character, or a fragment like "-tion". Breaking text into tokens is tokenization, and the full set of tokens a model can use is its vocabulary. As a rough rule of thumb, one token is about three-quarters of a word, so 100 tokens is roughly 75 words. Tokens exist as a middle ground: they're more efficient than tracking every possible whole word (fewer unique units to manage), but they keep more meaning than single characters, and they let a model handle made-up or unfamiliar words by splitting them into familiar pieces.

There are two broad families of language models, and they differ in what information they're allowed to use when predicting a token. A masked language model (BERT is the well-known example) predicts a missing token using context from both sides, so it's good at fill-in-the-blank style tasks like classification. An autoregressive language model predicts the next token using only what came before it, which is exactly what you need for generating text one token at a time. Autoregressive models are the ones behind almost all modern text generation, because "keep predicting the next token" is a simple, repeatable loop you can run indefinitely.

It's worth being precise about what a language model is actually doing: it's a completion machine. Give it "To be or not to be" and it might complete it with ", that is the question." That sounds simple, but completion is a surprisingly general tool. Frame a translation as "How are you in French is …" and the completion is a translation. Frame a yes/no question as a completion prompt ("Is this email spam? Answer:") and the completion is a classification. The catch is that a completion is a probabilistic guess, not a guarantee, and a model trained purely to complete text isn't automatically good at holding a conversation (later stages of training exist specifically to fix that).

Self-supervision: the trick that let language models scale

Language modeling is one of many kinds of machine learning, alongside things like object detection or demand forecasting. What made language models special enough to kick off the current scaling race is that they can be trained with self-supervision, while most other ML approaches need supervision.

With supervision, a human labels every training example. Train a fraud detector and someone has tagged a pile of transactions "fraud" or "not fraud" by hand. This works, and it's how the deep learning boom of the 2010s largely got started (AlexNet, trained on over a million hand-labeled ImageNet photos sorted into 1,000 categories, is the textbook example). But hand labeling is expensive and slow, and the cost grows with the number of categories and the difficulty of the task: labeling everyday objects is cheap because almost anyone can do it, but labeling whether a CT scan shows cancer requires expensive specialist time. As of September 2024, for comparison, Amazon SageMaker Ground Truth charged roughly 8 cents per image for smaller labeling jobs and about 2 cents per image at larger scale, and that's just for ordinary photo labels, not anything specialized.

Self-supervision sidesteps the labeling bottleneck entirely. Instead of a human supplying the label, the label is inferred from the data itself. A plain sentence already contains its own "next word" labels: everything before a given word is context, and the word itself is the thing to predict. That means any ordinary text (books, articles, forum posts) becomes usable training data with zero manual labeling. This is a genuine simplification worth flagging: self-supervision isn't the same as unsupervised learning. Unsupervised learning has no labels at all; self-supervision still trains on labels, it just gets them for free from the structure of the data instead of paying someone to write them.

Because self-supervision removes the biggest cost of scaling up training data, it's the reason language models could grow the way they have. A model that OpenAI released in mid-2018 had 117 million parameters and was considered large at the time; a follow-up about eight months later had 1.5 billion parameters, which immediately made the earlier one look small. As of this chapter's writing, a 100-billion-parameter model was considered large, and that bar keeps moving. More parameters generally (though not always, this is a real exception, not a rounding error) means more capacity to learn, and a bigger model needs more training data to make good use of that capacity, which is exactly what self-supervision made affordable.

From large language models to foundation models

People perceive the world through more than text, so language models are being extended to handle other kinds of data too. Models that understand both images and text (GPT-4V and Claude 3 are named in this chapter as examples) can do things a text-only model can't. A model that works with more than one data type is a multimodal model, and a generative one is sometimes called a large multimodal model (LMM).

This is the point where the term "large language model" stops being accurate enough, and "foundation model" takes over. The word points at two things at once: these models matter enough to be foundational to a huge range of applications, and they're explicitly meant to be built upon, not used as a finished, single-purpose product. Before foundation models, AI research was split by data type: natural language processing dealt only with text, computer vision only with images, and so on, with each producing narrow, task-specific models (a translation model couldn't also do sentiment analysis, and vice versa). Self-supervision made scaling up multimodal training possible too. OpenAI's CLIP, for example, learned to jointly represent images and text by scraping 400 million naturally occurring (image, text) pairs from the internet instead of paying for labels, producing a dataset 400 times bigger than ImageNet with no manual labeling cost. CLIP itself isn't generative, it's an embedding model, meaning it turns images and text into vectors that capture their meaning, and those embeddings underpin later generative multimodal models.

The practical upshot is a shift from task-specific models to general-purpose ones. A single foundation model can do translation and sentiment analysis and summarization reasonably well out of the box, something that simply wasn't true of the narrow models that came before it. That doesn't mean an out-of-the-box model is optimal for your specific job (a retailer's product descriptions might come back generic and full of marketing cliches rather than matching the brand's voice), which is exactly why model adaptation techniques exist: prompt engineering (better instructions and examples), RAG (connecting the model to your own data, like a database of customer reviews), and finetuning (further training the model on your own high-quality examples). Adapting an existing model this way is generally far less work than training a model of your own from nothing, though task-specific models still have real advantages: they can be much smaller, and therefore faster and cheaper to run.

From foundation models to AI engineering

"AI engineering," as used in this chapter, means building applications on top of a foundation model someone else trained, distinct from "ML engineering," the older, broader discipline that includes training your own models. Three things came together to make AI engineering explode as a discipline:

  • General-purpose capability. Because one model can do many tasks, and does more of them well every year, the pool of buildable applications keeps growing, which pulls in more builders and more demand.
  • A sharp rise in investment. Goldman Sachs Research estimated AI investment could approach $100 billion in the US and $200 billion globally by 2025 (a projection, not a guarantee), and by Q2 2023 about one in three S&P 500 companies mentioned AI on earnings calls, three times the year before, according to FactSet.
  • A much lower barrier to entry. The "model as a service" pattern means you don't need infrastructure to host a model, you call an API. Combined with AI itself being able to write code and being usable through plain English instructions, building a first AI application no longer requires a software engineering background.

The result, per this chapter, is that AI engineering has become one of the fastest-growing engineering disciplines: within about two years, four open source AI tools (AutoGPT, Stable Diffusion web UI, LangChain, and Ollama) had already out-starred Bitcoin on GitHub and were closing in on frameworks like React and Vue.

Here's a simple way to picture what actually changed structurally: where you used to have to build the whole pipeline yourself, you now sit on top of someone else's already-trained model and add an adaptation layer.

What AI engineers actually do differently from ML engineers

This chapter frames AI engineering as having evolved out of ML engineering rather than replacing it, and lays out three structural differences:

  1. You use a model someone else trained instead of training your own, so the work shifts away from modeling and toward adaptation.
  2. The models involved are bigger and more compute-hungry, which raises the importance of efficient training and inference, and increases the need for engineers who can work with large GPU clusters.
  3. The models produce open-ended output (not a fixed set of possible answers), which makes evaluation a much bigger and harder problem than it was for close-ended tasks like spam classification, where there's a clear right answer to check against.

It helps to be precise about "training" here, because the word gets used loosely. Training always changes model weights, but not every change to weights counts as training (quantization, which reduces the precision of stored weights to make a model smaller and faster, changes weight values without being training). Within training, pre-training means starting from randomly initialized weights and is by far the most resource-intensive step (for one well-known model, pre-training was reported to use up to 98% of the total compute and data). Finetuning means continuing to train a model that's already been trained once, which is typically far cheaper since the model already has a head start. "Post-training" is used mostly by model developers (the people who build the base model) for the training work they do after pre-training, while "finetuning" is used mostly by application developers doing further training on a model someone else already post-trained; conceptually they're the same operation, just applied by different people at different points. None of this is the same as prompt engineering, even though people sometimes casually say they "trained" a chatbot when what they actually did was write it a good prompt.

Zooming out, this chapter organizes the whole AI application stack into three layers, and traditional ML engineering and AI engineering divide up the responsibility for each layer differently.

In model development, ML knowledge (gradient descent, loss functions, model architectures) is described in this chapter as a "nice to have, not a must-have" now, rather than a requirement, though it's still described as valuable for troubleshooting and for widening the set of tools you can reach for. Dataset engineering shifts too: close-ended tasks with tabular data (traditional ML) give way to unstructured data and challenges like deduplication, tokenization, and quality filtering (removing toxic or sensitive content). Inference optimization, already important in traditional ML, becomes more important still, partly because autoregressive models generate one token at a time, so getting a response out fast (getting anywhere near typical web-app latency) is a genuinely hard engineering problem.

In application development, evaluation becomes both more necessary and harder, precisely because outputs are open-ended: there's no fixed list of correct chatbot replies to check against, the way there is for a fraud/not-fraud label. This chapter uses a real example to show how much technique, not just the model, can change an evaluation result: when Google reported that Gemini beat GPT-4 on the MMLU benchmark in December 2023, that comparison used a prompting technique (Gemini given 32 examples) that wasn't matched against GPT-4 (only 5 examples); when both were tested with 5 examples, GPT-4 actually scored higher. Prompt engineering alone moved Gemini Ultra's own reported MMLU score from 83.7% to 90.04% depending on the technique used. AI interface work also grows in importance, since foundation models put app-building within reach of far more people, and those apps need somewhere for users to actually interact with them (standalone apps, browser extensions, chat integrations, or plug-ins into existing products).

Planning before you build

This chapter is clear that a fun demo is not the same thing as a good reason to ship a product, so it lays out a few questions worth answering first.

Why build this at all? The chapter frames use-case motivation on a rough spectrum from highest to lowest urgency: an existential competitive threat (in a 2023 Gartner survey, 7% of executives cited "business continuity," meaning the business could suffer if it didn't adopt generative AI), a clear profit or productivity opportunity, or a fuzzier "we don't want to be left behind" instinct (with Kodak, Blockbuster, and BlackBerry cited as the classic cautionary tales of waiting too long).

What role should AI actually play? A few small distinctions here change how strict the requirements need to be. Is AI critical to the product, meaning it stops working without it (Face ID), or complementary, meaning the product still works if the AI part is removed (Gmail without Smart Compose)? The more critical it is, the higher the accuracy bar. Is it reactive, responding to a specific user action (a chatbot), or proactive, surfacing something unprompted (a traffic alert)? Proactive features get judged more harshly because users didn't ask for them. Is it dynamic, continually updated from a specific user's ongoing feedback (a personalization feature), or static, shared across a group of users and updated only when the whole model is updated? On top of this, it's worth deciding upfront how much of a role humans play in the decision loop, an idea this chapter calls human-in-the-loop, and a framework it cites from Microsoft (2023), Crawl-Walk-Run, gives a simple way to phase that in: Crawl means a human must be involved, Walk means AI can interact directly with internal employees, and Run means AI can interact directly with external users.

Can you actually defend this product? Because the barrier to entry is low for you, it's low for competitors too. This chapter names three broad kinds of competitive advantage in AI: technology, data, and distribution (the ability to get your product in front of users). With foundation models, the underlying technology tends to look similar across competitors, and distribution tends to favor companies that already have a large user base. That leaves data, which is more nuanced: a startup that gets to market first and gathers real usage data can turn that into a genuine moat, even where the data can't be used to directly retrain a model, because it still reveals what users actually want and where the product falls short.

How will you know if it's working? The chapter recommends tying the application to a concrete business metric (for a support chatbot: percent of messages automated, response time, human labor saved) and separately defining a "usefulness threshold": the bar the product has to clear before it's shown to real users. That threshold usually spans a few metric groups: quality of the output itself, latency (including TTFT, time to first token, and TPOT, time per output token), cost per request, and sometimes other concerns like interpretability or fairness.

How much work is actually left after the demo works? This is where this chapter's most useful caution sits: a good first demo tells you almost nothing about how much work remains. As one paper on training chat models put it, the early progress is easy and the later progress is "exceedingly challenging" (Ding et al., 2023). This chapter shares a concrete illustration: one company reported reaching about 80% of the experience it wanted within a month, then needing four more months of work to get past 95%, much of it spent chasing down inconsistent behavior. Set expectations accordingly: the gap between "impressive demo" and "shippable product" is usually the hardest and slowest part, not the fastest.

What happens after launch? AI products need ongoing maintenance in a way that's shaped by how fast the field moves. Some of that change is good news you still have to react to: model providers cutting prices, or one provider's pricing shift making an in-house build suddenly cheaper (or more expensive) than you planned. Some of it is regulatory and harder to predict: the chapter notes the EU's GDPR was estimated to cost businesses around $9 billion to comply with, and that a US executive order in October 2023 shows how quickly compute access itself can become a regulated resource. And some of it touches intellectual property in ways that are still legally unsettled, which is a real concern this chapter raises for IP-heavy businesses building on models trained on other people's data.

Where this shows up: what people are actually building

This chapter surveys use cases across roughly eight categories (coding, image and video production, writing, education, conversational bots, information aggregation, data organization, and workflow automation), drawn from interviews with 50 companies, over 100 case studies, and 205 open source applications with at least 500 GitHub stars. A few examples grounded in that survey:

  • Coding. Consistently the most popular use case in these surveys. GitHub Copilot, one of the earliest production successes, crossed $100 million in annual recurring revenue within two years of launch. McKinsey researchers found AI roughly doubled developer productivity on documentation and boosted code generation and refactoring by 25 to 50%, with much smaller gains on highly complex tasks.
  • Image and video production. Midjourney reached $200 million in annual recurring revenue at about a year and a half old. As of December 2023, half of the top 10 free Graphics & Design apps on the Apple App Store had "AI" in their name.
  • Writing. An MIT study (Noy and Zhang, 2023) gave 453 college-educated professionals occupation-specific writing tasks and randomly gave half of them access to ChatGPT; the group with access finished 40% faster with output quality up 18%, and the gains were bigger for workers who were weaker writers to begin with.
  • Education. Chegg, a homework-help company, saw its share price fall from $28 when ChatGPT launched in November 2022 to $2 by September 2024, as an illustration of how disruptive this can be for incumbents who don't adapt.
  • Conversational bots, information aggregation, data organization, and workflow automation. These cover customer support and product-copilot bots, summarization and "talk to your documents" tools, extracting structured data out of unstructured documents, and agents, defined in this chapter as AI systems that can plan and use external tools to get a multi-step task done (like booking a restaurant, which needs permission to search, call, and add a calendar entry).

The throughline across all of these: because foundation models are general-purpose, a single application often spans more than one category at once (a support bot that also aggregates information, for instance), which is part of why the space is hard to cleanly categorize in the first place.

Why it matters

If you're building software today, this shift is the reason "add an AI feature" went from "hire an ML team and spend months collecting data" to "call an API this afternoon and see what happens." That's genuinely new leverage, and it's also why AI engineering is described in this chapter as one of the fastest-growing engineering disciplines: the barrier dropped for a huge number of people at once. But the flip side matters just as much: because the barrier is low for you, it's low for everyone else building the same idea, so the questions this chapter raises about defensibility, evaluation, and the last-mile gap between demo and product aren't optional extras, they're the actual hard part of the job now. The next few chapters build directly on this: chapter 2 goes deeper into how foundation models are actually built and adapted, and chapter 3 tackles evaluation, which this chapter already flags as one of the biggest differences from traditional ML work.

Remember this

  • Self-supervision is the real reason language models could scale into the hundreds of billions of parameters: it turned ordinary, unlabeled text into free training data.
  • "Foundation model" describes both scale and purpose. It's meant to be built on top of, not used as a finished, single-purpose tool.
  • AI engineering isn't "easier ML engineering." It's a different job: less model training, more adapting an existing model, evaluating open-ended output, and building the interface around it.
  • Prompt engineering, RAG, and finetuning are three separate levers for adapting a model, and they trade off differently in how much data and effort they need.
  • A working demo tells you almost nothing about whether a product is ready to ship. The gap between "good enough to show" and "good enough to ship" is usually the hardest and slowest part of the whole project.

Try it yourself

  • Count tokens versus words. Take a paragraph you've written (100 to 300 words) and run it through any public tokenizer you have access to (many model providers expose one, and open source tokenizer libraries exist too). Compare the token count to the word count. Done when you can state your own ratio and note whether it's close to the "100 tokens ≈ 75 words" rule of thumb from this chapter.
  • Build the training-pairs table by hand. Pick any short sentence (6 to 10 words) and write out, on paper or in a spreadsheet, every (input, next-token) pair it produces for next-token prediction, the way the schematic above does. Done when you have a full table from the first word through the end of the sentence.
  • Compare two models on the same prompt. Pick a task (summarize a paragraph, write a short email, explain a concept simply) and run the identical prompt through two different publicly accessible chat models or assistants. Done when you've written two or three sentences comparing the outputs: where they agreed, where they differed, and which you'd trust more and why.
  • Sketch your own foundation-model application. Pick a real idea (even a small one). Draw the architecture as boxes: your data/prompt, the foundation model, an adaptation layer, your application, the user, using the diagram above as a starting shape. Label which adaptation technique(s) you'd use (prompt engineering, RAG, finetuning, or a mix) and why. Done when you have a one-page diagram plus two or three sentences justifying the adaptation choice.
  • Run a use-case evaluation on a task you actually do. Pick something repetitive you do often (writing similar emails, summarizing notes, drafting first-pass content). Answer, in writing: what's the risk of not automating it, what would "good enough to trust" look like for quality, latency, and cost, and how would you actually measure whether it's working. Done when you have short, specific answers to all three, not vague ones.
