MySQL Architecture

Why MySQL is really three layers wearing a trench coat, and why the bottom layer is the one that changes everything.

In one line

MySQL isn't one monolithic thing, it's a layered system where the top handles connections and query logic the same way no matter what, and the bottom layer (the storage engine) is swappable and decides how your data actually gets stored, locked, and recovered.

The big idea

Most beginners picture "the database" as a single black box: you send SQL in, rows come out. MySQL's own documentation and this chapter push back on that picture. Underneath the SQL you type, MySQL splits the work into a layer that talks to clients and parses queries, and a separate, pluggable layer that actually stores and retrieves the bytes on disk. That second layer is called a storage engine, and MySQL lets you pick a different one per table. Once you see that split, a lot of "why does MySQL behave this way" questions answer themselves, because the answer is usually "that's the storage engine you're using, not MySQL itself."

Key terms

  • Storage engine — the component that actually stores and retrieves data on disk for a table. MySQL supports several (InnoDB, MyISAM, and others), and different tables in the same database can use different ones.
  • Connection/thread-handling layer — the top layer of MySQL, responsible for accepting client connections, authenticating them, and checking privileges. By default each connection gets its own thread.
  • SQL layer (query/optimizer layer) — the middle layer: parses your SQL into a parse tree, decides the fastest way to run it (the optimizer), and hosts features shared across all storage engines like stored procedures, triggers, and views.
  • Storage engine API — the internal interface the SQL layer uses to talk to whichever storage engine a table uses. It hides each engine's internal differences behind a small set of operations like "begin a transaction" or "fetch the row with this primary key."
  • InnoDB — the default, transactional storage engine in modern MySQL, and the one the rest of this book (and this track) focuses on.
  • Transaction — a group of SQL statements treated as one all-or-nothing unit of work.
  • ACID — atomicity, consistency, isolation, durability: the four properties a transaction system needs to be trusted with important data.
  • Isolation level — a setting that controls how much of one transaction's in-progress changes another transaction is allowed to see.
  • Lock — a mechanism that stops two clients from making conflicting changes to the same data at the same time. Locks can be shared (for reading) or exclusive (for writing).
  • Deadlock — a situation where two transactions are each waiting on a lock the other one holds, so neither can proceed without one being rolled back.
  • MVCC (multiversion concurrency control) — a technique where the engine keeps multiple versions of a row around so readers can see a consistent snapshot without blocking writers.
  • Replication — MySQL's built-in way of copying writes from one server to one or more other servers.
  • AUTOCOMMIT — a MySQL session setting that, when on (the default), wraps every single statement in its own automatic transaction.

How it works

Before touching any MySQL-specific command, it helps to have a mental map of what "MySQL's architecture" even refers to. It's not the server binary, it's the layering: a top layer that deals with clients and connections, a middle layer that understands SQL and decides how to run it, and a bottom layer that actually stores the bytes. That bottom layer, the storage engine, is the part that's pluggable. That single fact (that storage is a plug-in, not a fixed part of the server) explains why so many MySQL behaviors, from locking style to whether a table supports transactions at all, come down to "which storage engine is this table using," not "what does MySQL do."

The three-layer picture

The top layer is the stuff any client/server tool needs: accepting connections, authenticating who you are (username, host, password, optionally TLS certificates), and checking privileges before it lets a query run. By default, each connection gets its own thread on the server, and the server keeps a small pool of ready threads around so it doesn't have to spin one up from scratch every time.

The middle layer is where MySQL "thinks." It parses your SQL into an internal structure (a parse tree), then the optimizer decides things like which order to read tables in and which indexes to use. Anything shared across all storage engines, like stored procedures, triggers, and views, lives here too. Older MySQL versions had a query cache in this layer that tried to reuse results for repeated queries, but as of MySQL 5.7.20 it was deprecated, and MySQL 8.0 removed it entirely, so if you're on a recent version don't expect it to exist. (Caching results yourself, e.g. in Redis or memcached, is still a common and reasonable pattern, just not a built-in MySQL feature anymore.)

The bottom layer is the storage engine, or engines, since different tables can use different ones. The SQL layer never touches disk directly. It asks the storage engine, through the storage engine API, to do things like "start a transaction" or "give me the row with this primary key," and the engine handles the actual reading and writing. The optimizer doesn't care which engine a table uses in principle, but it does ask each engine about its capabilities and cost estimates, because a plan that's fast on one engine can be slow on another.

Pluggable storage engines

The part that surprised me most as a beginner: the storage engine isn't baked into "MySQL," it's a plug-in. The SQL layer above it stays the same regardless of which engine a table uses, it just calls the same small set of API functions ("begin a transaction," "fetch this row") and lets the engine underneath decide how to actually do it. Different engines can trade off differently, in the same way different filesystems on Linux trade off differently for the same job. Because of this, MySQL doesn't force one single locking or storage strategy on every table. InnoDB, the default and by far the most commonly used transactional engine today, is what the rest of this chapter (and the rest of this book) focuses on, but it's worth remembering it's one engine among several MySQL can plug in underneath the same SQL layer.

Concurrency control: why locking exists at all

Any time more than one query might change data at the same time, you need a rule for who goes first. The chapter's own analogy is a shared spreadsheet: if you're the only editor, there's no conflict, but the moment a coworker can edit the same file, you need rules. Reading is mostly safe to do simultaneously, but writing needs exclusivity: the safest policy is one writer at a time, with reads paused while a write is in progress. That's the whole reason locks exist. A shared (read) lock lets many clients read the same thing at once without conflict. An exclusive (write) lock blocks both other writers and other readers, because letting a read happen mid-write risks reading a half-changed, inconsistent result.

Lock granularity: table locks vs row locks

How much you lock at once is a tradeoff. Locking an entire table is cheap to manage but blocks everyone else out of the whole table, even rows nobody else cares about. Locking just the row being changed lets far more work happen in parallel, but the engine now has to track many more individual locks, which costs memory and overhead. In a common MySQL setup, this choice is made by the storage engine, not the server: a table lock locks the whole table (cheap, low concurrency), a row lock locks only the specific row being touched (more overhead, much higher concurrency). InnoDB implements row-level locking; some other MySQL storage engines only support table-level locking. This is a good concrete example of "ask which engine, not what does MySQL do."

Transactions and ACID

A transaction groups several statements into one all-or-nothing unit. The classic example: moving $200 from checking to savings needs three steps (check the balance, subtract from checking, add to savings), and if the process crashes between steps 3 and 4, you don't want the money to vanish or double. Wrapping the steps in a transaction means either all three happen or none do. In SQL, you open one with START TRANSACTION, then either make it permanent with COMMIT or discard it with ROLLBACK.

For a transaction system to actually be trustworthy, it needs to satisfy ACID:

  • Atomicity — the transaction is all or nothing, there's no such thing as "half applied."
  • Consistency — the database moves from one valid state directly to the next valid state, never a broken in-between.
  • Isolation — other transactions usually can't see a transaction's changes until it commits (more on the "usually" below).
  • Durability — once committed, a change survives a crash. Nothing is ever perfectly 100% durable in an absolute sense, but the practical bar is "recorded somewhere that survives a restart."

In InnoDB specifically, ACID transactions are one of the most mature parts of MySQL. They come with a throughput cost, but they save you from re-implementing a lot of that all-or-nothing logic yourself in application code.

Isolation levels: how much can one transaction see of another

The ANSI SQL standard defines four isolation levels, and lower ones generally trade correctness guarantees for higher concurrency. In InnoDB, all four are supported:

A quick definition of each problem: a dirty read is seeing another transaction's uncommitted changes. A nonrepeatable read is running the same query twice in one transaction and getting different results because someone else committed a change in between. A phantom read is selecting a range of rows, having someone else insert a new row into that range, then re-running the same range query and seeing the new "phantom" row appear. Worth calling out: REPEATABLE READ is MySQL's default isolation level, which is a bit unusual since READ COMMITTED is the default for most other database systems. Each storage engine can implement these levels somewhat differently, so this table describes the general ANSI SQL model, not a guarantee about every engine's exact internals.

Deadlocks

A deadlock happens when two transactions each hold a lock the other one is waiting for, so neither can move forward. The book's example: Transaction 1 updates row A then tries to update row B, while Transaction 2 updates row B then tries to update row A. Both end up stuck waiting on each other forever, unless something intervenes. InnoDB detects this and rolls back the transaction estimated to have the fewest exclusive row locks (a rough proxy for "cheapest to undo"), returning an error immediately rather than letting both queries hang. Because deadlocks are storage-engine specific, the same sequence of statements might deadlock in one engine and not another. Applications should expect this and be built to retry a failed transaction from the start.

Transaction logging (write-ahead logging)

Updating the on-disk table for every single change would be slow, since it means random reads and writes scattered across the disk. Instead, a transactional storage engine updates its in-memory copy of the data (fast) and writes a record of the change to a transaction log on disk (also fast, because appending to a log is sequential I/O in one place rather than random I/O everywhere). The actual table on disk gets updated later. This pattern is called write-ahead logging, and it's also what makes crash recovery possible: if the server crashes after the log write but before the table update, the engine can replay the log on restart and recover the change. This means most transactional engines using this technique end up writing each change to disk twice (once to the log, once eventually to the table), which is a real cost worth knowing about, not a free lunch.

AUTOCOMMIT and mixing storage engines

By default, MySQL runs with AUTOCOMMIT on, meaning a single INSERT, UPDATE, or DELETE is automatically wrapped in its own transaction and committed right away. Turning AUTOCOMMIT off lets you run several statements as one transaction, ending with an explicit COMMIT or ROLLBACK. One important, easy-to-miss gotcha: because storage engines (not the MySQL server itself) implement transactions, you can't reliably mix a transactional engine (like InnoDB) and a nontransactional engine (like MyISAM) in the same transaction. If a rollback is needed, the nontransactional table's changes can't be undone, which can leave the database in an inconsistent, hard-to-recover state, often without MySQL clearly warning you. The practical takeaway is to pick one storage engine per table deliberately and avoid mixing engines within the same application logic.

Multiversion concurrency control (MVCC)

Most of MySQL's transactional storage engines don't rely purely on locking reads. Instead they use MVCC alongside row-level locking. The idea: instead of blocking readers while a write happens, the engine keeps enough historical information to hand each transaction a consistent snapshot of the data as it looked at some point in time. This means two transactions can legitimately see different versions of the same row at the same moment, which feels strange at first but is exactly the point: reads mostly don't need to wait for writes at all. In InnoDB specifically, this works by giving every transaction an ID, and when a row is changed, the engine writes an "undo record" describing how to reverse that change. A different session reading that row compares its own view against the row's transaction ID and, if needed, walks backward through undo records until it finds a version it's allowed to see. MVCC isn't unique to MySQL (Oracle and PostgreSQL use versions of it too), but there's no single standard for how it must work, so the details genuinely differ engine to engine and database to database. One more bounded fact worth knowing: MVCC only works with the REPEATABLE READ and READ COMMITTED isolation levels, since READ UNCOMMITTED always reads the newest version regardless of snapshot, and SERIALIZABLE locks every row it reads instead of using snapshots at all.

Replication

In a common MySQL source-replica setup, one server (the source) accepts writes and one or more replicas receive a copy of those writes and apply them, catching up over time. The source keeps a dedicated thread per connected replica that wakes up and streams new data whenever a write happens. This is useful for spreading out read traffic and for disaster recovery: for anything running in production, it's worth having replicas, ideally in more than one physical location, so a single server or site failure doesn't take everything down. Replication in MySQL has grown more capable over the years (features like global transaction identifiers and parallel replication on replicas exist), but the core mental model stays the same: one accepted-write path per replication topology, replicated outward.

InnoDB in more depth

Since InnoDB is the engine most modern MySQL deployments default to and the one the rest of this track will keep coming back to, a few of its specifics are worth knowing early. InnoDB stores its data in datafiles collectively called a tablespace, which it manages internally. It builds tables around a clustered index (the table data itself is physically organized by the primary key), which makes primary-key lookups very fast, but means secondary indexes have to also store the primary key value internally, so a large primary key makes every other index on that table larger too. InnoDB defaults to REPEATABLE READ and uses a locking strategy called next-key locking to close the phantom-read gap at that level: instead of only locking rows it actually touched, it also locks the gaps between index entries, so nothing new can be inserted into a range it just read. InnoDB also supports "hot" online backups (the database keeps serving traffic while being backed up) and, since MySQL 5.6, various forms of online DDL that let you change a table's structure without a full table lock.

MySQL 8.0 changed how table metadata itself is stored: instead of separate files, it moved to a transactional data dictionary stored inside InnoDB, which also enabled atomic DDL (structural changes like ALTER TABLE can now fully succeed or fully roll back, the same all-or-nothing guarantee transactions give your data). InnoDB also gained native JSON document support (starting in MySQL 5.7), with validation and storage optimized for JSON, which used to require awkward BLOB storage instead.

Why it matters

Almost every "why is my MySQL database behaving this way" question eventually comes down to this chapter's split: is this a server-wide behavior (connection handling, the optimizer, isolation-level semantics as a concept) or a storage-engine behavior (locking granularity, whether transactions are even supported, how MVCC is implemented)? Getting that split right early means you'll know where to look when something surprises you later: at the SQL layer, or one level down inside whichever engine that table is using. It's also why later parts of this track focus almost entirely on InnoDB specifically rather than "MySQL" in the abstract: InnoDB is the default and by far the most broadly useful engine, and most of what people mean by "how does MySQL handle X" is really "how does InnoDB handle X."

Remember this
  • MySQL is layered: a connection/auth layer, a SQL/optimizer layer, and a swappable storage engine layer. Only the bottom layer is pluggable.
  • Locking, transaction support, and MVCC details all live in the storage engine, not the server, so "what does MySQL do" is often really "what does this table's engine do."
  • InnoDB is the default, transactional, row-locking, MVCC-based engine, and the one worth learning deeply first.
  • REPEATABLE READ is MySQL's default isolation level (unusual compared to most other databases, which default to READ COMMITTED).
  • Transaction logs use write-ahead logging: change the in-memory copy, log it to disk fast (sequential I/O), then update the actual table later.

Try it yourself

Try it yourself
  1. See the engines available. Install MySQL locally (or spin up a container), connect with the mysql client, and run SHOW ENGINES;. Completion: you can name at least two engines listed and note which one is marked DEFAULT.
  2. Compare two engines on the same schema. Create the same simple table twice, once with ENGINE=InnoDB and once with ENGINE=MyISAM, insert a few rows in each, then run SHOW TABLE STATUS; and compare what's reported. Completion: you can point to at least one visible difference between the two rows in the output.
  3. Watch AUTOCOMMIT change behavior. In one session run SET AUTOCOMMIT=0;, then INSERT a row, and check from a second connection whether that row is visible yet before you COMMIT. Completion: you can explain in your own words what you observed and why.
  4. Force a deadlock on purpose. Recreate the two-transaction example from this digest against a small InnoDB test table with two rows: open two sessions, have each update the two rows in opposite order, and watch one fail. Completion: you see MySQL return a deadlock error to one of the two sessions.
  5. Read a query's plan. Run EXPLAIN SELECT ... on a simple query against a table with a few hundred rows. Completion: you can identify from the output whether the query used an index or scanned the whole table.
  6. Check your isolation level. Run SELECT @@transaction_isolation; (or SELECT @@tx_isolation; on older versions) on a fresh connection. Completion: you confirm it reports REPEATABLE READ by default, then try changing it for your session with SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; and check it changed.

If one thing here is worth re-reading slowly, it's the MVCC section: the idea that two transactions can legitimately see different data in the same table at the same moment is the piece that trips up most newcomers, and it's the foundation for a lot of what later chapters build on (replication consistency, backup strategy, and locking behavior all lean on it).
