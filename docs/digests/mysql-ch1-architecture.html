MySQL Architecture

Why MySQL feels different from other databases: it's really a query brain bolted onto swappable storage engines, and that one design choice explains most of its quirks.

The big idea

Most databases ship with one fixed way of storing data on disk. MySQL splits that decision out into a pluggable layer called a storage engine, and lets you pick (or mix) engines per table. Almost everything in this chapter, locking, transactions, concurrency, is really about explaining what InnoDB (the default and, in practice, the only engine most people use today) does with that pluggable layer.

Key terms

  • Storage engine: the component actually responsible for reading and writing data to disk. MySQL's query layer talks to whichever engine a table uses through a common API, so the engine can be swapped per table.
  • Lock: a mechanism that stops two things from changing (or reading-while-changing) the same data at once. Shared/read locks don't block each other; exclusive/write locks block everyone.
  • Transaction: a group of statements treated as one all-or-nothing unit. Either every statement in it applies, or none do.
  • ACID: the four guarantees a real transaction system needs: atomicity, consistency, isolation, durability.
  • Isolation level: how much of one in-progress transaction's changes another transaction is allowed to see before it commits.
  • Deadlock: two transactions each waiting on a lock the other one holds, forever, unless something forcibly breaks the cycle.
  • MVCC (multiversion concurrency control): a technique that lets readers see a consistent snapshot of data without locking it, by keeping multiple versions of a row around.

How it works

Three layers, one API in the middle

MySQL's architecture is basically three stacked layers. The top layer handles the boring-but-necessary stuff every network client/server tool needs: connection handling, authentication (username, host, password, or TLS certificates), and checking whether you're allowed to run the query you just sent. The middle layer is where the real brains live: parsing your SQL into a parse tree, then optimizing it (deciding table read order, which indexes to use, rewriting the query), plus anything that isn't specific to one storage engine, like stored procedures, triggers, and views. The bottom layer is the storage engines themselves, the part that actually stores and retrieves data on disk. The middle layer talks to whichever engine each table uses through a shared storage engine API, a couple dozen low-level operations like "begin a transaction" or "fetch the row with this primary key." The engines never talk SQL and never talk to each other directly, they just answer requests from the layer above.

Note: MySQL used to have a built-in query cache to skip re-running identical queries, but it became a concurrency bottleneck as usage grew, was deprecated in 5.7.20, and fully removed in MySQL 8.0. If you want that kind of caching today, it lives outside MySQL (an external cache like Redis), not inside the server.

The query layer never touches disk itself. It hands rows back and forth across the storage engine API.

Concurrency control: who gets to touch the data, and when

Picture a shared spreadsheet. Reading it with several people at once is harmless, nobody's changing anything. The trouble starts the moment someone edits a cell while others are reading or editing. MySQL's answer is the classic two-lock system: read (shared) locks that don't block each other, and write (exclusive) locks that block everything else, both reads and other writes, until released.

How much of the "spreadsheet" you lock at once (the lock granularity) is a genuine trade-off. MySQL supports two main strategies:

  • Table locks: lock the entire table. Cheapest overhead, but only one writer at a time for the whole table.
  • Row locks: lock just the specific rows being touched. Lets many people edit different rows at once, but costs more overhead to track who holds what.

Row locks are implemented inside the storage engine, not the shared query layer, which is exactly why different engines can behave differently under concurrent load even though you're writing the same SQL against them.

Transactions and the ACID guarantees

A transaction bundles several statements into one all-or-nothing unit. The book's example is moving $200 between a checking and savings account: check the balance, subtract from checking, add to savings. If the server crashes between steps two and three, you need a way to guarantee the money doesn't just vanish. That's what a transaction is for:

START TRANSACTION;
SELECT balance FROM checking WHERE customer_id = 10233276;
UPDATE checking SET balance = balance - 200.00 WHERE customer_id = 10233276;
UPDATE savings  SET balance = balance + 200.00 WHERE customer_id = 10233276;
COMMIT;

For that to actually be safe, the system needs to pass the ACID test:

  • Atomicity: the whole transaction applies, or none of it does. No partial transfers.
  • Consistency: the database moves from one valid state to the next; a crash mid-transaction can't leave $200 missing from everywhere.
  • Isolation: other transactions generally can't see this one's changes until it commits.
  • Durability: once committed, the change survives a crash. (The book is honest that "durability" is fuzzier than it sounds, since nothing is ever 100% durable, backups exist precisely because the primary copy isn't either.)

MySQL implements four ANSI SQL isolation levels, and how strict each one is trades off directly against how much concurrency you get:

  • READ UNCOMMITTED: transactions can see other transactions' uncommitted changes (a "dirty read"). Rarely used.
  • READ COMMITTED: you only see committed changes, but running the same query twice in one transaction can return different rows (a "nonrepeatable read").
  • REPEATABLE READ: rows you've already read stay consistent for the rest of your transaction. MySQL's default. It can still theoretically allow "phantom reads" (a new row appearing in a range you already queried), which InnoDB specifically closes off using MVCC.
  • SERIALIZABLE: the strictest level, locks every row it reads. Safest, but the most contention and timeouts. Rarely used in practice.

When two transactions each hold a lock the other one wants, you get a deadlock, a permanent stalemate that can only be broken by rolling one of them back. InnoDB actively detects these cycles and rolls back whichever transaction holds the fewest exclusive row locks (roughly, the "cheapest to undo" one), instead of letting both sides hang forever. The book's advice: design your application to retry a transaction that gets rolled back for a deadlock, since a retry usually just succeeds.

A deadlock: neither transaction can proceed, and neither will ever release its lock voluntarily.

MVCC: letting readers skip the line

Plain row locking still forces readers and writers to wait on each other sometimes. InnoDB (like Oracle and Postgres, each with their own implementation) instead leans on MVCC: it keeps enough historical information (via an undo log) to reconstruct what a row looked like at an earlier point in time. When you start reading, you effectively get a consistent snapshot, so most reads never need to acquire a lock at all, they just look at the version of the data that matches when the transaction started, even while other transactions are actively changing that same row. The cost is bookkeeping: extra data stored per row and extra work maintaining the undo log. MVCC only applies at the REPEATABLE READ and READ COMMITTED isolation levels; READ UNCOMMITTED always reads the newest version regardless, and SERIALIZABLE locks every row it touches, so neither one uses MVCC's snapshot trick.

Replication and InnoDB, briefly

MySQL accepts writes on exactly one node at a time and ships those changes out to other servers, called replicas. The book flags this plainly: any production data should be replicated with at least three additional replicas, ideally spread across different physical locations, for disaster recovery. (Replication gets its own full chapter later in the book, this is just the architectural headline.)

Nearly everything above describes InnoDB's behavior specifically, because InnoDB is MySQL's default and, per the book, the engine that "should cover nearly every use case" today. It uses MVCC, supports all four ACID isolation levels, defaults to REPEATABLE READ, and organizes tables around a clustered index (more on that in a later chapter on indexing). It also supports JSON documents natively (added in 5.7, with validation and optimized storage) and, since MySQL 8.0, atomic DDL, meaning schema changes themselves (like ALTER TABLE) are now all-or-nothing instead of possibly leaving things half-changed.

Why it matters

Every performance and correctness question you'll hit later in this book (indexing, replication lag, backup strategy, query tuning) sits on top of the architecture in this chapter. If you don't know that locking and transactions are implemented per storage engine, not by the server itself, a lot of "why does MySQL behave differently here" moments stop making sense. And since InnoDB is the engine you'll use almost all the time in practice, understanding its MVCC-based approach to concurrency is the single most reusable idea in this chapter.

Remember this

  • MySQL's storage engine is pluggable per table; the query layer never touches disk directly, it hands work to whichever engine that table uses.
  • Row locks give more concurrency than table locks but cost more overhead to track.
  • ACID = atomicity, consistency, isolation, durability. A transaction that fails any of these isn't really a transaction.
  • MySQL's default isolation level is REPEATABLE READ, not READ COMMITTED (the default most other databases pick).
  • MVCC lets most reads skip locking entirely by reading a consistent snapshot, at the cost of extra bookkeeping per row.

Try it yourself

  • Deadlock on purpose. Install MySQL locally (or use any free sandbox), open two terminal sessions, and recreate the two-transaction deadlock example from this digest (update two rows in opposite order in each session). Watch MySQL detect it and roll one side back.
  • AUTOCOMMIT explorer. Turn off AUTOCOMMIT in a session (SET AUTOCOMMIT=0), run a few statements, then open a second session and try to read the changed rows before you COMMIT. Confirm they're invisible until you commit.
  • Isolation level comparison. Run the same two-session experiment under READ COMMITTED and then REPEATABLE READ (SET SESSION TRANSACTION ISOLATION LEVEL ...) and see the nonrepeatable-read difference for yourself: does a second SELECT in the same transaction see a row that changed in between?
  • Table lock vs row lock timing. Create one table using MyISAM (table locks) and one using InnoDB (row locks), then fire concurrent updates at each from two sessions and time how much they block each other.
  • Build a tiny append-only log. Write a script that "writes" changes to an in-memory structure and separately appends a line to a log file for every change (mimicking write-ahead logging). Kill the script mid-run and write a recovery routine that replays the log to rebuild state, the same idea InnoDB's transaction log uses for crash recovery.
  • Read the query plan. Run EXPLAIN on a couple of your own queries against a real InnoDB table and see what the optimizer says about read order and index choice, the same optimizer behavior this chapter describes at a conceptual level.

Go deeper

The one section worth rereading slowly is "Multiversion Concurrency Control." It's short here, but MVCC quietly underlies almost every InnoDB behavior that seems surprising the first time you meet it (like two sessions seeing "different" data in the same table at the same moment), and this chapter explicitly says the details get much deeper in the indexing and replication chapters later on.
