🧭 AWS — Topic 1: Core Networking

VPCs, subnets, route tables, NAT gateways, and internet gateways: the five pieces that decide whether a server can be reached from the internet, and whether it can reach out to it.

In one line

A VPC is your own private slice of AWS network, and everything else in this topic (subnets, route tables, gateways) is just the plumbing that decides which parts of that slice can talk to the internet, and how.

The big idea

Before you can put a server on AWS, you need somewhere to put it. That "somewhere" is a VPC (Virtual Private Cloud): a private network you define yourself, sitting inside AWS's infrastructure. On its own, a VPC is just an empty, sealed-off address space. You carve it into subnets, and then you use route tables to tell traffic where to go. Whether a subnet can reach the internet (and whether the internet can reach it back) comes down entirely to what's in its route table, not to anything inherent about the subnet itself.

Key terms

VPC (Virtual Private Cloud)
A logically isolated network you define inside AWS. Think of it as your own data center's network, minus the physical hardware.
CIDR block
The range of IP addresses assigned to a VPC or subnet, written like 10.0.0.0/24. Every subnet you create is carved out of the VPC's CIDR block.
Subnet
A slice of IP addresses inside your VPC. Every subnet lives entirely inside one Availability Zone (AWS's term for an isolated data center within a Region), it can't span two.
Availability Zone (AZ)
One of several isolated locations within an AWS Region. Spreading subnets across AZs is how you avoid one data center outage taking your whole app down.
Route table
A set of rules attached to a subnet (or a gateway) that says "traffic headed to this destination goes out through this target." It's the actual traffic controller, subnets don't route anything on their own.
Local route
The route every route table gets automatically, for traffic staying inside the VPC. It's what lets two subnets in the same VPC talk to each other by default.
Main route table
The route table your VPC comes with automatically. Any subnet you don't explicitly attach to a different route table uses this one.
Internet gateway (IGW)
A component you attach to a VPC that lets resources with a public IP talk directly to the internet, in both directions.
NAT gateway
A managed service that sits in a public subnet and lets instances in a private subnet start connections out to the internet, without allowing anything from the internet to start a connection back in.
Elastic IP address
A static, public IPv4 address AWS gives you, that you attach to something like a NAT gateway so it has a fixed public address to translate traffic through.

How it works

A VPC is an empty network until you add subnets

When you create a VPC, you give it a CIDR block, a range of private IP addresses, and that's basically it. It has no servers, no internet access, nothing. To actually run anything, you divide that CIDR block into subnets, and each subnet has to live entirely within a single Availability Zone. A common pattern is one public subnet and one private subnet per AZ, so if you're spreading across two AZs for redundancy, that's four subnets total.

What makes a subnet "public" or "private" isn't the subnet, it's the route table

This is the part that took me a minute to really get. A subnet doesn't have a "public" or "private" flag you toggle. According to the AWS docs, a subnet is public if its route table has a route pointing to an internet gateway, and private if it doesn't. That's the whole definition. Two subnets can have identical CIDR ranges and settings, and the only thing making one public and the other private is what's written in their route tables.

Every route table also gets a "local" route automatically, for traffic addressed to somewhere inside the VPC itself. That's a simplification worth flagging: the docs note this local route exists for both IPv4 and IPv6 if the VPC has both, but the mechanics of dual-stack routing go beyond what this digest covers.

Internet gateways: the door for public subnets

An internet gateway attaches to the whole VPC (not to a specific subnet), and it's what makes internet access possible at all. But attaching one to your VPC isn't enough on its own. A subnet only becomes "public" once its route table has a route sending internet-bound traffic (conventionally written as destination 0.0.0.0/0, meaning "anywhere") to that internet gateway as the target. And the instance itself still needs a public IP address, or the internet gateway has nothing to translate traffic to.

Worth noting: AWS's docs point out that even if an instance in a subnet has a public IP, it still can't reach the internet if its route table has no route to an IGW. The route table is what actually gates the traffic, the public IP alone doesn't do it.

NAT gateways: outbound-only access for private subnets

Instances in a private subnet don't get public IPs and their route table has no route to an internet gateway, so nothing from the outside can reach them directly. But they often still need to reach out, say, to download a software update or call an external API. That's what a NAT gateway is for.

A NAT gateway is created inside a public subnet (not the private one), and it gets its own Elastic IP address. You then add a route in the private subnet's route table sending outbound traffic to the NAT gateway instead of an IGW. Traffic flows: private instance to NAT gateway (still inside the VPC) to internet gateway (because the NAT gateway lives in a subnet that does have that route) to the internet. The NAT gateway swaps the instance's private IP for its own address on the way out, and swaps it back on the way in, but it will only let a connection through if the private instance started it. Nothing on the internet can initiate a new connection to the private instance through the NAT gateway.

Two things flagged directly from the docs: a NAT gateway lives in one Availability Zone, so if you want resilience across AZs you need a separate NAT gateway per AZ (with routing set up so each private subnet uses the one in its own AZ). And a NAT gateway is a paid resource, you're billed for the hours it runs plus the data it processes, unlike an internet gateway, which has no charge itself (though data transfer through it can still be billed elsewhere).

Route tables: the one thing actually deciding all of this

It's worth restating because it's the whole mental model: a route table is a list of "if traffic is headed here, send it out through this target" rules. Every route table gets a local route for free (traffic staying inside the VPC). Beyond that, you add routes yourself: a route to an internet gateway makes a subnet public, a route to a NAT gateway gives a private subnet outbound-only internet access, and no matching route at all (beyond local) makes a subnet fully isolated. AWS also has the idea of a "main route table" (the one your VPC starts with) versus custom route tables you create yourself. Any subnet you don't explicitly associate with a custom route table just uses the main one.

Why it matters

This is the layer underneath almost everything else on AWS. A web server that needs to be reachable from the internet goes in a public subnet. A database that should never be reachable directly (only from your app servers) goes in a private subnet, reaching out through a NAT gateway only when it needs to, say, pull a security patch. Getting this wrong is a classic AWS security misstep: putting a database in a public subnet with a public IP and an open route to an IGW is one of the more common ways people accidentally expose data. Once you've got this mental model (route table decides everything), later topics like Transit Gateway, load balancers, and VPC peering are really just "more sophisticated things you can point a route at."

REMEMBER THIS

  • A VPC is just an address range until you add subnets, route tables, and gateways to it.
  • "Public" and "private" aren't subnet settings, they're a consequence of what's in the subnet's route table. A route to an internet gateway is what makes a subnet public.
  • Every route table gets a free "local" route for traffic inside the VPC; everything else you add yourself.
  • A NAT gateway lives in a public subnet and gives a private subnet outbound-only internet access, connections can't be started from the internet side.
  • NAT gateways are billed by the hour plus data processed; internet gateways have no charge of their own.

TRY IT YOURSELF

  1. Build a VPC with one public and one private subnet. In the VPC console, use "Create VPC" -> "VPC and more," pick a CIDR block, and choose 1 public + 1 private subnet across 1 AZ (skip NAT gateway for now to avoid charges). Open the "Resource map" tab and trace how the console has wired the route tables.
    Done when: you can point at each subnet's route table and say why it's public or private.
  2. Recreate it with the CLI, one command at a time. Run aws ec2 create-vpc, create-subnet (x2), create-internet-gateway, attach-internet-gateway, create-route-table, create-route --gateway-id <igw-id> --destination-cidr-block 0.0.0.0/0, and associate-route-table for the public subnet. Compare what you typed to what the console did in project 1.
    Done when: aws ec2 describe-route-tables shows your public subnet's route table with a route to the IGW.
  3. Launch an instance in the private subnet and confirm it's unreachable. Launch a small EC2 instance into the private subnet (no public IP). Try to SSH to it directly from your laptop.
    Done when: the connection times out, and you can explain why (no public IP, no route to an IGW) instead of just observing it fails.
  4. Add a NAT gateway and watch outbound-only access appear. Create a NAT gateway in the public subnet with a new Elastic IP, then add a route in the private subnet's route table: 0.0.0.0/0 -> nat-gateway-id (CLI: create-route --nat-gateway-id <nat-id>). From a bastion or Session Manager into the private instance, run curl against any public site.
    Cost flag: a NAT gateway is billed hourly plus per-GB processed for as long as it exists, delete it (and release the Elastic IP) right after this project.
    Done when: the private instance can curl out successfully, but you still can't reach it from the outside.
  5. Break it on purpose, then fix it. Delete the public subnet's route to the internet gateway (leave the local route alone). Confirm outbound access from a public-subnet instance now fails. Re-add the route and confirm it recovers.
    Done when: you've watched the exact same subnet flip from "internet-reachable" to "not" purely by editing a route table entry, nothing else changed.
  6. Draw your own route table by hand. On paper (or in a notes app), sketch a VPC with 2 AZs, 1 public + 1 private subnet per AZ, and write out what each of the 4 route tables should contain (local route, plus IGW or NAT target). No AWS console needed.
    Done when: your sketch has 4 distinct route tables and you can explain why the private subnets in each AZ shouldn't share one NAT gateway in production.
