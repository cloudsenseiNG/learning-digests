Transit Gateway: One Hub Instead of a Web of VPC Peering

How AWS turns a tangle of point-to-point VPC connections into a single hub that every VPC just plugs into once.

The big idea

VPC peering only ever connects two VPCs directly, and it can't forward your traffic through a third VPC to reach a fourth (no transitive routing, more on that word below). That's fine when you have two or three VPCs. It stops being fine once a company has ten or twenty VPCs that all need to reach each other, because peering them all individually turns into a real mess to build and operate. Transit Gateway is AWS's answer: a single regional hub that every VPC, VPN, and on-prem connection attaches to once, so you get a hub-and-spoke network instead of a full mesh. It's less a new protocol and more a way to stop managing a pile of one-off connections by hand.

Key terms

Transit gateway — a regional network hub you create in your account. VPCs, VPN connections, Direct Connect links, and even other transit gateways plug into it, and it routes traffic between whatever's attached.

Attachment — a connection from something (a VPC, a VPN connection, a Direct Connect gateway, a peering link to another transit gateway, a Client VPN endpoint, or an SD-WAN/Connect appliance) into the transit gateway.

Transit gateway route table — works like a normal route table, but it lives on the transit gateway and decides which attachment to forward traffic to next, based on the destination IP.

Association — which route table a given attachment uses to look up where its own outbound traffic should go. Every attachment is associated with exactly one route table, never zero, never more than one.

Route propagation — how a route gets added into a route table automatically instead of you typing it in by hand. An attachment can propagate its routes into a route table.

Transitive routing — traffic hopping through a middle network to reach a third one. VPC peering can't do this. Being the thing that does this is the whole point of a transit gateway.

Hub-and-spoke — a topology where everything connects to one central point instead of connecting to each other directly.

Segmentation — deliberately keeping certain networks from reaching certain other networks, even though they share the same hub.

MTU (maximum transmission unit) — the biggest chunk of data a network link will carry in one go before it has to be split up.

Why full-mesh peering falls over

VPC peering is strictly one-to-one: you create a peering connection between exactly two VPCs, and only those two can talk through it. If VPC A needs to reach VPC C through VPC B, that traffic just doesn't go, peering has no transitive routing. So if you want every VPC in a group to be able to reach every other VPC, you need a separate peering connection between every single pair.

That count grows as N × (N − 1) / 2, where N is the number of VPCs. Ten VPCs need 45 peering connections. Twenty VPCs need 190. And each connection isn't just a checkbox: it's route table entries on both sides, its own IP planning to avoid overlapping CIDR ranges, and usually its own thinking about security groups. This is exactly why AWS's own guidance for multi-account setups (the Control Tower landing zone reference architecture) recommends hub-and-spoke through a transit gateway instead of peering everything together.

What a transit gateway actually is

Strip away the name and a transit gateway is a router that lives in AWS's network instead of in one of your VPCs. You attach things to it (a VPC, a VPN connection, a Direct Connect gateway, a peering connection to another transit gateway, a Client VPN endpoint, or an SD-WAN appliance), and its route table decides where to forward traffic next based on the destination IP address, exactly like a normal route table decides between a NAT gateway and an internet gateway.

One detail worth knowing: a transit gateway supports up to 8,500 bytes MTU between VPC, Direct Connect, Connect, and peering attachments, but a VPN attachment caps out at 1,500 bytes. If you're pushing a lot of data through a VPN-based attachment, that smaller MTU is worth remembering. Pricing is hourly per attachment plus a charge for the data actually processed through the gateway, so more attachments and more traffic both show up on the bill.

Association vs. propagation, the part that trips people up

These two words get used together so often that it's easy to assume they're the same thing. They're not, and mixing them up is the single most common way people get confused reading transit gateway docs for the first time.

Association answers "which route table does this attachment use to send its own traffic?" Every attachment is associated with exactly one route table, no exceptions. That table is what the attachment consults when it needs to know the next hop for outgoing traffic.

Propagation answers a different question: "which route tables know how to reach this attachment?" An attachment's routes can be propagated into more than one route table at once, so other attachments associated with those tables can find their way back to it. Propagation isn't automatic for everything either: a VPC never auto-propagates its routes, you have to add a static route pointing at the transit gateway yourself. A VPN connection propagates its routes automatically over BGP. A peering attachment to another transit gateway also needs a static route, it won't propagate on its own.

Segmentation: letting prod and dev share a hub without touching

Because association and propagation are separate, you can use them on purpose to keep networks apart. Instead of leaving every attachment on the default route table (where everything can typically reach everything), create a second route table: associate your prod VPCs with a "prod" route table, and your dev VPCs with a "dev" route table. If the dev VPCs' routes never get propagated into the prod route table, and vice versa, then a dev VPC has no route to a prod VPC through the hub, even though both are physically attached to the very same transit gateway.

This is the actual mechanism behind network segmentation on a transit gateway. It's not a special "isolation mode" you switch on, it's just choosing carefully which attachment goes on which route table, and which routes you let propagate where. You can also mix static and propagated routes in the same table, for example a static backup route that only kicks in if a dynamically propagated one goes down.

Remember this

  • VPC peering has no transitive routing, so a full mesh of N VPCs needs N × (N − 1) / 2 connections. 10 VPCs is 45 connections, 20 VPCs is 190.
  • A transit gateway turns that mesh into hub-and-spoke: each VPC needs just one attachment to the hub.
  • An attachment is associated with exactly one route table, that's where its own outbound traffic looks things up.
  • Propagation is a separate thing: it's how an attachment's routes get published into a route table (possibly several), so other attachments can find their way back to it.
  • VPC routes never auto-propagate, you always add a static route pointing at the transit gateway yourself. VPN routes propagate automatically via BGP.
  • Segmentation (like keeping dev away from prod) is just separate route tables with deliberate association and propagation choices, not a special toggle.

Try it yourself

  1. Peer three VPCs by hand. Spin up three small VPCs in one AWS account (say 10.0.0.0/24, 10.0.1.0/24, 10.0.2.0/24) and connect them with VPC peering so all three can reach each other. Count how many peering connections and route table entries you actually needed.
  2. Rebuild it with a transit gateway. Tear down the peering setup and reconnect the same three VPCs with a single transit gateway instead: one attachment per VPC, one shared route table. Compare how many resources you touched versus the peering version.
  3. Draw a 5-VPC org. On paper, sketch a hub-and-spoke diagram for a hypothetical company with 5 VPCs, then work out how many peering connections a full mesh of the same 5 VPCs would need.
  4. Write a segmentation plan. In a text file, plan the route tables for a shared transit gateway that keeps a "dev" VPC from reaching a "prod" VPC. Name the route tables, decide which attachment associates with which, and note what you'd deliberately leave unpropagated.
  5. Do the math for 15 VPCs. Calculate N × (N − 1) / 2 for a company with 15 VPCs, then say out loud how many attachments the same company would need with a transit gateway instead.
  6. Audit your own account. Look at your AWS account's VPC list (or a free-tier sandbox) and sketch what a transit gateway migration would actually look like: which VPCs would attach, and whether you'd need more than one route table.
