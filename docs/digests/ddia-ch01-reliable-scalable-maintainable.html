SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What it actually means to call a system "good", broken into three separate questions you can ask about anything you build: does it keep working when things go wrong, does it hold up as load grows, and can people keep working on it without everything turning into a mess?

In one line

Reliability, scalability, and maintainability aren't vague compliments, they're three specific, separately-answerable questions worth asking about any data system you design or use.

The big idea

Applications today are mostly built by wiring together standard pieces: a database to store data, a cache to speed up reads, a search index, a queue for async work, a batch job to crunch numbers later. The interesting engineering problem usually isn't inventing new pieces, it's combining them well. Before you can judge whether a design is any good, it helps to break "good" into three separate, checkable qualities: does it stay correct when something breaks (reliability), does it hold up as usage grows (scalability), and can people keep changing it safely over time (maintainability)? None of these are binary yes/no labels. They're angles you keep returning to as a system evolves.

Key terms

  • Data-intensive (vs. compute-intensive) — an application where the hard part is the amount, complexity, or rate of change of the data, not raw CPU power. Most web and business apps are this way; a video encoder or a physics simulator is more compute-intensive.
  • Fault — one component of a system deviating from its spec. A disk dying, a process crashing, a person typing the wrong command.
  • Failure — the system as a whole stops providing the service the user needed. A fault and a failure are not the same event; a fault only becomes a failure if nothing stops it from spreading.
  • Fault-tolerant / resilient — a system built to keep working even when certain kinds of faults happen. Nothing can tolerate every possible fault, so this always means tolerant of specific, chosen kinds of faults.
  • Load parameters — the small set of numbers you use to describe how much a system is being asked to do right now (requests per second, read/write ratio, number of concurrent users, cache hit rate). Which numbers matter depends on your system.
  • Throughput — how many things you can process per second (common in batch systems, e.g. records processed per second).
  • Response time — what the client actually experiences: the time from sending a request to getting a response back, including network delay, queueing, and the actual work.
  • Latency — specifically the time a request spends waiting to be handled, before work on it starts. Often used loosely as a synonym for response time, but it's really one piece of it.
  • Percentile (p50, p95, p99...) — sort every response time from fastest to slowest; the p50 (median) is the value halfway down that list, p95 is the value 95% of the way down, and so on. "p99 = 800ms" means 99 requests out of 100 finished in under 800ms.
  • Tail latency — the high percentiles (p99, p999). The slowest requests, which are rare individually but matter a lot in aggregate.
  • Tail latency amplification — when one user request depends on several backend calls, the chance that at least one of them is slow goes up with every extra call, so a small tail gets amplified into a bigger share of slow end-user requests.
  • SLA / SLO — a service level agreement/objective: a stated performance and availability target (e.g. "median under 200ms, p99 under 1s, up 99.9% of the time"), often with consequences if it's missed.
  • Scaling up vs. scaling out — scaling up (vertical) means moving to a bigger single machine; scaling out (horizontal) means spreading load across more, smaller machines. A "shared-nothing architecture" is the common name for a scaled-out design where machines don't share memory or disk.
  • Elastic — a system that can add resources automatically when it detects more load, as opposed to one a human scales manually.
  • Operability — how easy the system is for an operations team to keep healthy day to day.
  • Accidental complexity — complexity that comes from how something was built, not from the actual problem it's solving. The kind you can remove with a better abstraction, as opposed to complexity that's inherent to the problem itself.
  • Evolvability — how easy it is to change the system later as requirements change. Also called extensibility or modifiability.

How it works

What "data-intensive" actually means

A data-intensive application is one where the bottleneck is the data itself: how much of it there is, how complicated its structure is, or how fast it's changing, rather than how much CPU you can throw at it. Almost nobody builds a new database engine from scratch for a project; instead you assemble standard building blocks (a database to store things, a cache to speed up repeated reads, a search index, a message queue for async work, a batch job for periodic crunching) and wire them together with application code. The tricky part is that once you glue several of these tools together behind one API, you've effectively built your own small data system, and now you're on the hook for its correctness and performance the same way a database vendor would be. That's the frame for the rest of the material: reliability, scalability, and maintainability are the three lenses for judging whether that composite system is actually any good.

Reliability: faults vs. failures

Reliability means the system keeps doing the right thing, at the performance you expect, even when something goes wrong internally, including someone using it in an unexpected way or trying to abuse it. The key distinction worth burning into memory: a fault is one part going wrong (a disk dies, a process crashes). A failure is the whole system failing to do its job for the user. Because it's basically impossible to drive the chance of any fault to zero, the practical goal isn't "prevent every fault," it's "build mechanisms that stop a fault from turning into a failure." A fault-tolerant system is one built to absorb specific kinds of faults without the user ever noticing.

Hardware faults

Hard disks are typically rated with a mean time to failure (MTTF) of around 10 to 50 years. That sounds reassuring until you have thousands of disks: on a cluster of 10,000 disks, you should expect roughly one to fail on an average day. The usual first response is redundancy: RAID for disks, dual power supplies, backup generators, so a single dead component doesn't take the machine down while it's replaced. For a long time, redundancy on one well-built machine was enough for most applications. As systems have grown to use many more machines, and as some cloud platforms are willing to take an instance away without warning in exchange for flexibility elsewhere, there's been a shift toward designing software that expects whole machines to disappear, not just single components.

Software errors

Hardware faults tend to be random and mostly independent of each other. Software bugs are the opposite: they're systematic, and because the same buggy code often runs on every node, a software fault can hit many machines at once, which is why software faults are often behind the biggest outages. Common shapes: a bug triggered by one particular bad input that crashes every instance that sees it, a runaway process eating up shared CPU/memory/disk/bandwidth, a dependency that slows down or starts returning corrupted responses, or a cascading failure where one component's small hiccup trips a fault in the next component downstream. These bugs often sit dormant for a long time, quietly relying on some assumption about the environment that's usually true, until the day it isn't. There's no single fix; it's a combination of careful assumption-checking, thorough testing, isolating processes from each other, letting things crash and restart cleanly, and watching production behavior closely enough to catch drift.

Human errors

People build and operate these systems, and people make mistakes even with the best intentions. One cited study of large internet services found configuration errors by operators were the leading cause of outages, well ahead of hardware faults (which played a role in only 10-25% of the outages studied, so this is specific to that study rather than a fixed universal split). The mitigations that show up repeatedly: design interfaces that make the right thing easy and the wrong thing hard (without being so locked-down that people work around them), give people a safe sandbox to experiment in with real data without touching production, test thoroughly at every level, make it fast and easy to roll back a bad change, and invest in monitoring so problems surface early and get diagnosed fast.

Why bother, and a counterintuitive trick

Reliability isn't just for safety-critical systems. Bugs in ordinary business software cost lost productivity, and outages of ordinary online services cost real revenue and trust; think about someone who's stored years of family photos in your app and what a silent data-corruption bug would mean to them. It's fine to consciously trade away some reliability for speed on an early prototype or a very thin-margin product, as long as you're deliberately choosing that trade-off rather than drifting into it. One genuinely counterintuitive idea worth remembering: in an already fault-tolerant system, it can help to deliberately increase the fault rate on purpose, for instance by randomly killing processes in production, because that keeps the fault-handling code continuously exercised instead of sitting untested until the day it's needed for real. Netflix's Chaos Monkey is the well-known example of this idea in practice.

Scalability: coping with growth

A system that's reliable today can stop being reliable simply because load grew. Scalability is the word for a system's ability to cope with added load; it's not a single yes/no label you can slap on a system ("X is scalable" is a meaningless sentence on its own). The useful question is always "if load grows in this particular way, what are our options?"

Describing load

Before you can talk about growth, you need a compact way to describe the load you have right now. These are called load parameters: requests per second to a server, the ratio of reads to writes, how many users are active in a room at once, a cache hit rate. Which parameter actually matters depends entirely on your system's shape.

A useful illustration (data as published around November 2012, so treat the specific figures as a historical snapshot, not current numbers): Twitter's "post tweet" operation ran at roughly 4.6k requests/sec on average and over 12k/sec at peak, which on its own is a fairly easy write volume to handle. The real difficulty was "home timeline" reads (about 300k requests/sec) combined with fan-out: each user follows many people and is followed by many people. Twitter described two designs. Approach 1: store every tweet in one big collection, and build a user's timeline on read by joining across everyone they follow. Approach 2: precompute a "mailbox" cache per user, and when someone tweets, push that tweet into every follower's mailbox immediately. Approach 1 struggled under the read load, so they moved to approach 2, because reads (300k/sec) vastly outnumbered writes (4.6k/sec), so it made sense to shift the work to write time. The catch: a tweet from someone with tens of millions of followers means tens of millions of mailbox writes for that one tweet, so at the time, Twitter had moved toward a hybrid, fanning out normal users' tweets on write but fetching a small number of very-high-follower accounts' tweets on read instead. The lesson isn't "copy Twitter's exact design," it's that the right load parameter (here, the follower-count distribution) determines which trade-off is worth making, and that trade-off can keep evolving as the system matures.

Describing performance

Once you can describe load, you can ask what happens as it grows: either "if I hold resources fixed and load grows, how does performance change?" or "how much do I need to scale resources to hold performance steady as load grows?" Answering either needs performance numbers. In batch systems (think Hadoop-style jobs), throughput, records processed per second or total job runtime, is usually the number that matters. In online, request/response systems, what matters more is response time: the time between sending a request and getting a response back, which bundles the actual processing time with network delay and any time spent waiting in a queue. Latency, strictly, is just the waiting-to-be-handled part, though people often use it loosely to mean response time.

Even identical repeated requests to the same service won't all take exactly the same time. Response time is really a distribution, not a single number, and real-world distributions are usually right-skewed: a cluster of typical fast requests, and a long thin tail of slow ones caused by things like garbage collection pauses, a lost network packet needing retransmission, a context switch, or a page fault forcing a disk read.

The median (p50) is the halfway point of that sorted distribution: half of requests finish faster, half slower, which makes it a decent read on "what a typical user experiences." The mean (the plain average) is worse for this purpose, because it doesn't tell you how many people actually hit the slow end. To see how bad your worst cases are, look at high percentiles: p95, p99, p999. "p99 = 1.5s" means 99 out of 100 requests finished under 1.5s, and 1 in 100 took 1.5s or longer. These high percentiles are called tail latencies, and they matter more than they might seem to, since (per the source's own example) the customers hitting your slowest responses are often your heaviest users, i.e. the ones with the most data and the most history, which in a commerce context can mean your most valuable customers. Amazon has reportedly framed internal response-time targets around the 99.9th percentile for exactly this reason, though they've also found that chasing the 99.99th percentile stopped being worth the cost, since those extreme outliers are heavily influenced by random events outside your control and the returns diminish fast. Percentile targets like these are exactly what SLAs/SLOs are built from: "median under 200ms, p99 under 1s, up 99.9% of the time" is a typical shape.

A large share of high-percentile response time is often queueing delay: a server can only run so many things in parallel (bounded by CPU cores, say), so a small number of slow requests ahead in line can hold up faster requests behind them, an effect called head-of-line blocking. This is why response times should be measured on the client side, not just server-side, and why a load-testing client needs to keep firing requests at a fixed rate regardless of how slow previous responses were; a client that waits for each response before sending the next one artificially shrinks the queue compared to what would really happen under load, which skews the measurement.

There's also tail latency amplification: when serving one end-user request needs several backend calls (even done in parallel), the whole request is only as fast as the slowest of those calls. The more backend calls involved, the higher the odds that at least one of them lands in the slow tail, so a small individual tail probability can turn into a much bigger share of slow end-user requests overall.

Approaches for coping with load

An architecture sized for one order of magnitude of load usually won't cope with 10x that load, so fast-growing systems tend to need re-architecting fairly regularly, not just once. The classic dichotomy is scaling up (vertical: a bigger single machine) versus scaling out (horizontal: more, smaller machines, sometimes called a shared-nothing architecture). A single powerful machine is often simpler to reason about, but very intensive workloads eventually can't avoid scaling out because top-end hardware gets disproportionately expensive. In practice, most real architectures mix both. Some systems are elastic (they add resources automatically when load rises), which suits unpredictable traffic, while manually scaled systems trade that convenience for fewer operational surprises. One useful, historically common piece of advice: distributing a stateless service across machines is fairly straightforward, but taking a stateful data store from one node to a distributed setup adds real complexity, so a common (not universal) pattern has been to keep a database on a single node until cost or availability needs force a distributed setup. There's genuinely no one-size-fits-all "magic scaling sauce": the right architecture depends on your specific mix of read volume, write volume, data volume, complexity, and response-time requirements, built from general-purpose building blocks arranged to fit your actual load parameters, not someone else's.

Maintainability: operability, simplicity, evolvability

Most of the lifetime cost of software isn't writing it, it's everything that happens after: fixing bugs, keeping it running, adapting it to new platforms and use cases, paying down technical debt. Three design principles help keep that ongoing cost from spiraling:

Operability

Making it easy for an operations team to keep the system healthy day to day: good visibility into runtime behavior via monitoring, good support for standard automation tooling, not depending on any one specific machine so nodes can be pulled for maintenance without downtime, clear documentation of "if I do X, Y happens," sensible defaults with room to override them, self-healing where it makes sense but manual override where it doesn't, and generally predictable behavior with as few surprises as possible. As one saying puts it: good operations can work around flawed software, but good software can't run reliably under bad operations.

Simplicity

Small projects can be simple and clear; big ones tend to accumulate complexity that slows everyone down (sometimes called a "big ball of mud"). Making a system simpler doesn't necessarily mean cutting functionality; it usually means removing accidental complexity, complexity that came from how the thing was implemented rather than from the actual problem it solves. The main tool for this is a good abstraction: something that hides a lot of implementation detail behind a clean interface, and that can be reused across many use cases instead of being rebuilt each time. SQL is a good example: it hides on-disk data structures, concurrent access, and crash-recovery details behind a declarative query language. Good abstractions are genuinely hard to find, especially in distributed systems, but they're worth chasing because they let you pull complexity out of the main system into well-defined, separately reusable pieces.

Evolvability

Requirements almost never stay fixed. New facts get learned, priorities shift, new use cases appear, regulations change, growth forces architecture changes. Evolvability (also called extensibility or modifiability) is how easy it is to adapt the system to all of that later, and it's closely tied to simplicity: simple, well-abstracted systems are usually easier to change than tangled ones. It's the same instinct behind practices like test-driven development and refactoring, just applied at the scale of a whole data system (maybe several services with different characteristics) instead of a single file.

Why it matters

These three lenses show up constantly once you start looking for them: choosing a database isn't really "which one is fastest," it's "which reliability, scalability, and maintainability trade-offs fit what I'm building." When an outage happens, "was this a fault or did it become a failure, and why" is usually the most useful post-incident question. When someone reports a service "feels slow," asking for the p50 and p99 separately, not just an average, usually reveals a completely different story than the headline number suggests. And every time you're tempted to add a clever workaround instead of fixing the underlying design, that's the maintainability question asking to be answered honestly.

Remember this

  • A fault is one part breaking. A failure is the whole system failing the user. Good design is mostly about stopping the first from becoming the second.
  • You can't drive faults to zero, so aim to tolerate the ones you can and contain the rest.
  • Never trust a single average response time; look at the distribution, especially p95/p99, because that's where real users' pain actually lives.
  • "Scalable" isn't a property a system has in isolation, it only means something relative to a specific load parameter and a specific kind of growth.
  • Simplicity isn't about doing less, it's about removing complexity that came from the implementation rather than the actual problem.

Try it yourself

  • Fault injector. Wrap a function call so it randomly throws an error some percentage of the time (start at 20%), and add a retry-with-fallback around the caller. Done when: you can run it 100 times at a 30% fault rate and the caller never crashes, even though individual calls do.
  • Percentile meter. Write a script that calls a slow endpoint (or just sleep()s for a randomized duration) 200 times, records every timing, sorts them, and prints the mean, p50, p95, and p99. Done when: your printed p99 is clearly and noticeably higher than your mean.
  • Closed-loop vs. open-loop load test. Build two tiny load generators against the same slow endpoint: one that waits for each response before sending the next request, one that fires at a fixed rate regardless of response time. Done when: you can show the "waiting" client reports a lower p99 than the fixed-rate client, on the same server.
  • Redundancy demo. Run two identical copies of a tiny stateless service behind a basic round-robin proxy you write yourself, then kill one process mid-run. Done when: the client sees zero failed requests, just a brief pause, while one instance is down.
  • Tail latency amplification experiment. Simulate a request that fans out to N parallel backend calls, each independently slow with some fixed probability (say 5%), and measure how often the overall request is slow at N = 1, 5, and 20. Done when: you have a printed table showing the "at least one slow call" probability rising as N grows.
  • Rolling window percentile tracker. Keep an in-memory rolling window of the last 60 seconds of response times and recompute p50/p95 every 5 seconds. Done when: you can watch the printed percentiles visibly jump when you feed the window a short burst of artificially slow requests.

Go deeper

If one thing is worth re-reading slowly, it's the percentile section: the gap between "average response time" and "p99 response time" is one of those ideas that seems obvious once you see it and then quietly explains a huge number of "why does this feel slow even though our dashboard looks fine" incidents. A natural next question to sit with: for a system you use or run today, do you actually know its p99, or only its average?
