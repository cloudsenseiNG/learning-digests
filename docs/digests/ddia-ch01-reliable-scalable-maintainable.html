SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

In one line: what it actually means to call a data system "good," broken into three things you can design for: staying correct when things break, staying fast as more people and data show up, and staying workable for however many years it lives.

The big idea

Before you write a line of code, it helps to agree on what you're actually building toward. This chapter splits "a good system" into three separate goals: staying correct when things go wrong (reliability), staying fast as load grows (scalability), and staying easy to work on for however long it's alive (maintainability). None of these show up by accident. They're choices you make in how you design, test, and operate a system, not properties that appear on their own once the code compiles.

Key terms

  • Data-intensive application — an app where the hard part is the amount, complexity, or rate of change of the data, not raw CPU power. Most are built from a handful of standard blocks: databases, caches, search indexes, message queues, and batch or stream processors.
  • Fault — one component doing something other than what it's supposed to (a disk dies, a process crashes on bad input).
  • Failure — the system as a whole stops delivering the service its users expect. A fault only becomes a failure if nothing stops it from spreading.
  • Fault-tolerant / resilient — built to keep working correctly even when some parts have faults.
  • Load parameters — the small set of numbers that describe how hard a system is being used right now (requests/sec, read:write ratio, concurrent users), so you can reason about what happens if that number grows.
  • Throughput — how many things a system processes per unit time; the number that matters most for batch jobs.
  • Response time — what the client actually experiences: time to send a request and get a response, including network and queueing delays, not just processing.
  • Latency — the time a request spends waiting to be handled, before anyone starts working on it. The chapter is careful to note this isn't the same thing as response time, even though people use the words interchangeably.
  • Percentile (p50, p95, p99) — sort every response time you measured and ask "what value do X% of requests fall under?" p50 (the median) is the typical case; p95/p99 show how bad your slow outliers really are.
  • Operability, simplicity, evolvability — the three properties this chapter groups under "maintainability": easy to run, easy to understand, easy to change later.

How it works

What "data-intensive" actually means

For most applications today, raw compute power isn't the bottleneck. The real constraints are the volume of data, how complex it is, and how fast it's changing. Because of that, most data-intensive apps aren't built from scratch, they're assembled from standard pieces: a database to store data so it can be found later, a cache to remember expensive results, a search index for keyword lookups, a message queue for handing work off asynchronously, and batch processors for crunching accumulated data periodically.

The tricky part is that real applications rarely use just one of these. It's common to have a main database, a separate cache, and a separate search index all backing the same feature, with your own application code responsible for keeping them in sync (for example, making sure the cache gets invalidated correctly whenever the underlying data changes). Once you're stitching several tools together like that, you've effectively become the designer of a small custom data system, with your own correctness and performance guarantees to uphold, not just a consumer of someone else's database.

Reliability: staying correct when things go wrong

"Reliable" has a simple everyday meaning: the app does what the user expects, tolerates the user making mistakes, performs well under the load it was built for, and keeps unauthorized access out. Put together, reliability means continuing to work correctly, even when things go wrong.

The chapter draws a distinction worth holding onto: a fault is one component deviating from what it's supposed to do (a disk dying, a bug crashing a process). A failure is when the system as a whole stops serving its users. You can't push the probability of a fault to zero, no system can survive every conceivable disaster, so the realistic goal is to build fault-tolerance mechanisms that keep individual faults from turning into a full failure.

Interestingly, the chapter points out that fault-tolerant systems sometimes get more reliable by deliberately triggering faults, for example randomly killing processes on purpose (the Netflix Chaos Monkey is the chapter's example). The logic: a lot of critical bugs come from error-handling code that almost never runs, so exercising that path regularly builds real confidence it'll work when a fault happens for real, not just in theory.

Hardware faults are the ones people picture first: disks crash, RAM goes bad, someone unplugs the wrong cable. The chapter cites hard disks having a mean time to failure of roughly 10 to 50 years, which sounds fine for one disk, but on a cluster of 10,000 disks that works out to about one disk dying per day on average. The traditional fix is redundancy at the hardware level: RAID arrays, dual power supplies, backup generators, so a broken part can be swapped without the machine going down. That was enough for most applications historically. But as systems have grown to use far more machines, and as cloud platforms like AWS can retire a virtual machine with little warning (the platform is optimizing for flexibility, not single-machine reliability, per the chapter), more systems now lean on software fault-tolerance to survive losing an entire machine, not just a single component. A nice side effect: a system built to tolerate whole-node loss can be patched one node at a time (a rolling upgrade) instead of needing planned downtime.

Software errors are a different animal: unlike hardware faults, which tend to be random and uncorrelated, software bugs are systematic and can hit every node at once. The chapter's examples: a bug triggered by one specific bad input crashing every instance of a service at the same time (it cites a real 2012 leap-second bug that hung many Linux-based systems simultaneously), a runaway process quietly consuming shared resources like memory or disk, a dependency that slows down or starts returning bad data, and cascading failures where one component's fault trips a fault in the next. There's no single fix; what actually helps is a mix of things: questioning your assumptions about the environment, thorough testing, isolating processes from each other, letting a process crash and restart cleanly instead of limping along, and watching production behavior closely enough to catch drift early.

Human errors turn out to matter more than most people assume. The chapter cites a study of large internet services where configuration mistakes by operators were the leading cause of outages, with hardware faults responsible for only 10 to 25% of them. The mitigations the chapter lays out: design APIs and admin tools that make the right thing easy and the wrong thing awkward (without being so locked-down that people route around them); give people a full-featured sandbox to experiment in with real data, away from production; test at every level, unit tests up through full manual walkthroughs; make recovery fast (quick rollbacks, gradual rollouts so a bad change only touches a slice of users, tools to recompute data if a past calculation turns out wrong); and invest in real monitoring and telemetry so problems surface early instead of being discovered by users.

Reliability isn't just for "serious" systems like flight control software. The chapter's own reminder: think about a parent who stores every photo and video of their kids in your app. What happens to their trust if that database silently corrupts? It's fine to consciously trade some reliability for speed while validating an early prototype, the chapter says as much, but that should be a deliberate call, not something that just happens because nobody thought about it.

Scalability: coping with growth

A system that works fine today might not hold up once load changes, more users, more data, more complexity. Scalability is the term for a system's ability to cope with that growth. The chapter is specific that it isn't a single label you can slap on a system ("X is scalable" doesn't really mean anything on its own); the useful question is always "if this system grows in a particular way, what are our options for handling it?"

To even ask that question, you first need a way to describe the current load using a few numbers, called load parameters. What they are depends entirely on your system: requests per second to a web server, the ratio of reads to writes, the number of people active in a chat room at once, a cache's hit rate. The chapter illustrates this with Twitter's 2012 numbers: posting a tweet ran around 4.6k requests/sec on average (peaking near 12k), while reading a home timeline ran around 300k requests/sec, nearly two orders of magnitude more. So the real scaling problem wasn't the tweet-writing volume, it was fan-out: every user follows many people and is followed by many people.

The chapter walks through two ways to serve a home timeline. One: keep tweets in one big table and, at read time, join across everyone a user follows and merge the results by time. That's simple, but reading becomes expensive as follow-graphs grow. Two: precompute each user's timeline ahead of time, like a personal mailbox, inserting a new tweet into every follower's mailbox the moment it's posted. That makes reads cheap, but writes get much more expensive and wildly uneven: a tweet from someone with millions of followers means millions of mailbox writes for that single tweet. Twitter actually started with the first approach, moved to the second once reads became the bottleneck, and eventually landed on a hybrid: fan-out on write for ordinary accounts, fan-out on read for the small number of accounts with huge followings. Simplification: I've compressed the exact numbers and timeline here; the point to take away is the trade-off (cheap writes and expensive reads, or the reverse), not the specific figures.

Once you can describe load, you need a way to describe performance under that load. In batch systems (think a nightly data job) the number that matters is throughput, records processed per second, or total time to finish a job. In online systems, what matters to the person waiting is response time, the total time between sending a request and getting a response back. The chapter draws a careful line here: response time is what the client experiences (processing time, plus network delay, plus time spent waiting in a queue); latency, strictly, is just the waiting part, the time a request sits idle before anyone starts on it. People use the words as if they're interchangeable, but they aren't quite.

Here's the part that reshaped how I think about performance: even if you send the exact same request over and over, you won't get the same response time twice. Some variation is expected (bigger requests cost more), but even identical requests vary because of things outside your control: a context switch, a dropped network packet needing a retransmit, a garbage-collection pause, a page fault. So response time isn't one number, it's a distribution, and the way most people summarize a distribution (the average) is a poor fit here. The mean hides how many people actually experienced the slow end.

Percentiles are the better tool. Sort every response time you measured from fastest to slowest. The median (also called p50) is the halfway point: half your requests finished faster than it, half slower, so it's a solid answer to "how long does a typical user wait." To see how bad your worst cases are, look further up the sorted list: p95, p99, and p999 mark the response time below which 95%, 99%, or 99.9% of requests finished. These slow-end percentiles are called tail latencies, and the chapter argues they matter more than they might seem to, because the customers hitting your slowest responses are often your heaviest users, the ones with the most data and the most history with your product, which usually makes them your most valuable customers too.

This is also why SLOs and SLAs (service level objectives/agreements, contracts spelling out expected performance) are usually written in percentiles, something like "median under 200ms, p99 under 1 second, up at least 99.9% of the time," rather than in averages. And a lot of that slow tail comes from queueing: a server can only run a limited number of things in parallel, so once a few requests are slow, everything queued behind them gets delayed too (an effect called head-of-line blocking) even if those later requests would've been fast on their own. That's part of why the chapter insists you measure response time on the client side, and why a load-testing tool has to keep firing requests on a fixed schedule instead of waiting for each response before sending the next, otherwise you'll understate how deep your queues really get under real load.

There's a compounding effect worth knowing about too: when a single user request depends on several backend calls, even done in parallel, the overall request is only as fast as its slowest call. So even if each backend is individually fast 99% of the time, needing five of them in parallel measurably raises the odds that at least one is slow, which the chapter calls tail latency amplification. And computing percentiles cheaply and continuously (say, over a rolling 10-minute window for a dashboard) is a real engineering problem: naively re-sorting the whole window on every update is wasteful, which is why the chapter points to approximation algorithms like forward decay, t-digest, and HdrHistogram. One more gotcha it flags: averaging percentiles together (say, across machines or time windows) is not mathematically meaningful. The right way to combine them is to add up the underlying histograms first, then recompute the percentile.

Once you can describe load and performance, you can talk about actually coping with growth. Scaling up (vertical: move to a beefier machine) and scaling out (horizontal: spread load across many smaller machines, sometimes called a shared-nothing architecture) are the two broad levers, and most real systems end up mixing both rather than picking one purely. Some systems are elastic (auto-add resources when load spikes), others are scaled manually by a human deciding when to add capacity; manual scaling is simpler and has fewer surprises, elastic scaling helps more when load is genuinely unpredictable.

The chapter also flags something easy to miss: distributing stateless services is fairly straightforward, but taking a stateful data system (a database holding the actual data of record) from one node to many adds real complexity, which is why common wisdom has long been "keep the database on one node until cost or availability needs force you off it." And there's no such thing as a generic, one-size-fits-all "magic scaling sauce": architecture has to be built around the load parameters that are actually common for your application, and if those assumptions turn out wrong, the engineering effort spent scaling for them is wasted at best and actively counterproductive at worst. That's part of why an early-stage product usually gets more value from iterating quickly than from pre-building for a scale it may never reach.

Maintainability: designing for the system's whole life, not just launch day

Most of the cost of software isn't writing the first version, it's everything after: fixing bugs, keeping it running, investigating incidents, adapting it to new platforms and use cases, paying down technical debt, and shipping new features on top of it all. The chapter organizes "designing for that reality" into three principles.

Operability means making it easy for the people running the system to keep it running smoothly. A good operations team's job, per the chapter, spans monitoring system health and restoring it quickly when something's wrong, tracking down root causes, keeping software patched, understanding how changes in one system ripple into others, planning capacity ahead of need, maintaining good deployment and configuration practices, handling complex migrations, keeping the system secure as it changes, and preserving institutional knowledge as people come and go. A system helps its own operators when it offers real visibility (good monitoring), plays nicely with standard automation tooling, doesn't depend on any one specific machine staying alive, documents a predictable "if I do X, Y happens" model, ships sensible defaults while still letting an administrator override them, can self-heal but still allows manual control when needed, and generally behaves predictably instead of surprising people.

Simplicity, here, isn't about the user interface, it's about how easy the system is for an engineer to actually understand. Small projects tend to start clean; as they grow, they tend to accumulate tangled dependencies, inconsistent naming, one-off hacks, and special-case logic (the chapter calls a system that's drowned in this a "big ball of mud"). That complexity isn't free: it slows down everyone who has to work on the system and makes it easier to introduce bugs, because nobody can hold every hidden assumption in their head at once. The chapter draws a useful line between essential complexity (inherent to the actual problem you're solving) and accidental complexity (created purely by how it happens to be implemented). The best defense against accidental complexity is a good abstraction, hiding messy implementation detail behind a clean interface. High-level programming languages hiding machine code, or SQL hiding the on-disk data structures and concurrency control underneath, are the chapter's own examples of this working well.

Evolvability is about how easy it is to change the system later, and the chapter is blunt that your requirements are basically guaranteed to keep shifting: new use cases show up, the business changes direction, regulations change, growth forces architectural rethinks. Small-scale Agile practices like test-driven development and refactoring already address this at the level of a single codebase; the chapter borrows the same instinct and applies it to an entire data system, potentially spanning several services, and that's specifically what it means by evolvability. It's closely tied to simplicity: a system built on clean abstractions is a system you can actually afford to change.

Why it matters

These three ideas show up the moment you touch a real system, not just in a book about them. Debugging a 3am page is living inside the gap between a fault and a failure. Watching your user count go from a thousand to a million is living inside the scalability chapter. Opening a codebase someone else wrote two years ago and not being able to figure out what it does is living inside the maintainability chapter. Most of the rest of this book, and honestly a lot of distributed-systems knowledge in general, is really this chapter zoomed in on specific techniques.

Remember this

  • A fault is one broken part; a failure is the whole system letting the user down. Good design is about stopping the first from becoming the second.
  • You can't get fault probability to zero, so aim to contain faults (redundancy, isolation, fallbacks) instead of assuming they won't happen.
  • Averages hide your worst experiences. Percentiles (p50, p95, p99) show you what a typical user feels and how bad your worst moments actually are.
  • "Scalable" isn't a property a system has or doesn't have; it's a question: what happens as a specific load parameter grows, and what are the options?
  • Most of a system's real cost is paid after launch. Simplicity and good abstractions are what make years of maintenance survivable.

Try it yourself

  1. Build a tiny fault injector. Write a small HTTP server with 2–3 endpoints in any language, then wrap it with a proxy or middleware that randomly delays, drops, or kills a percentage of requests. Done when: you can flip fault injection on/off with a flag, and a log shows which requests hit the fault versus which succeeded.
  2. Build a percentile meter. Generate 1,000 synthetic response times (mostly 50–150ms, with an occasional 2–5 second outlier mixed in on purpose), then sort them yourself and compute the mean, p50, p95, and p99 without a stats library. Done when: you can print all four numbers side by side and explain in one sentence why the mean is misleading here.
  3. Build a two-node redundancy demo. Run two copies of a trivial server behind a small proxy script that forwards to whichever instance is alive, then kill one mid-run. Done when: a client hitting the proxy in a loop keeps getting successful responses through the kill, with only that instant's request (if any) failing.
  4. Write down your own app's load parameters. Pick any app you use or are building and list 3–5 numbers that would actually matter if it had to scale (requests/sec, read:write ratio, concurrent users, whatever fits). Done when: you have a short list someone else could read and immediately understand what "10x growth" would mean for that app.
  5. Simulate tail latency amplification. Write a function that "calls" 5 simulated backend services in parallel, each with its own random latency (say 95% fast, 5% slow), and measure how often the combined result is slow. Done when: your numbers show the combined slow-rate is higher than any single backend's slow-rate, and you can explain why in one sentence.

Go deeper

If one part is worth re-reading slowly, it's "Percentiles in Practice." The idea that you can't average percentiles together, you have to combine the underlying histograms first, trips people up even after they've fully bought into why percentiles beat averages. A natural next question once you've built the percentile meter above: how would you keep that calculation cheap if you had to update it continuously over a rolling 10-minute window of live traffic, instead of recomputing it from scratch every time?
