SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What we actually mean when we call a system "good," and the three qualities worth designing for on purpose instead of hoping for.

In one line

Before you can build a good data system you need a shared vocabulary for "good," and this chapter hands you three words for it: reliable, scalable, and maintainable.

The big idea

Most applications today are what the chapter calls data-intensive: the hard part isn't raw CPU power, it's the amount of data, how complicated that data is, and how fast it's changing. You build these apps out of familiar pieces (a database to store things, a cache to speed up reads, a search index, a queue, a batch job), and the chapter argues that whatever you build, you're really chasing three properties: it should keep working when things go wrong (reliability), it should cope as usage grows (scalability), and it should stay pleasant for people to work on for years (maintainability). None of these are things you get by accident. They're design goals you have to think about on purpose, and this chapter is really just building you a precise vocabulary for each one before the rest of the material dives into the how.

Key terms

Data-intensive application — an app where the limiting factor is data volume, data complexity, or the rate data changes, not raw compute power.
Fault — one component of the system deviating from what it's supposed to do (a disk dies, a process crashes).
Failure — the system as a whole stops giving the user the service they expected. A fault doesn't automatically become a failure.
Fault-tolerant / resilient — a system that keeps working correctly even when specific, anticipated faults happen. Nobody can build tolerance for every conceivable fault (there's no budget for "the planet gets swallowed by a black hole"), so in practice this always means tolerant of certain kinds of faults, not all of them.
MTTF (mean time to failure) — the average time a hardware part is expected to run before it breaks. Used for disks, power supplies, etc.
Redundancy — having a backup component ready to take over when the primary one fails (a second power supply, a spare disk in a RAID array).
Chaos engineering — deliberately triggering faults on purpose (like Netflix's Chaos Monkey randomly killing processes) so your fault-tolerance code gets exercised constantly instead of only during a real emergency.
Load parameters — the small set of numbers you use to describe how much traffic or data a system is handling right now: requests per second, read/write ratio, concurrent users, cache hit rate, whatever fits your system.
Fan-out — how many downstream things one incoming request triggers (a term borrowed from electronics). One tweet fanning out to millions of followers' feeds is an extreme case.
Throughput — records or jobs processed per unit of time. The main performance number that matters for batch jobs.
Response time — what the client actually experiences: time to process the request, plus network delays, plus time spent waiting in a queue.
Latency — strictly, the time a request spends waiting before it's handled (queueing time), which is only part of response time. People often use "latency" and "response time" loosely as synonyms, so watch context.
Percentile (p50 / p95 / p99 / p999) — sort every response time from fastest to slowest; the 50th percentile (p50, the median) is the halfway point, p95 is the point where 95% of requests were faster, and so on. High percentiles are called tail latencies.
SLO / SLA — a service level objective / agreement: a contract or internal target that states expected performance and uptime (e.g. "p50 under 200ms, p99 under 1s, up 99.9% of the time").
Head-of-line blocking — a server can only run a few things in parallel, so one slow request stuck at the front of the line delays every fast request queued behind it.
Tail latency amplification — if a single user request needs several backend calls, the odds that at least one of them lands in the slow tail go up, so more end-user requests end up slow than the raw per-call slow rate would suggest.
Scaling up (vertical) / scaling out (horizontal) — moving to a bigger, more powerful machine, versus spreading the load across many smaller machines.
Shared-nothing architecture — a horizontally scaled setup where each machine doesn't share memory or disk with the others; they coordinate over the network instead.
Elastic system — one that adds computing resources automatically when it detects more load, as opposed to a human deciding to add machines by hand.
Operability — how easy the system is for an operations team to keep running smoothly day to day.
Accidental complexity — complexity that comes from how a system happens to be built, not from the actual problem it's solving. The opposite is essential complexity, which is inherent to the problem itself.
Abstraction — hiding messy implementation detail behind a clean interface (SQL hiding on-disk data structures is the chapter's example).
Evolvability — how easy it is to change the system later as requirements change. Also called extensibility, modifiability, or plasticity.

How it works

Before getting into any of the three qualities, it's worth sitting with what "data-intensive" means, because it explains why this book (and this chapter) exists. If your app's bottleneck were CPU cycles, you'd mostly be optimizing algorithms. But for most real applications today, the bottleneck is the data itself: how much of it there is, how tangled its structure is, and how quickly it's changing underneath you. That's why data-intensive apps get built out of a standard toolkit (databases, caches, search indexes, queues, batch jobs) stitched together with application code, and why the chapter spends the rest of its pages defining, precisely, what "working well" means for a system like that. It lands on three goals: reliability, scalability, and maintainability. Each one gets its own section below.

Reliability: faults vs. failures

Reliability, in plain terms, means "the system keeps doing the right thing, even when things go wrong." The trick is that "things going wrong" is guaranteed, not hypothetical, so the chapter draws a sharp line between two words people usually use interchangeably:

  • A fault is one component misbehaving: a disk dies, a service returns garbage, a process crashes.
  • A failure is the user-facing outcome: the system as a whole stops doing its job for the person using it.

The goal isn't zero faults (that's not achievable), it's designing so that faults don't turn into failures. The chapter walks through three sources of faults:

Hardware faults. Disks are reported with a mean time to failure (MTTF) of roughly 10 to 50 years. That sounds long until you have 10,000 disks in a storage cluster, at which point you should expect to lose about one disk a day, on average. The traditional fix is redundancy: RAID for disks, dual power supplies, backup generators. That still doesn't stop all hardware failures, but it buys years of uninterrupted uptime in a common on-prem setup. Worth noting: on some cloud platforms, virtual machines can simply disappear without warning because the platform is optimized for flexibility over any one machine staying up, which is one reason more systems now lean on software-level fault tolerance (redundancy that spans whole machines) rather than only hardware redundancy inside one box.

Software errors. These are systematic rather than random: a bug that crashes every instance of a service given the same bad input, a runaway process eating shared memory or CPU, cascading failures where one component's hiccup triggers another's. They're nastier than hardware faults because they can hit many machines at the same instant, since it's the same buggy code everywhere. The chapter's real example: a leap-second bug in the Linux kernel on June 30, 2012 caused many applications to hang simultaneously. There's no silver bullet here, just a pile of practices that help: careful thinking about assumptions, thorough testing, isolating processes so one crash doesn't take others down, and constantly monitoring for the system quietly breaking its own guarantees.

Human errors. People configure and operate these systems, and people make mistakes. The chapter cites a study (Oppenheimer, Ganapathi & Patterson, 2003) that found configuration errors by operators were the leading cause of outages at the internet services studied, with hardware responsible for only 10 to 25% of outages. Simplification flagged: that's one study of a specific set of services from over two decades ago, not a universal law about all outages everywhere, but the underlying instinct (assume humans will make mistakes, design for it) holds up broadly. Good defenses combine several things: interfaces that make the right thing easy and the wrong thing awkward, sandbox environments to experiment safely with real data away from production, testing at every level, fast and easy rollback when something does go wrong, and detailed monitoring (telemetry) so problems surface early instead of silently.

One counterintuitive move the chapter highlights: in a genuinely fault-tolerant system, it can help to increase the fault rate on purpose, by randomly killing processes without warning (Netflix's Chaos Monkey is the canonical example). Since a lot of critical bugs come from bad error handling that only shows up when a fault actually occurs, deliberately triggering faults keeps that machinery exercised, so it actually works when a real fault shows up unannounced.

Reliability isn't just for "critical" systems like air traffic control. The chapter's example: think about a parent whose only copies of their kids' photos live in your app's database. If it gets corrupted, that's not a minor bug for them. There are legitimate reasons to trade away some reliability, like an early prototype where speed of iteration matters more, but the chapter's point is that this should be a conscious trade-off, not a default.

Scalability: describing load, then describing performance

Scalability is a system's ability to cope with increased load, and the chapter is emphatic that it's not a single yes/no label. "Is X scalable?" is close to meaningless on its own; the real question is "if load grows in a specific way, what are my options for handling that?" Answering that starts with being able to describe load precisely, using load parameters: a handful of numbers that fit your specific system (requests per second, read/write ratio, concurrent users, cache hit rate, whatever's relevant).

The chapter walks through Twitter as a worked example, using data the company published in November 2012 (worth flagging: these are more than a decade old and used purely to illustrate the reasoning, not as current Twitter/X numbers). Two operations dominate: posting a tweet (around 4.6k requests/second on average, spiking past 12k/second) and reading a home timeline (about 300k requests/second). The naive design does a database join at read time: look up who a user follows, grab their recent tweets, merge them by time. That's simple, but it buckles under 300k reads/second when each read triggers a join across potentially thousands of followed accounts. So the actual bottleneck isn't tweet volume, it's fan-out: every followed user's new tweet has to reach every follower's timeline somehow.

The alternative flips the work to write time: maintain a precomputed "inbox" cache per user, and when someone tweets, push that tweet into every follower's cache immediately. Reads become cheap (just read the cache), but writes get expensive: an average tweet reaching about 75 followers turns 4.6k tweets/second into roughly 345k cache writes/second. And averages hide the real problem, because some accounts have tens of millions of followers, so a single tweet from one of those accounts can mean tens of millions of writes that all need to land within seconds. Twitter's actual answer, per the chapter, ended up a hybrid: fan out at write time for ordinary accounts, but fetch high-follower ("celebrity") accounts' tweets at read time and merge them in, since those accounts' write-time fan-out would be far too expensive to redo on every tweet. Simplification flagged: this is one company's solution to one specific load shape at one point in time, not a universal rule that write-time fan-out always beats read-time joins. The generalizable lesson is the reasoning process (identify your dominant load parameter, then pick where to pay the cost), not the specific fan-out design.

Once you can describe load, you need a way to describe performance under that load. For batch systems, that's usually throughput (records processed per second, or total job runtime). For online, request-driven systems, it's usually response time: the full time between a client sending a request and getting a response back, which includes actual processing time, network delays, and time spent waiting in a queue. The chapter is precise that latency, strictly, refers only to the waiting-to-be-handled part, even though people often say "latency" when they mean the whole response time.

Response time isn't one number, it's a distribution: run the exact same request repeatedly and you'll still see it vary, because of context switches, garbage collection pauses, network hiccups, and more. Reporting a single "average" response time hides this. A better tool is percentiles: sort every response time fastest to slowest, and the median (p50) is the halfway point, meaning half of requests were faster and half were slower. Higher percentiles like p95, p99, and p999 show you how bad the slow outliers actually get, and those high percentiles are called tail latencies. The chapter's example: Amazon reportedly tracks internal services by their p999, even though that's technically only 1 request in 1,000, because the customers hitting that slow tail tend to be the ones with the most data on their account, which often means the most purchase history, which often means the most valuable customers. Simplification flagged: that's Amazon's specific observation about its own customer base and access patterns, not a claim that every business's slowest users are automatically its best users; it's a reason to actually go check your own tail before assuming it doesn't matter.

Two more things worth carrying forward. First, queueing delays often dominate response time at high percentiles: a server can only run a handful of things in parallel, so a few slow requests at the front of the line hold up everything behind them, an effect called head-of-line blocking. Second, tail latency amplification: if serving one user request requires calling several backend services (even in parallel), the whole request is only as fast as the slowest of those calls, so the more backend calls involved, the more likely at least one of them lands in the slow tail, and the more end-user requests end up slow overall.

Approaches for coping with load

Once you can describe load and performance, the practical question becomes: how do you add capacity? The chapter frames this as a spectrum between scaling up (vertical scaling: move to a bigger, more powerful machine) and scaling out (horizontal scaling: spread the load across more, smaller machines, sometimes called a shared-nothing architecture because those machines don't share memory or disk). A single powerful machine is often simpler to reason about, but high-end hardware gets expensive fast, so very demanding workloads usually end up scaling out at least partly. In practice, most real architectures mix both rather than picking one extreme.

Systems can also be elastic (automatically adding resources when load increases) or manually scaled by a human deciding when to add machines. Elastic systems help when load is unpredictable, but manual scaling tends to be simpler and has fewer operational surprises. Simplification flagged: stateless services (like web servers with no local data) are relatively straightforward to distribute across machines, but stateful systems (databases holding your actual data) get meaningfully more complex once you go from one node to many, which is why, historically, common practice was to keep a database on a single machine for as long as possible before distributing it.

The chapter is blunt that there's no "magic scaling sauce," no generic architecture that scales every kind of system. A system built for 100,000 requests/second at 1kB each looks nothing like one built for 3 requests/minute at 2GB each, even though both push the same total bytes per second. A scalable architecture is built around assumptions about which load parameters actually matter for you, and if those assumptions turn out wrong, all that engineering effort was wasted (or worse, counterproductive). For an early-stage product with unproven usage, the chapter's advice is that iterating quickly usually matters more than pre-building for hypothetical future scale.

Maintainability: operability, simplicity, evolvability

Most of the lifetime cost of software isn't writing it, it's maintaining it: fixing bugs, keeping it running, adapting it to new requirements, paying down technical debt. The chapter breaks "make maintenance less painful" into three design goals:

Operability is about making life easy for whoever keeps the system running day to day. A good operations team monitors system health, tracks down the causes of problems, applies security patches, plans for future capacity, and preserves institutional knowledge about how the system behaves. Systems can help their own operability by exposing good visibility into runtime behavior, supporting standard automation tooling, not depending on any single machine staying up, documenting a predictable operational model ("if I do X, Y happens"), and generally behaving predictably instead of surprising the humans running it.

Simplicity here doesn't mean a simple user interface, it means keeping the system's internals easy for a new engineer to understand. As projects grow, complexity tends to creep in: tangled dependencies, inconsistent naming, one-off hacks layered on top of each other (sometimes called a "big ball of mud"). The chapter draws a useful distinction between essential complexity (inherent to the actual problem you're solving) and accidental complexity (complexity that crept in purely from how the system happens to be implemented, not from the problem itself). The best tool for cutting accidental complexity is abstraction: hiding messy implementation detail behind a clean interface. SQL is the chapter's example, since it hides on-disk data structures, concurrent access, and crash-recovery logic behind a small query language.

Evolvability is how easily you can change the system as requirements shift, and the chapter treats this as effectively guaranteed to matter, since requirements essentially never stay frozen: new use cases emerge, regulations change, the business changes direction. Evolvability is closely tied to simplicity, since a system that's easier to understand is generally easier to safely modify. It's also the reason the chapter singles out the Twitter fan-out example earlier: if approach 1 (read-time joins) and approach 2 (write-time fan-out) are both valid designs for the same feature, how easily could you actually migrate a live system from one to the other? That's evolvability in practice, not in theory.

Why it matters

These three words (reliable, scalable, maintainable) aren't just chapter vocabulary, they're the lens the rest of the material keeps coming back to. Every design decision in later chapters (which database to pick, how to replicate data, how to partition it) gets evaluated against these same three questions: does this keep working when something breaks, does this cope as load grows, and does this stay workable for the people maintaining it. It's also a genuinely useful lens outside any book: the next time you're debugging an outage or reviewing an architecture decision at work, "was this a fault that became a failure, and why," "what load parameter actually changed," and "did we just add accidental complexity" are all real, answerable questions.

Remember this

  • A fault is one part breaking. A failure is the user actually feeling it. Good design is mostly about stopping the first from becoming the second.
  • "Is it scalable?" is the wrong question. "Scalable for which load parameter, and to what level?" is the right one.
  • Averages lie. Percentiles (p50, p95, p99) show you the tail, and the tail is where your unhappiest users are sitting.
  • Maintainability isn't one thing, it's three: keeping it running (operability), keeping it understandable (simplicity), keeping it changeable (evolvability).
  • There's no generic "scalable architecture." Every real design is built around assumptions about your specific load, and those assumptions need checking.

Try it yourself

  • Fault injector. Write a tiny script that calls a function 100 times, and on a random 10% of calls, throws an exception instead of returning normally (a stand-in fault). First run it with no handling and watch it crash the whole loop (a failure). Then wrap each call in try/catch with a retry-once policy and see how many "faults" get contained before they become a "failure."
  • Percentile meter. Generate (or collect from a real API) a list of 200+ response times. Sort them and compute p50, p95, and p99 by hand (index math, no library). Then compute the mean too, and compare: how far off is the mean from p95? Try it again with a second, more skewed distribution and see how much percentiles move versus the mean.
  • Mini load test. Write a dummy "server" function with an artificial random delay (e.g. 50-200ms, occasionally spiking to 2s). Hit it with 1, then 10, then 50 concurrent callers and log response times for each batch. Plot how the tail (not the average) changes as concurrency rises.
  • Read-time vs. write-time fan-out. Build two tiny functions for computing a user's "feed" from an in-memory list of posts and a follow graph: one that joins at read time, one that maintains a precomputed per-user inbox updated at write time. Simulate one "celebrity" account with far more followers than everyone else and time both approaches.
  • Redundancy simulator. Model N independent components, each with the same small random chance of failing on any given check. Compute how often "at least one of N is up" happens for N = 1, 2, 3, 5, and 10. Notice how fast the odds improve, and where they start flattening out.
  • Head-of-line blocking demo. Build a queue that processes one job at a time. Feed it a mix of fast (10ms) and occasional slow (2s) jobs, and measure how much the fast jobs stuck behind a slow one get delayed, even though their own processing time never changed.

Go deeper

If you want to push past this digest, re-read the "Percentiles in Practice" portion of the chapter and think through the practical problem it raises: how would you cheaply keep a rolling p99 over the last 10 minutes of traffic without re-sorting a huge list every minute? (The chapter name-drops forward decay, t-digest, and HdrHistogram as real approaches, worth a search on their own.) A natural next question heading into later chapters: what happens when a fault isn't a clean crash, but a node that keeps running and responding, just wrongly, without telling anyone?
