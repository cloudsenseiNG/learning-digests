SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What it actually means to call a system "good", and why those three words are choices you design for, not luck.

In one line

Before you touch any specific database or tool, it helps to agree on what you're even aiming for: a system that keeps working when things go wrong (reliable), keeps working as load grows (scalable), and stays easy for people to work on for years (maintainable).

The big idea

Most apps today are limited by data, not CPU: how much of it there is, how complicated it is, and how fast it's changing. So instead of writing your own storage engine, you stitch together tools that already exist (a database, a cache, a search index) and your application code glues them together. That makes you a data system designer, not just an app developer, and it means you have to think about three things on purpose: will it keep working when a part breaks, will it keep working when way more people show up, and will the next engineer who touches it be able to understand and change it. This chapter is Kleppmann setting up vocabulary for those three ideas so the rest of the book can build on them.

Key terms

  • Data-intensive application — an app where the hard part is the volume, complexity, or rate of change of data, not raw compute power.
  • Fault — one component of the system deviating from what it's supposed to do. Example: a single hard disk dying.
  • Failure — the system as a whole stops delivering the service the user expects. A fault doesn't automatically become a failure.
  • Fault-tolerant / resilient — a system built to keep working correctly even while some faults are happening. Nobody can tolerate every possible fault, so this really means "tolerant of the specific faults we planned for."
  • Load parameters — the numbers you use to describe how much load a system is under, e.g. requests per second, ratio of reads to writes, number of active users.
  • Throughput — how many records or jobs you process per unit of time. The main thing you care about in batch systems.
  • Response time — what the client actually experiences: the time to process the request, plus network delay, plus any time spent waiting in a queue.
  • Latency — strictly, the time a request spends waiting before it's handled (not the whole response time). People often use the two words loosely, but the book draws this distinction on purpose.
  • Percentile (p50 / p95 / p99) — sort every response time from fastest to slowest; p50 (the median) is the halfway point, p95 means 95 out of 100 requests were faster than that value, p99 means 99 out of 100 were.
  • Tail latency — the slow end of that sorted list (the high percentiles). It's small in count but large in impact on real users.
  • Scaling up vs. scaling out — scaling up (vertical) means moving to a bigger machine; scaling out (horizontal) means spreading the load across more machines.
  • Operability, simplicity, evolvability — the three facets the chapter uses to break down maintainability. More on each below.

How it works

Reliability: keep doing the right thing, even when something breaks

Reliability is basically "continuing to work correctly, even when things go wrong." The chapter is careful to split two words that sound almost interchangeable: a fault is one part misbehaving (a disk dies, a process crashes), and a failure is the whole system letting the user down. Good design is mostly about stopping faults from turning into failures, because you can't get the fault rate to zero. There's always going to be a broken disk somewhere.

The chapter groups faults into three buckets, and they behave really differently:

  • Hardware faults are mostly random and independent (one disk dying doesn't make another disk more likely to die). The book cites hard disks having a typical lifespan (mean time to failure) of 10 to 50 years, which sounds fine until you have 10,000 disks in one storage cluster, at which point you should expect roughly one to die on an average day. The standard fix is redundancy: RAID, dual power supplies, backup generators, so a broken part can be swapped without the system going down.
  • Software faults are systematic, not random. A bug is either triggered by an input or it isn't, and because the same buggy code often runs on every node, one bad input can crash every instance at once. These are harder to guard against because they're correlated instead of independent.
  • Human errors turn out to be the biggest source of real-world outages. The book cites a study of large internet services where configuration mistakes by operators were the leading cause of outages, and hardware faults were only involved in 10-25% of them. The fixes here are less about code and more about process: good abstractions and admin tools that make the right thing easy, sandbox environments to practice in, thorough testing, fast rollback, and solid monitoring so you catch a bad change quickly.

Concrete example (outside the book): on February 28, 2017, S3 in AWS's us-east-1 region (Northern Virginia) went down for several hours. According to AWS's own incident summary, an authorized engineer was running a command as part of debugging a slow billing subsystem, meant to remove a small number of servers. One of the inputs to that command was entered incorrectly, and it removed a much larger set of servers than intended, including ones supporting two core S3 subsystems (the index and placement systems, which track where every object lives). Both subsystems needed a full restart, and while they were down, S3 itself was unavailable in that region, which took a big chunk of the wider internet down with it since so many other services depend on S3. That's the fault-vs-failure distinction in one real incident: the fault was a single mistyped input on one command. The failure was a multi-hour regional outage. The gap between the two is exactly what fault-tolerance work (blast-radius limits, safer tooling, staged rollouts) is trying to close.

Simplification: I'm using this incident purely to illustrate "one bad input escalating into a system-wide outage." The book doesn't discuss this specific event; it's my own outside example chosen because it fits Kleppmann's human-error category well.

Scalability: having a plan for more load, not a label

The book is blunt about this: it's meaningless to say "this system is scalable." Scalability is really a question you keep re-asking: "if load grows this way, what are our options?" To even discuss it, you first need to describe your current load with a few concrete numbers, called load parameters. Those depend entirely on your system: requests per second, the ratio of reads to writes, how many people are active at once.

The book's running example is Twitter circa 2012. Posting a tweet was relatively rare (a few thousand writes per second at peak). Reading a home timeline was extremely common (hundreds of thousands of reads per second). The naive approach (look up everyone you follow and merge their tweets at read time) was too slow because reads were so much more frequent than writes. So Twitter flipped it: on every new tweet, immediately push it into a precomputed "mailbox" for each follower, so reads become cheap. The catch is that one tweet from someone with 30 million followers now means 30 million separate writes. Neither approach is universally "right"; the best one depends on your specific load parameters, in this case the huge imbalance between how often people tweet and how often people read their timeline.

Once you know your load, you need a way to talk about performance, and the chapter makes a case for percentiles over averages. If your median (p50) response time is 200ms, half of requests are faster and half are slower. That's useful for "typical" experience, but it hides your worst cases. The 95th, 99th, and 99.9th percentiles tell you how bad your slow requests actually are, and those slow requests matter more than their small count suggests: the book notes Amazon watches performance at the 99.9th percentile because the slowest customers tend to be the ones with the most order history, i.e. the most valuable customers. Amazon has also found that a 100ms increase in response time can cost about 1% of sales. There's also a compounding effect worth knowing: if a single user request has to call five backend services and each one only has a 5% chance of being slow, the odds that at least one of the five is slow (and so the whole request looks slow) go up fast. That's called tail latency amplification.

Once you can describe load and measure performance, the chapter lays out two blunt tools for coping with growth: scaling up (a bigger machine) and scaling out (more, smaller machines, aka a shared-nothing architecture). Some systems auto-scale (elastic); most teams still scale manually because it's simpler and has fewer surprises. There's no universal "scaling sauce"; the right architecture depends entirely on which load parameters actually matter for your app.

Maintainability: make it easy for the next person

Most of the lifetime cost of software isn't writing it, it's living with it afterward: fixing bugs, adapting it, keeping it running, undoing technical debt. The chapter breaks "easy to maintain" into three separate design goals:

  • Operability — how easy the system is for an operations team to keep healthy day to day: good monitoring, good documentation, predictable behavior, not depending on any one specific machine so it can be patched without downtime.
  • Simplicity — how easy the system is for a new engineer to understand, mainly by cutting "accidental" complexity (complexity that comes from how it was built, not from the actual problem it solves). The book's example is abstraction: SQL hides a mountain of on-disk data structure detail behind a simple query language, and a high-level programming language hides machine code the same way.
  • Evolvability — how easy it is to change the system later as requirements change, which they always do. This is tightly linked to simplicity: a system that's easy to understand is usually also easy to safely modify.

A system that's built from a database, a cache, and a search index (a common combination the chapter opens with) is a good lens for this: the app code is responsible for keeping the cache and the index in sync with the database. That's a real maintainability cost every time someone adds a feature, because now there are three places that have to agree, not one.

Why it matters

These aren't academic labels, they're the questions you should be asking before you pick a database, add a cache, or agree to an SLA. Reliability decides whether your users trust you with their data (think of the parent storing every photo of their kid in your app, a scenario the book explicitly raises). Scalability decides whether your architecture survives the next order-of-magnitude jump in users. Maintainability decides whether the person who inherits your codebase in two years (possibly you) can actually work with it. Every chapter after this one is really a deeper answer to "how do we get more of these three things."

Remember this

  • A fault is one broken part. A failure is the user-facing outage. Good design is mostly about stopping the first from becoming the second.
  • Hardware faults tend to be random and independent; software bugs and human mistakes tend to be correlated, which is exactly why they cause bigger outages.
  • Never trust a single "average" response time. Look at percentiles, because your p99 is what your unluckiest users actually feel.
  • "Is this system scalable?" is the wrong question. "What happens to this system if load grows 10x?" is the right one.
  • Complexity is the real enemy of maintainability, and a good abstraction is one of the best tools you have for cutting it down.

Try it yourself

1. Fault injectorRun 3-5 small "worker" processes or threads doing a repeating task (e.g. incrementing a shared counter every second). Every few seconds, randomly kill one and restart it a moment later.
Done when: the overall task keeps making progress the whole time, even while individual workers are dying and restarting.
2. Percentile meterHit any HTTP endpoint (a public API or something running on your machine) 200+ times, logging each response time. Sort the list yourself and compute p50, p95, and p99 by hand (no library).
Done when: you can say, in one sentence, how many times slower your p99 is than your p50.
3. Redundancy demoWrite the same small record to 3 separate "replica" files or folders. Then delete one of them to simulate a disk dying, and write a read function that still returns the correct record.
Done when: your read function returns the right value even with 1 of the 3 replicas missing.
4. Load parameter loggerAdd a few counters to any toy app (a to-do list, a URL shortener) that print requests per minute and the ratio of reads to writes after a 5-minute run.
Done when: you can point at one number and say "this is my bottleneck" and explain why.
5. Tail latency amplifierWrite a function that calls 5 fake "backend" functions in parallel. Give each one a small random delay, except make one of them randomly take a long pause about 5% of the time. Run it 100 times and record how often the combined call was slow.
Done when: you can show the combined p95 is worse than any single backend's own p95.
6. Abstraction teardownPick one small library function you use often (a JSON parser, an ORM query, a date formatter) and write out, in plain English, everything it's quietly handling for you.
Done when: your list has at least 5 things you'd have had to write yourself without it.

Go deeper

If one part of this chapter deserves a reread, it's the percentiles section (the book's "Describing Performance" pages). It's short but it changes how you look at every dashboard you'll ever build. A natural next question to sit with: for a system you actually use daily (your bank app, a game, whatever), what do you think its load parameters are, and which one would break it first if it 10x'd overnight?
