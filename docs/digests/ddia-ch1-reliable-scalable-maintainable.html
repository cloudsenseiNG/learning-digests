SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What we actually mean when we call a data system "good," and why reliability, scalability and maintainability are choices you design for, not luck you hope for.

In one line

This chapter defines the three qualities worth designing a data system around, reliability, scalability, and maintainability, and gives you the vocabulary (faults vs. failures, load parameters, percentiles, accidental complexity) to actually reason about them instead of just saying "make it good."

The big idea

Most apps today are limited by data, not CPU: how much of it there is, how complicated it is, and how fast it changes. Kleppmann calls these data-intensive applications, and points out they're usually built by stitching together standard building blocks (a database, a cache, a search index, maybe a message queue) rather than writing storage engines from scratch. Once you're the person deciding how those blocks fit together, you're not just an app developer anymore, you're a data system designer, and three questions become your job: will it keep working when things go wrong (reliability), will it keep working as load grows (scalability), and will the people maintaining it in two years still be able to work with it (maintainability)? The rest of the chapter just unpacks what each of those words actually means.

Key terms

  • Data-intensive: an app whose bottleneck is data volume, complexity, or rate of change, not raw compute. Contrast with compute-intensive (video encoding, say).
  • Fault: one component of the system deviating from what it's supposed to do (a disk dies, a service returns garbage). Small and local.
  • Failure: the system as a whole stops providing the service the user needed. What the user actually notices.
  • Fault-tolerant / resilient: a system built to keep working even when specific, anticipated faults happen. Nobody can tolerate every possible fault, so this really means "tolerant of the faults we planned for."
  • Load parameters: the small set of numbers you use to describe how much your system is doing right now (requests/sec, read:write ratio, concurrent users, cache hit rate...). The right ones depend on your architecture.
  • Throughput: records processed per second (or total time for a batch job). The number that matters for offline/batch systems.
  • Response time: what the client actually experiences: the time between sending a request and getting a response back, including network and queueing delays, not just processing time.
  • Latency: technically the time a request spends waiting to be handled, before service even starts. Often used loosely as a synonym for response time, but the book is explicit that they aren't the same thing.
  • Percentile (p50, p95, p99...): sort all your response times fastest to slowest; p95 is the value below which 95% of them fall. p50 is the median.
  • Tail latency: the high percentiles (p99, p999). The slow, rare requests that still shape how the service feels.
  • SLA / SLO: service level agreement/objective: a contract or target defining "up," usually in terms of a percentile threshold and an uptime percentage.
  • Scaling up / scaling out: scaling up (vertical) means a bigger single machine; scaling out (horizontal, "shared-nothing") means spreading load across more machines.
  • Operability: how easy the system is for an operations team to keep healthy day to day.
  • Accidental complexity: complexity that comes from how a system was built, not from the actual problem it solves. The kind good abstractions can remove.
  • Evolvability: how easy it is to change the system later as requirements shift. Agility, but at the scale of a whole data system instead of one file.

How it works

What "data-intensive" actually means, first. The book opens by pointing out that raw CPU power is rarely the limiting factor for most applications built today; the hard part is the amount of data, how complex it is, and how quickly it's changing. To deal with that, most applications lean on a handful of standard building blocks instead of reinventing them: a database to store data so it (or another process) can find it later, a cache to remember the result of something expensive, a search index to let users filter or search, some form of stream processing to hand a message to another process asynchronously, and batch processing to periodically crunch a big pile of accumulated data. Individually those tools are boring and well understood. The interesting (and hard) part shows up when you combine several of them into one service and hide the seams behind your own API, because now your application code is responsible for keeping the cache, the index, and the database in agreement. You've built a small, special-purpose data system out of general-purpose pieces, and that's exactly where reliability, scalability and maintainability stop being abstract nouns and start being your actual job.

Reliability: keeps working correctly, even when something goes wrong

The book's working definition is deliberately plain: a reliable system does what the user expects, tolerates the user making mistakes, performs well enough for the load it's actually under, and keeps unauthorized people out, and it keeps doing all of that even when things go wrong. The two words that carry the whole section are fault and failure, and they are explicitly not the same thing. A fault is one component deviating from its spec, a single disk dying, a single service crashing. A failure is the system as a whole no longer providing the service to the user. You can't drive the probability of a fault to zero (short of absurd, unbudgetable levels of redundancy), so the realistic goal is fault-tolerance: design things so that faults, which will happen, don't turn into failures, which is what the user actually feels.

A fault is local and constant; whether it becomes a failure is the part you actually get to design.

Hardware faults

These are the ones everyone pictures first: a disk crashes, RAM goes bad, a network cable gets unplugged, the power grid blips. The book cites hard disks as having a mean time to failure (MTTF) of roughly 10 to 50 years, which sounds fine for one disk, but on a cluster of 10,000 disks that math means you should expect about one disk to die most days. That's not a hypothetical, it's a planning input. The traditional response is redundancy: RAID for disks, dual power supplies, hot-swappable parts, generators for the data center, so a broken component can be swapped without the machine going down. This worked well enough for a long time that only a small class of high-availability applications bothered with redundancy across machines. The book notes that's shifting: as apps run on more machines, the odds that some machine fails goes up proportionally, and on cloud platforms (AWS is the book's own example) a VM can simply vanish without warning because the platform prioritizes flexibility over any single instance's reliability. That's pushing more systems toward tolerating the loss of an entire machine in software, which has a nice side effect: you can patch one node at a time (a rolling upgrade) instead of needing a full-system maintenance window.

Software errors

Hardware faults tend to be random and roughly independent (one disk dying doesn't make a disk in a different machine more likely to die). Software bugs are the opposite: they're systematic, and because the same buggy code runs on every node, a triggering condition can take out many machines at once, which is why the book says these correlated faults tend to cause more failures overall than the uncorrelated hardware kind. The book's own example: a leap-second bug in the Linux kernel on June 30, 2012 made many applications hang simultaneously, purely because they all shared the same underlying assumption that turned out to be wrong that day. Other shapes this takes: a runaway process eating all the CPU, memory, disk or bandwidth on a shared resource; a dependency that slows down or starts returning corrupted data; or cascading failures, where a small fault in one component pushes a neighboring component past its limit too. There's no single fix here. The book's list is mostly discipline: question your assumptions, test thoroughly, isolate processes so one crash doesn't take the rest down, let things crash and restart cleanly, and where a system is supposed to hold an invariant (like "messages in equals messages out" for a queue), have it check that invariant itself and raise an alert if it's violated.

Human errors

People build these systems and people operate them, and the book cites a study finding configuration mistakes by operators were the leading cause of outages at large internet services, with hardware faults responsible for only 10 to 25% of them. So the reliability question becomes: how do you build systems that stay reliable despite the humans running them? The book's answer is a combination of things, not one silver bullet: design APIs and admin tools that make the right thing easy and the wrong thing awkward (without being so locked-down that people just work around them); give people a full-featured sandbox with real data where they can experiment without touching production; test at every level from unit tests up to full manual system tests; make it fast and cheap to undo a mistake (quick rollbacks, gradual rollouts so a bad change only ever hits a slice of users, tools to recompute data if a past computation turns out to have been wrong); and put good monitoring and telemetry in place so problems show up as an early warning instead of a surprise outage.

How important is reliability, really?

The book pushes back on the idea that reliability only matters for "serious" software like air traffic control. Bugs in ordinary business software cost lost productivity and create legal exposure; an outage at an ecommerce site costs revenue and trust directly. And it points at something more personal: think about a parent who stores every photo and video of their kid in your app. If that database silently corrupted, how would they feel, and would they even know how to get it back? That said, the book is honest that reliability sometimes gets consciously traded away, an early prototype for an unproven market, a service with razor-thin margins, but the key word is consciously. The problem isn't cutting corners, it's cutting them without realizing you did.

Scalability: coping with growth without starting over

A system working fine today says nothing about whether it'll still work fine once load increases, whether that's 10x the users, 10x the data, or just more complexity. The book is careful to say scalability isn't a label a system either has or doesn't ("X is scalable" is close to meaningless on its own); it's a question you ask about a specific kind of growth: if load grows this way, what are our options?

Describing load: load parameters

Before you can talk about "what happens if load doubles," you need a concrete, current number for load. The book calls these load parameters, and which ones matter depends entirely on your system's shape: requests/sec to a web server, the ratio of reads to writes in a database, how many users are active in a chat room at once, a cache's hit rate. Sometimes the average is what you care about; sometimes a small number of extreme cases dominate everything else.

The book grounds this with Twitter's own November 2012 numbers. Posting a tweet ran about 4.6k requests/sec on average (peaking over 12k/sec); reading a home timeline ran about 300k requests/sec, roughly two orders of magnitude more. Twitter's actual scaling problem wasn't tweet volume, it was fan-out: every user follows many people and is followed by many people, so one tweet potentially needs to reach a lot of home timelines. The book walks through two designs. Approach 1: store tweets in one global table, and when someone opens their timeline, look up everyone they follow, pull each of those users' tweets, and merge them by time (a join query). Approach 2: give every user a precomputed home-timeline cache (like a mailbox); when someone tweets, push that tweet into every one of their followers' caches immediately, so reading a timeline later is just reading the cache. Twitter started with approach 1 and it couldn't keep up with 300k reads/sec, so they moved to approach 2, since it's cheaper to do the extra work once at write time than to redo the merge on every single read.

Approach 2 trades cheap reads for expensive writes, and it works because Twitter's reads (300k/s) vastly outnumber writes (4.6k/s).

Approach 2 has a catch, though: a tweet now needs writing into every follower's cache. On average that's about 75 followers per tweet, so 4.6k tweets/sec becomes roughly 345k cache writes/sec, and Twitter tries to deliver a tweet within five seconds. But the average hides skew: some accounts have over 30 million followers, so a single tweet from one of them can mean over 30 million writes. The book's punchline is that Twitter ended up moving to a hybrid: most tweets still fan out at write time, but tweets from very-high-follower accounts are excluded from fan-out and instead fetched and merged in at read time (approach 1, just for them). The distribution of followers per user isn't a footnote here, it's itself a load parameter that shapes the whole architecture.

Describing performance: throughput vs. response time, and why the mean lies

Once you can describe load, you can ask what happens to performance as it grows. For batch systems (think a big overnight job), the number that matters is throughput: records processed per second, or total time to finish a dataset. For online systems, what matters is response time, the time between a client sending a request and getting a response, which includes the actual processing (service time) plus network delay plus any time spent waiting in a queue. The book is precise about a distinction that gets blurred constantly: latency is specifically the time a request spends waiting to be handled before service starts, while response time is the whole thing the client actually experiences. They're related but not interchangeable.

Here's the part that reframes how you should look at performance data: even the exact same request, repeated, won't take the same amount of time twice. A context switch, a lost network packet needing retransmission, a garbage collection pause, a page fault, even mechanical vibration in a server rack, any of these can add random extra time. So response time isn't one number, it's a distribution, and the book argues the mean (the plain average) is a poor way to summarize it, because it doesn't tell you how many actual users experienced the slow end of that distribution.

Instead, sort every response time from fastest to slowest and use percentiles. The median (p50) is the halfway point: half of requests finished faster, half took longer. Higher percentiles show you how bad the slow outliers, the tail latencies, really are: if p95 is 1.5 seconds, that means 95 of 100 requests finish under 1.5s, and 5 of 100 take 1.5s or more. The book flags why this matters more than it sounds like it should: Amazon defines internal service performance targets around the 99.9th percentile (p999), even though that only affects 1 in 1,000 requests, because the customers hitting the slowest requests are often the ones with the most data on their account, i.e., their highest-spending, most valuable customers. The book cites figures from sources it references: a 100ms increase in response time has been linked to a roughly 1% drop in sales at Amazon, and a 1-second slowdown to a roughly 16% drop in a customer satisfaction metric elsewhere. Worth flagging plainly: those specific numbers are the book quoting other people's studies, not something I've re-verified myself, so treat them as illustrative rather than something to cite as a primary fact.

Response times are right-skewed: most requests cluster fast, but a shaded tail of slow ones drags at p95/p99. The mean sits somewhere in the crowded part and hides that tail completely.

Percentiles matter even more once a single user-facing request depends on several backend calls (common in any service made of microservices). Even running those calls in parallel, the whole request has to wait for the slowest one to finish, so the more backend calls a request fans out to, the more likely it is that at least one of them lands in the slow tail, a compounding effect the book calls tail latency amplification. There's also a queueing angle worth understanding: a server can only run a limited number of things in parallel (bounded by CPU cores, say), so a few slow requests can back up everything behind them in the queue (head-of-line blocking), even though those later requests would have been fast on their own. That's why response times should be measured on the client side, and why, if you're load-testing, your test client needs to keep firing requests on a fixed schedule regardless of how slow previous responses were; waiting for each response before sending the next artificially shrinks the queue compared to what would happen for real.

The book also notes SLAs/SLOs are usually built directly on percentile thresholds, e.g., "median response time under 200ms and p99 under 1 second, or it doesn't count as up," alongside an uptime percentage. It briefly mentions that computing rolling percentiles cheaply (say over the last 10 minutes, refreshed every minute) gets harder at scale; sorting a full list of raw response times every minute is the naive approach, and the book just names a few approximation algorithms that exist for this (forward decay, t-digest, HdrHistogram) without walking through how they work, so I'm not going to pretend to explain algorithms the chapter itself doesn't detail. One thing it is explicit about: averaging percentiles together (across time windows, or across machines) is mathematically meaningless; the correct way to combine them is to add up the underlying histograms, not the percentile numbers themselves.

Approaches for coping with load

Once you can describe load and performance, the actual coping strategies split along a well-known line: scaling up (vertical, move to a bigger single machine) versus scaling out (horizontal, spread the load across more, smaller machines, sometimes called a shared-nothing architecture). A single powerful machine is often simpler to reason about, but high-end hardware gets expensive fast, so very demanding workloads usually can't avoid scaling out eventually. In practice the book says most good architectures mix both, a handful of fairly beefy machines is often simpler and cheaper than a swarm of tiny ones. Some systems are elastic, automatically adding capacity when they detect more load, which helps when load is unpredictable; others are scaled manually by a human deciding when to add machines, which is simpler and has fewer operational surprises. Stateless services are relatively easy to spread across machines. Stateful systems, like a database holding the data itself, are much harder to distribute, which is why the traditional default was "keep the database on one machine and scale it up, until cost or availability needs force you off that."

Scaling out is usually a means to an end, not a virtue: it costs complexity, so you reach for it once one machine genuinely isn't enough.

The chapter is blunt that there's no universal "magic scaling sauce": an architecture that handles 100,000 tiny 1KB requests/sec looks nothing like one built for 3 huge 2GB requests/minute, even though both move the same total bytes. A good scalable architecture is built around real assumptions about which operations are common and which are rare, i.e., your load parameters, and if those assumptions turn out wrong, the scaling work you did is wasted at best and actively counterproductive at worst. That's also why the book says an early-stage or unproven product usually gains more from shipping and iterating quickly than from pre-building for a hypothetical future scale.

Maintainability: making life better for whoever touches this next

The book opens this section with a fact worth sitting with: most of the cost of software isn't building it, it's everything after, fixing bugs, keeping it running, investigating what broke, porting it to new platforms, adapting it to use cases nobody planned for, paying down technical debt, and adding features. It names three design principles aimed squarely at keeping that ongoing cost sane.

Operability: make it easy to keep running

Good operations teams do a lot: watching system health and recovering it quickly when it degrades, tracking down root causes, keeping software patched, understanding how systems affect each other before a change causes damage, planning capacity ahead of need, running good deployment/config practices, handling platform migrations, keeping things secure through change, keeping processes predictable, and holding onto institutional knowledge as people come and go. The book's framing is that good software can support all of that, but can't replace it: even with great software, someone still has to set the automation up correctly and keep it working. What good software can do is give operators visibility into the system's runtime behavior, integrate cleanly with standard automation, avoid depending on any one specific machine (so a node can be taken down for maintenance without downtime), document a predictable operational model ("if I do X, Y happens"), ship sane defaults while still letting an admin override them, self-heal where that makes sense while still allowing manual control when it doesn't, and generally behave predictably rather than surprising the people running it.

Simplicity: managing complexity, not features

Small projects tend to start out clean; large ones tend to accrete complexity, an exploding number of possible states, modules that are too tightly coupled, tangled dependencies, inconsistent naming, one-off hacks to fix a performance problem, special cases layered on special cases. The industry term for the end state is a "big ball of mud." The cost isn't just aesthetic: complexity slows down everyone working on the system and makes it more likely that a change introduces a bug nobody saw coming, because the person making the change can no longer hold the whole system in their head. The useful distinction the book draws (from Moseley and Marks' work) is between essential complexity, the complexity genuinely inherent in the problem you're solving, and accidental complexity, complexity that only exists because of how the system happens to be implemented. Simplifying a system doesn't mean cutting what it does; it means cutting the accidental part. The best tool the book names for that is abstraction: hiding a pile of implementation detail behind a clean interface that can be reused across many use cases. A high-level programming language is an abstraction over machine code and CPU registers. SQL is an abstraction over on-disk data structures, concurrent access, and crash recovery. The book is honest that finding genuinely good abstractions, especially in distributed systems, is hard; it's a goal to aim for, not a solved problem.

Evolvability: making change possible

Requirements never sit still: you learn new things, unanticipated use cases show up, priorities shift, platforms change underneath you, regulations change, growth itself forces architectural rethinks. Agile practices like test-driven development and refactoring give you tools for handling that kind of change at a small, local scale (a few files in one app). The book is interested in that same agility, but at the scale of a whole data system, potentially spanning several services, and gives it its own name: evolvability. The Twitter fan-out example from earlier is a good illustration of what evolvability looks like in practice: moving from approach 1 to approach 2 (and later to the hybrid) was a "refactor" at the level of an entire data system's architecture, not a single file. And evolvability isn't independent of the earlier ideas, it's tightly linked to simplicity: a system that's simpler and better abstracted is, almost by definition, easier to reshape later.

Why it matters

This chapter is the vocabulary you'll keep reaching for in the rest of the book, and honestly in most real engineering conversations: when someone says "the service degraded," you'll want to know if that was a fault contained in time or an actual failure; when someone reports "average latency is fine," you should immediately want to see p95 and p99 before believing it; when a system that worked great at 10x smaller scale starts falling over, the first useful question is which load parameter actually changed. It also shows up directly in day-to-day tradeoffs on the AWS and MySQL side of things: elastic auto-scaling is a direct answer to "how do we cope with unpredictable load," and a read replica setup is a direct answer to "our read:write ratio is skewed, so let's design around that load parameter" rather than throwing a bigger single machine at the problem.

Remember this

  • A fault and a failure are not the same thing. A fault is one part misbehaving; a failure is the user noticing. Good design is mostly the machinery that keeps the first from becoming the second.
  • Load isn't a single number. It's a handful of load parameters chosen for your specific architecture, and sometimes the skew in that distribution (like followers-per-user) matters more than the average.
  • Never trust the mean response time on its own. Look at p50 and then p95/p99, because the tail is what your unluckiest, and often most valuable, users actually feel.
  • "Scalable" isn't a badge a system earns once. It's relative to a specific kind of growth, and an architecture built for one order of magnitude of load usually needs rethinking at the next.
  • Most of the complexity you fight day to day is accidental, not essential to the problem, which is exactly why a good abstraction is often worth more than a clever patch.

Try it yourself

  • Build a tiny fault injector. Wrap a function or API call so it randomly throws, hangs, or returns garbage some percentage of the time. Call it from real code that wasn't written expecting that, and see what actually happens (this is the same instinct behind the book's mention of deliberately induced faults).
  • Build a percentile meter. Time 200+ calls to something (a real endpoint, or just a function with a random sleep), log every duration, then compute the mean and the p50/p95/p99 by sorting the list and indexing in. Compare how different the mean looks from p99.
  • Demonstrate tail latency amplification. Simulate 20 "backend calls," each with a small (say 5%) independent chance of being slow. Run the batch 100 times and count how often the overall request (which waits for the slowest of the 20) ends up slow, versus a single call's 5% on its own.
  • Build a tiny redundancy demo. Run two copies of a minimal key-value server. Kill one mid-traffic and confirm the other keeps serving requests. Note what "restore it quickly" would actually require for real data.
  • Run a complexity audit on your own code. Pick something small you've built, list every hack or special case in it, and sort each one into "essential to the problem" or "accidental, from how I happened to build it." Try abstracting one accidental one away.

Go deeper (optional)

Reread "Percentiles in Practice" and think through how you'd actually compute a rolling p99 for a service you run, cheaply, without keeping every single raw response time in memory. That question is exactly what algorithms like t-digest and HdrHistogram exist to answer, and the chapter only names them, so it's a good next rabbit hole.
