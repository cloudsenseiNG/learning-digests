SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What we actually mean when we call a data system "good," and why reliability, scalability, and maintainability are things you design for, not things you get lucky into.

In one line

Before you pick a database or a framework, it helps to agree on what "working well" even means, and this chapter breaks that down into three concrete goals you can actually reason about: reliability, scalability, and maintainability.

The big idea

Modern applications are usually built by stitching together existing building blocks: a database to store things, a cache to speed up reads, a search index, a queue for async work, a batch job for crunching large datasets. Once you combine a few of these into your own system, you become a data system designer, not just an application developer, and that comes with real questions: what happens when a part of it breaks? What happens when usage grows tenfold? What happens two years from now when a different engineer has to change it? This chapter gives you a vocabulary for those three questions: reliability (keeps working when things go wrong), scalability (has a reasonable way to handle growth), and maintainability (stays workable for the humans who touch it later).

Key terms

  • Data-intensive application — an application where the hard part is the amount, complexity, or rate of change of data, not raw CPU power. Think a system storing and searching millions of records, versus a system doing heavy number-crunching on a small amount of input.
  • Fault — one component of the system deviating from what it's supposed to do. A disk that dies, a service that returns a bad response, a bug that trips on one bad input.
  • Failure — the system as a whole stops providing the service it's supposed to provide to the user. Not the same thing as a fault: a fault is local, a failure is what the user actually experiences.
  • Fault-tolerant / resilient — describes a system built to keep working even when some faults happen. It's bounded on purpose: no system can tolerate literally every possible fault (nobody is budgeting for "the planet gets swallowed by a black hole"), so in practice it means tolerating certain expected kinds of faults.
  • Reliability — a system continuing to do the correct thing, at acceptable performance, even when hardware, software, or people misbehave.
  • Scalability — a system's ability to cope with growth in load (more users, more data, more traffic) by adding resources in a reasonable way, rather than falling over.
  • Load parameters — the small set of numbers you use to describe how much load a system is under right now, so you have something concrete to reason about when asking "what if this grows?" Requests per second, read/write ratio, number of active users, that kind of thing.
  • Throughput — how many records or requests a system can process per unit time. The number that matters most for batch jobs.
  • Response time — what the client actually experiences: the time from sending a request to getting a response back, including network delay and any time spent waiting in a queue.
  • Latency — often used loosely as a synonym for response time, but strictly it's the time a request spends waiting to be handled before anyone starts working on it.
  • Percentile (p50, p95, p99...) — sort every response time from fastest to slowest; the 50th percentile (p50, the median) is the point where half the requests were faster and half slower. p95 means 95% of requests were faster than that value. Higher percentiles show you how bad your worst cases are.
  • Tail latency — the high percentiles (p99, p99.9). Called "the tail" because that's where they sit on a distribution curve of response times, and they matter because a small number of very slow requests can still ruin the experience for real users.
  • Head-of-line blocking — when one slow request stuck at the front of a queue delays every request behind it, even ones that would otherwise be fast.
  • SLA / SLO — service level agreement / service level objective: a stated promise about performance and uptime, usually expressed with percentiles (for example, "p99 under 1 second, up 99.9% of the time").
  • Scaling up vs scaling out — scaling up (vertical) means moving to a bigger, more powerful machine. Scaling out (horizontal) means spreading the load across more machines. A system built around scaling out is often called a shared-nothing architecture.
  • Operability — how easy the system is for an operations team to keep running smoothly day to day.
  • Simplicity (and accidental complexity) — how easy the system is for a new engineer to understand. "Accidental" complexity is complexity that comes from how the system happens to be built, not from the actual problem it's solving, and it's the kind you can often design away.
  • Evolvability — how easy it is to change the system later as requirements change, which they always do.

How it works

The chapter opens by pointing out something easy to miss: databases, caches, queues, and search indexes look like totally different tools, but we group them under one umbrella, "data systems," because most real applications end up stitching several of them together to get one working product. The moment you do that stitching yourself (say, keeping a cache in sync with your main database, or gluing a search index onto your app), you've become a data system designer whether you meant to or not, and now you own the tricky questions: does the data stay correct when something goes wrong internally? Does performance stay decent when part of the system is degraded? Can it handle more load later? This is the frame for the whole book: reliability, scalability, and maintainability are the three lenses it keeps coming back to, so it's worth getting a solid mental model of each one now.

Reliability: keep working, even when things go wrong

Reliability sounds fuzzy until you break it into a fault versus a failure. A fault is one part of the system deviating from spec: a disk crashes, a service returns garbage, a bug trips on an edge case. A failure is when the system as a whole stops doing its job for the user. The whole point of building a reliable system is designing so that faults, which are basically guaranteed to happen eventually, don't automatically turn into failures, which the user actually experiences.

Faults roughly fall into three buckets, and each one behaves differently:

  • Hardware faults. Disks are commonly reported with a mean time to failure (MTTF) somewhere around 10 to 50 years, so on a cluster with 10,000 disks that works out to roughly one disk dying per day on average. The traditional fix is redundancy: RAID setups, dual power supplies, backup generators, so a broken part can be swapped without the whole machine going down. That worked well enough when most systems ran on one or a handful of machines. Once systems span very large numbers of machines (or run on cloud platforms where a virtual machine can become unavailable without much warning), the odds of some hardware fault happening on any given day go up proportionally, which pushes systems toward tolerating the loss of a whole machine rather than just a whole disk.
  • Software errors. These are systematic rather than random: the same bug fires the same way everywhere it's triggered, which is why software bugs tend to be correlated across many machines at once in a way hardware faults usually aren't. One real example from the chapter: a leap second added on June 30, 2012 caused a bug in the Linux kernel that made many applications hang simultaneously. There's no single fix for this category. It's a mix of things: careful assumptions, thorough testing, isolating processes so one crash doesn't take others down, and monitoring that catches drift from expected behavior early.
  • Human errors. People configure and operate these systems, and people make mistakes. One study of large internet services found that configuration errors by operators were the leading cause of outages, with hardware faults responsible for only 10 to 25% of them. The best defenses aren't about expecting perfection from people: well-designed interfaces that make the right thing easy and the wrong thing hard, safe sandbox environments to practice in, thorough testing, quick rollback when something goes wrong, and good monitoring so problems get caught early rather than discovered by users.

One idea that sounds backwards at first: some teams deliberately increase the rate of faults on purpose, for example by randomly killing processes in production, so that their fault-tolerance mechanisms get exercised constantly instead of sitting untested until the day they're actually needed. The Netflix Chaos Monkey tool, mentioned in the chapter, is a well-known example of this approach.

It's also worth noting reliability isn't free, and sometimes it's fine to trade some of it away deliberately, for example on an early prototype where speed of iteration matters more than uptime. The point isn't "always maximize reliability." It's knowing when you're cutting that corner on purpose versus by accident.

Scalability: having an answer for "what happens when this grows?"

Scalability isn't a label you can just slap on a system ("this is scalable"). It only makes sense as a question: if load grows in a particular way, what are your options for coping with it? To even ask that question you first need a way to describe the load you have right now, and a way to measure whether performance is holding up.

Describing load

Load parameters are the handful of numbers that describe current load, and which ones matter depends entirely on your system: requests per second, the ratio of reads to writes, how many users are active at once, your cache hit rate, or something specific to your architecture.

The chapter walks through a real example from Twitter, using numbers Twitter published in November 2012. Posting a tweet ran at around 4.6k requests/second on average (peaking over 12k/second). Reading a home timeline ran at around 300k requests/second, nearly two orders of magnitude higher. Twitter's actual scaling challenge wasn't the volume of tweets, it was the "fan-out": every user follows many people and is followed by many people. There were two broad ways to build the home timeline feature:

  1. Store every tweet in one global table, and when a user asks for their timeline, look up everyone they follow, pull each of those users' tweets, and merge them by time, on every single read.
  2. Give every user a precomputed timeline cache. When someone posts, look up everyone who follows them and insert the new tweet into each of those followers' cached timelines right away. Reading a timeline is then just a cheap lookup, because the work already happened at write time.

Twitter started with approach 1, but it couldn't keep up with the read volume, so they switched to approach 2, because reads happened far more often than writes and it made sense to do more work up front and less on every read. The catch: approach 2 makes posting expensive. On average a tweet fans out to about 75 followers, so 4.6k tweets/second becomes roughly 345k writes/second to timeline caches, but that average hides a lot: some accounts have over 30 million followers, so a single tweet from one of them can mean over 30 million cache writes, and Twitter aims to deliver tweets to followers within about five seconds. In practice, Twitter ended up moving to a hybrid: most users still get fanned out at write time (approach 2), but tweets from accounts with huge follower counts get fetched at read time instead (approach 1) and merged in, because fanning those out to millions of caches on every post isn't worth it. This is a specific, attributed example from one company's system at one point in time, not a universal rule for how every application should be built. The transferable idea is the general one: your own load parameters (whichever ones matter for your system) are what should drive which operations you optimize for.

Describing performance

Once you can describe load, you need a way to describe performance so you can see what happens as load changes. Batch systems (crunching a big dataset in one go) usually care about throughput: records processed per second. Online systems, the kind responding to individual requests, usually care more about response time: how long a client waits between sending a request and getting a response back. That response time includes network delays and any queueing, not just the actual work done, which is a subtly different thing from latency (the time a request spends waiting before anyone starts handling it).

The important shift here is realizing response time isn't one number, it's a distribution. Run the exact same request over and over and you'll get a slightly different time each try, and in a real system handling varied load, that spread gets a lot wider (context switches, garbage collection pauses, network retransmits, disk reads, and plenty of other causes pile on top of the "real" work). The mean (average) response time gets reported a lot, but it's a weak signal for "what does a typical user experience," because it doesn't tell you how many people actually hit that number. Percentiles are the better tool: sort every response time from fastest to slowest, and the median (p50) is the halfway point. A p95 or p99 tells you where your worst cases sit; if p95 is 1.5 seconds, that means 95 out of 100 requests were faster than that, and 5 were slower.

High percentiles ("tail latencies") matter more than they sound like they should, because they're what your worst-treated users actually feel. The chapter mentions Amazon describing its internal service requirements in terms of the 99.9th percentile, even though that only touches 1 in 1,000 requests, on the reasoning that the customers hitting those slow requests are often the ones with the most data on their account, which tends to mean the most valuable customers. Amazon has also reported that a 100 ms increase in response time can reduce sales by about 1%, and other reports cited in the chapter found a 1-second slowdown cutting a customer satisfaction metric by about 16%. On the flip side, Amazon apparently found that chasing the 99.99th percentile (the slowest 1 in 10,000) wasn't worth the cost, since gains that far into the tail get expensive fast and are easily swamped by random factors outside your control. These are specific, attributed figures from specific companies at a point in time, not universal constants, but they're a useful illustration of why tail percentiles get taken seriously.

Percentiles show up directly in SLAs and SLOs: a service level agreement might define "up" as a median response time under 200 ms and a p99 under 1 second, with a requirement to be up at least 99.9% of the time. That's the chapter's own illustrative example of what an SLA can look like, not a claim that every service targets those exact numbers.

Two extra wrinkles worth knowing: queueing delays often dominate response time at high percentiles, because a server can only handle so many things in parallel, so a handful of slow requests at the front of the line can hold up everything behind them (head-of-line blocking) even if those later requests would individually have been fast. And when a single end-user request depends on several backend calls, it only takes one of those calls being slow to make the whole request slow, an effect the chapter calls tail latency amplification: the more backend calls a request fans out to, the higher the odds that at least one of them lands in the slow tail.

Approaches for coping with load

Once you can describe load and performance, the actual scaling question is: what do you do as load grows? Broadly there's scaling up (a bigger single machine) and scaling out (spreading load across more machines, sometimes called a shared-nothing architecture). A single powerful machine can be simpler to run, but very demanding workloads often can't avoid scaling out eventually, and in practice a lot of real architectures mix both, since a handful of solid machines can be simpler and cheaper than a huge fleet of tiny ones. Some systems are elastic and add capacity automatically when load spikes; others are scaled manually by a human deciding when to add machines. Elastic systems help with unpredictable load, but manual scaling tends to be simpler and more predictable operationally.

Distributing a stateless service (one that doesn't hold onto data between requests) across machines is fairly straightforward. Taking a stateful data system like a database from one machine to a distributed setup is a lot more involved, which is why a lot of common practice has been to keep a database on one beefy machine for as long as possible, and only go distributed once cost or availability needs force the issue. There's no single generic architecture that scales for every application. A system built for 100,000 small requests per second looks nothing like a system built for a handful of requests per minute each carrying gigabytes of data, even though the two might move the same total amount of data. A scalable architecture is built around assumptions about which operations are common and which are rare (your load parameters); if those assumptions turn out wrong, the engineering effort spent scaling for them can be wasted or even actively counterproductive. That's part of why an early-stage product usually gets more value from iterating quickly than from scaling for a load it doesn't have yet.

Maintainability: designing for the humans who come after you

Most of the lifetime cost of software isn't writing it, it's maintaining it afterward: fixing bugs, keeping it running, adapting it to new requirements, paying down shortcuts taken earlier. The chapter frames three design goals that make that ongoing work less painful.

Operability

Operability is about making life easier for whoever keeps the system running day to day. In practice that means things like: giving good visibility into what the system is doing internally, supporting automation instead of requiring manual babysitting, not depending on any one specific machine staying alive, documenting how the system behaves clearly enough that "if I do X, Y happens" is actually true, providing sane defaults while still letting an operator override them when needed, and generally behaving predictably so there are fewer surprises in production.

Simplicity and accidental complexity

Small projects can stay simple. Bigger ones tend to accumulate complexity: tangled dependencies, inconsistent naming, special-case hacks layered on top of each other, until the whole thing is hard for anyone to reason about, sometimes called a "big ball of mud." Making a system simpler doesn't have to mean cutting functionality. A useful distinction the chapter draws is accidental complexity: complexity that comes from how the system happens to be implemented, not from the actual problem it's solving. That kind is usually avoidable. One of the best tools for removing it is a good abstraction, something that hides a pile of implementation detail behind a clean interface. High-level programming languages hiding machine code and CPU registers, or SQL hiding the on-disk structures and concurrency control underneath it, are both examples of abstractions doing exactly that. Good abstractions are hard to find, especially in distributed systems, but they're worth hunting for because they pay off every time the system gets touched again.

Evolvability

Requirements change. You learn new things, business priorities shift, new platforms replace old ones, and a system that can't adapt becomes a system people are afraid to touch. Agile practices like test-driven development and refactoring address this at the scale of a single codebase, but the chapter is interested in the same idea at the scale of a whole data system, possibly spanning multiple services: how easily can you reshape it as requirements shift? That's what the chapter calls evolvability (also sometimes called extensibility or modifiability), and it's closely tied to how simple and well-abstracted the system already is: simpler systems are just easier to reshape than tangled ones.

Why it matters

These three lenses aren't just chapter-one throat-clearing, they're the frame the rest of the book keeps returning to. Every design choice covered later (which kind of database, how to replicate data, how to partition it, which consistency guarantees to pick) eventually gets evaluated against some mix of "does this stay reliable," "does this scale the way we need," and "can we still maintain this in two years." Having concrete definitions for those three words now means later chapters can talk about tradeoffs precisely instead of vaguely. It also matters outside of any book: the fault-versus-failure distinction, load parameters, and percentile thinking are all things you'll bump into in real incident reviews, real capacity planning conversations, and real "why does this feel slow sometimes" debugging sessions.

Remember this

  • A fault is one part breaking. A failure is the whole system failing the user. Reliable design is about stopping the first from becoming the second.
  • Scalability is a question, not a label: "if load grows this way, what are our options?" It always needs load parameters to be concrete.
  • The average response time hides more than it reveals. Percentiles, especially the tail (p95, p99, p999), tell you what your worst-treated users actually experience.
  • There's no universal "scales for everything" architecture. A good architecture is built around your specific, current assumptions about what's common and what's rare, and those assumptions can be wrong.
  • Maintainability comes down to operability (easy to run), simplicity (easy to understand), and evolvability (easy to change). All three compound over a system's life far more than they seem to on day one.

Try it yourself

  1. Percentile meter. Write a script that takes a list of numbers (simulate 100-500 fake "response times" if you don't have real ones) and prints the p50, p95, and p99. Done when: it matches a value you sanity-check by manually sorting a small sample by hand.
  2. Redundancy demo. Spin up 3 or more toy "worker" processes or threads behind a tiny dispatcher that routes fake requests to whichever ones are alive. Kill one at random every few seconds. Done when: you can watch requests keep succeeding in your terminal output even while a worker is down.
  3. Load parameter logger. Wrap a tiny local script or server so it counts requests per second and the ratio between two kinds of requests (reads vs writes, or any split that fits what you're testing). Done when: pointing a few seconds of simulated traffic at it prints a rolling requests/sec number and a read:write ratio.
  4. Fan-out simulator. Build a tiny in-memory version of the two Twitter timeline approaches from this digest, using maybe 20 fake users and a small follow graph. Time how long a "post" and a "read" take under each approach. Done when: you can show, with your own numbers, that approach 2 makes reads cheap and writes more expensive, the same tradeoff described above.
  5. Chaos-lite dependency call. Write a function that calls a "downstream service" function which randomly fails some fixed percent of the time. Wrap the caller with a retry (and optionally a simple circuit breaker that stops calling after repeated failures). Done when: running it 100+ times shows the raw failure rate roughly matches what you injected, and the wrapped version's effective failure rate is visibly lower.

Go deeper (optional)

Pick one system you use daily (an app, a website, anything) and try to guess its load parameters: what's the read/write ratio likely to be, what's probably the spikiest moment of its day, and which of reliability, scalability, or maintainability do you think it sacrifices most for the others? Sitting with that question is a good primer before later chapters start picking apart specific tools and techniques for achieving these three goals.
