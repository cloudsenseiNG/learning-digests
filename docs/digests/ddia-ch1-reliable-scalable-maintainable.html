SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

In one line: what it actually means for a data system to be "good," and the three qualities, reliability, scalability, and maintainability, worth designing for on purpose instead of hoping for.

The big idea

Before you write a line of code, it helps to agree on what "this system works well" even means. This chapter breaks that vague goal into three separate qualities: staying correct even when things go wrong (reliability), coping with more load over time (scalability), and staying easy to work on as it grows (maintainability). None of these are properties a system just happens to have. They're choices you design for, and each one shows up in very concrete, checkable ways.

Key terms

Data-intensive application
An app where the hard part is the amount, complexity, or speed of data, not raw CPU power. Think a social feed, not a video encoder.
Fault
One component of a system deviating from its spec. Example: one disk in a cluster dying.
Failure
When the system as a whole stops providing the service the user needed, no matter which internal part caused it.
Fault-tolerant / resilient
Built to keep working correctly even when some faults happen. Not the same as fault-proof, that's not achievable.
Load parameters
The small set of numbers that describe how much load a system is under, e.g. requests per second, read:write ratio, concurrent users.
Fan-out
How many downstream things one action touches. One tweet fanning out to every follower's timeline is a fan-out of "number of followers."
Response time
What the client actually experiences: time to process the request, plus network delay, plus any time spent waiting in a queue.
Latency
Specifically the time a request spends waiting before it's handled. People use it interchangeably with response time, but they're not quite the same thing.
Percentile (p50 / p95 / p99)
The value below which a given percentage of measurements fall. p50 is the median: half your requests are faster, half slower.
Tail latency
The response times out at the high percentiles, i.e. your slowest outliers.
SLA / SLO
A target or contract defining expected performance and availability, e.g. "p99 under 1 second, up 99.9% of the time."
Scaling up vs. scaling out
Vertical scaling (a bigger machine) vs. horizontal scaling (more, smaller machines working together, sometimes called shared-nothing).
Operability
How easy the system is for an operations team to keep running smoothly day to day.
Accidental complexity
Complexity that comes purely from how something was built, not from the problem it's actually solving.
Evolvability
How easy it is to change the system later as requirements shift.

How it works

Reliability: keep working correctly, even when things go wrong

The chapter's definition of reliable isn't "never breaks." It's closer to "keeps doing the right thing even when parts of it are broken." That distinction rests on one small but important pair of words: a fault is one component going wrong (a disk dies, a process crashes), and a failure is when the user actually feels it, the system as a whole stops delivering. You cannot get the probability of a fault to zero. So the real engineering question isn't "how do I prevent every fault," it's "how do I stop faults from turning into failures."

A concrete example the chapter uses: hard disks have a mean time to failure (MTTF) of roughly 10 to 50 years. That sounds safe until you run 10,000 of them in one cluster, at which point simple math says you should expect a disk to die on most days, not once a decade. A reliable storage system plans for that daily disk death (with redundancy like RAID, backups, or replicating data across machines) instead of treating each failure as a surprise.

Faults come from three places, and they behave differently. Hardware faults (disks, RAM, power, network cables) are mostly random and independent, which is why redundancy works so well against them. Software bugs are systematic: they tend to hit every instance at once because the same buggy code is running everywhere. The chapter's example is the June 30, 2012 leap second, which triggered a bug in the Linux kernel that made many applications hang simultaneously, a single logic error, correlated across every machine running it. Human error is the third source, and it's a big one: the chapter cites a study where configuration mistakes by operators were the leading cause of outages, ahead of hardware. Good systems respond to that not by expecting humans to stop making mistakes, but by making mistakes cheap: safe sandboxes to experiment in, fast rollbacks, gradual rollouts so a bug only hits a slice of users, and monitoring that surfaces problems quickly.

One genuinely counterintuitive move the chapter mentions: some teams deliberately increase the rate of faults on purpose, by randomly killing processes in production (Netflix's Chaos Monkey is the named example). The logic is that fault-tolerance code you never exercise is code you can't actually trust, so triggering faults on purpose keeps that machinery honest.

Why this matters: reliability isn't just for "critical" systems like air traffic control. The chapter's example is a parent's photo app: if that database silently corrupts their kids' photos, the stakes are entirely real to that user even though nothing about the system looked "mission critical." Reliability is a dial you're allowed to turn down (an early prototype, a low-margin side project) but you should always know when you're doing it on purpose versus by accident.

Scalability: coping with growth, not a single number

The chapter is blunt about this: it's meaningless to say a system "is scalable" or "doesn't scale." Scalability is really a question: if this specific kind of load grows, what are my options? That means step one is describing your current load precisely, using a handful of load parameters, requests per second, read:write ratio, number of concurrent users, cache hit rate, whatever actually drives your system's cost.

The chapter's worked example is Twitter, using numbers the company published in a 2012 conference talk (so treat these as illustrative and historical, not current figures). Two operations: posting a tweet (about 4.6k requests/sec on average, 12k at peak) and reading a home timeline (about 300k requests/sec). Naively, you'd expect the harder problem to be the higher number, timeline reads. But the chapter's real point is about fan-out: every tweet has to reach every follower, and some accounts had tens of millions of followers. There were two honest ways to build this:

  • Approach 1 (compute at read time): store all tweets in one place; when someone opens their timeline, look up who they follow and merge in all of those users' recent tweets on the spot.
  • Approach 2 (compute at write time): give every user a precomputed "mailbox" of tweets. When someone posts, push a copy into every follower's mailbox immediately, so reading a timeline is just reading your own mailbox.

Twitter started with approach 1 and it couldn't keep up with timeline reads, so they moved to approach 2: since reads (300k/sec) vastly outnumber writes (4.6k/sec), it's worth doing more work up front at write time to make reads cheap. But approach 2 has its own cost: a tweet from someone with 30 million followers means 30 million mailbox writes for that one tweet. So later, the real system became a hybrid: most tweets fan out at write time, but a small number of very high-follower accounts are excluded from fan-out and merged in at read time instead, approach 1 and 2, each used where it fits. The general lesson (not just a Twitter one): when reads vastly outnumber writes, push work to write time; when the reverse is true, don't.

Once you can describe load, the next question is describing performance. Batch systems care about throughput (records processed per second). Online systems care about response time, and response time is not one number, it's a distribution: run the exact same request twice and you'll get two different timings, because of things like garbage collection pauses, a lost network packet needing retransmission, or another process briefly hogging the CPU. Reporting the mean (average) response time hides this: the mean doesn't tell you how many users actually felt a delay. Percentiles are more honest. The median (p50) is the halfway point: half of requests were faster, half slower. Higher percentiles like p95 and p99 show you the tail, if p95 is 1.5 seconds, that means 5 out of 100 requests took 1.5 seconds or longer.

High percentiles matter more than they sound like they should. The chapter notes Amazon tracks internal services against p999 (the 99.9th percentile) even though it only affects 1 in 1,000 requests, because the slowest requests often belong to customers with the most data on their account, i.e. the most active, most valuable customers. It also matters more when one user action triggers several backend calls in parallel: the end-user request has to wait for the slowest of those calls, so the more backend calls you fan out to, the higher the odds that at least one of them lands in the slow tail. That effect is called tail latency amplification. As a simplified illustration (this exact math isn't spelled out in the chapter, so treat it as a teaching aid): if each of 5 parallel calls independently has a 1% chance of being slow, the chance that at least one of the 5 is slow is roughly 1 − 0.99⁵ ≈ 5%, five times worse than any single call's own odds.

Finally, coping with load in practice usually means choosing between scaling up (a bigger single machine, simpler, but can get expensive and eventually hits a ceiling) and scaling out (many smaller machines sharing the work, often called a shared-nothing architecture). Some systems are elastic and add capacity automatically when load rises; others are scaled manually, which is simpler and has fewer surprises but needs a human watching. The chapter is careful to say there's no "magic scaling sauce": an architecture built to handle 100,000 small requests per second looks nothing like one built for 3 huge requests per minute, even if the total data throughput is identical. What matters is which load parameter is actually growing for your system.

Maintainability: making life better for the people who come after

Most of the lifetime cost of software isn't writing it, it's maintaining it: fixing bugs, adapting it to new requirements, keeping it running. The chapter names three design principles that make maintenance less painful:

  • Operability, making it easy for an operations team to keep the system healthy: good monitoring and visibility, predictable behavior, sane defaults with room to override them, and not depending on any single machine staying alive.
  • Simplicity, fighting what the chapter calls a "big ball of mud," a system so tangled that no one fully understands it anymore. The main tool here is abstraction: hiding a pile of implementation detail behind a clean interface. SQL is the chapter's example, it hides on-disk data structures and concurrency bugs behind a simple query language. The key distinction: complexity is "accidental" when it comes purely from how something was built rather than from the problem itself, and that's the kind good abstractions can remove.
  • Evolvability, how easily the system can change as requirements shift, because they always do. This is closely tied to simplicity: a system that's easy to understand is usually a system that's easy to safely modify. Twitter's move from approach 1 to approach 2 for timelines (and later to the hybrid) is really a story about evolvability, the architecture had to change as the real bottleneck became clear.

Why it matters

These three qualities show up constantly outside the page of a textbook. Reliability is what you're actually promising when you write an SLA, or when you get paged at 2am. Scalability is the conversation you have before a product launch, when someone asks "what happens if this goes viral." Maintainability is why some codebases get easier to work in over time and others get scarier with every commit. None of the three come free, and all three trade off against short-term development speed, which is a real, legitimate reason to consciously under-invest in one of them for a prototype, as long as it's a choice and not an accident.

Remember this

  • Reliability isn't "never breaks," it's "keeps working correctly even when parts of it do break." A fault is one broken part; a failure is when the user actually feels it.
  • You can't scale a system to "handle 10x" in the abstract. You have to say which load parameter is growing 10x, because different kinds of growth need completely different fixes.
  • The average response time hides your worst-off users. Look at p95/p99 (tail latencies) if you actually want to know how bad things get for real people.
  • The more backend calls one request fans out to, the higher the odds it hits at least one slow one, so tail latency matters more the more things you call per request.
  • Good abstractions are the main tool for keeping a system maintainable: they hide accidental complexity so people, including future you, can change it without holding the whole thing in their head at once.

Try it yourself

  • Fault injector. Build a tiny HTTP server with 2-3 endpoints and a wrapper that randomly fails or hangs about 1 in 10 requests. Add one piece of fault tolerance (a retry, or a timeout with a fallback response) and rerun. Done when: your logs show the raw failure rate (~10%) but the wrapped client sees a much lower failure rate, because the fault got contained before it became a failure.
  • Percentile meter. Fire 200 requests at a local server (or simulate: mostly 20-60ms responses with a few 1-2s outliers), log every response time, then compute mean, p50, p95, and p99 from the sorted list. Done when: you can print a table showing mean next to p50/p95/p99, and write one sentence on how far apart the mean and p99 turned out to be.
  • Redundancy demo. Run two copies of a trivial "hello" server on two ports, put a ~5-line round-robin proxy in front, then kill one process mid-run while sending continuous requests. Done when: your request log shows zero failed requests across the kill, because traffic kept flowing to the surviving copy.
  • SLA checker. Using the log from the percentile meter (or a synthetic one), write a script that checks it against a simple rule, e.g. "p99 must stay under 500ms," and prints PASS or FAIL. Done when: injecting a batch of artificially slow entries makes it correctly flip to FAIL, and removing them flips it back to PASS.
  • Tail-latency amplification demo. Simulate one end-user request making N parallel backend calls, each independently "slow" 5% of the time. Run it for N = 1, 2, 5, 10 and record what fraction of simulated requests hit at least one slow call. Done when: you have a small printed table showing that fraction climbing as N grows.
  • Abstraction teardown. Take one script or module you've written, find one piece of accidental complexity (a hardcoded value, a tangled conditional, config mixed into logic), and refactor it behind a small clean function or interface. Done when: you have a before/after diff plus 2-3 sentences on what you removed and why it's now easier to change.

Go deeper (optional)

If the percentile math is what clicked for you, reread the "Percentiles in Practice" part of the chapter (the tail latency amplification idea) and try answering it for something you actually use: how many backend calls does a single page load of that app probably make, and what does that imply about its realistic p99?
