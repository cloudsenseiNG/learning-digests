SYSTEMS · CH.1 DIGEST

Reliable, Scalable, and Maintainable Applications

What we actually mean when we call a system "good," and the three qualities worth designing for on purpose instead of hoping for.

The big idea

Before you build anything, it helps to agree on what "working well" even means. This chapter breaks that vague idea into three concrete goals: reliability (keeps working when things go wrong), scalability (keeps working as load grows), and maintainability (stays workable for the humans who touch it later). None of these show up by accident. You design for them, the same way you'd design for any other requirement.

Key terms

  • Fault: one part of the system going wrong (a disk dies, a process crashes). Not the same as a failure.
  • Failure: the whole system stops doing its job for the user. A fault becomes a failure only if nothing catches it.
  • Fault-tolerant: a system built to keep working even when a fault happens. Not "fault-proof," just tolerant of the faults it was designed for.
  • Load parameters: the few numbers that describe how hard your system is being hit right now (requests per second, read/write ratio, active users). You need these before you can even ask "what happens if load doubles?"
  • Percentile (p50, p95, p99): sort every response time from fastest to slowest; p50 (the median) is the point where half of requests were faster and half slower. p95 and p99 tell you how bad your worst cases are.
  • Tail latency: how slow your slowest requests are (the high percentiles). Matters more than it sounds like it should, because it's what your unluckiest users actually feel.
  • Operability, simplicity, evolvability: the three ingredients of maintainability: easy to run, easy to understand, easy to change.

How it works

Reliability: keep working when things go wrong

The book's definition of reliability is basically "continuing to work correctly, even when things go wrong." The things that go wrong are faults, and the chapter is careful to separate a fault (one component misbehaving) from a failure (the user-facing system going down). A fault-tolerant system is built so that individual faults get absorbed instead of cascading into a failure.

There are three flavors of fault worth knowing apart, because you handle each differently:

  • Hardware faults: disks die, RAM goes bad, someone trips over a cable. The book cites hard disks having a mean time to failure (MTTF) of roughly 10 to 50 years, which sounds great until you remember a cluster of 10,000 disks means you should expect roughly one disk dying per day. The classic fix is redundancy: RAID, dual power supplies, backup generators. That works well for single-machine failures, but as fleets have grown (and cloud platforms like AWS treat instance loss as a normal, unannounced event rather than an emergency) the industry has leaned more on software-level fault tolerance that assumes whole machines will disappear.
  • Software errors: a bug that crashes every instance of your app on the same bad input, a runaway process eating shared memory or CPU, a slow dependency that other services quietly depend on, or a cascading failure where one small fault trips another. These are nastier than hardware faults because they're correlated: one bad deploy can take down every node at once, instead of dying independently like disks do. The book's example is the June 30, 2012 leap second, which triggered a Linux kernel bug that hung many applications simultaneously.
  • Human error: the book cites a study finding that operator configuration mistakes were the leading cause of outages in large internet services, ahead of hardware faults (which the study put at only 10 to 25% of outages). You can't out-discipline this away; you design around it: minimize ways to make a mistake, give people a sandbox to break things safely away from production, test thoroughly, make rollbacks fast and cheap, and monitor closely enough to catch problems before users do.

One counterintuitive move the chapter mentions: some teams deliberately trigger faults on purpose, like Netflix's Chaos Monkey randomly killing production processes. The logic is that fault-tolerance code you never exercise is fault-tolerance code you can't trust; forcing it to run constantly is how you find out if it actually works before a fault happens for real.

A fault only becomes a failure if nothing catches it in time. Redundancy, retries and fallbacks live at that "caught in time?" fork.

Scalability: keep working as load grows

Scalability isn't a yes/no label ("is X scalable?" is the wrong question per the book). It's really "if load grows this way, what are our options?" To even discuss that, you first need load parameters: a few numbers that describe current load, chosen based on your system (requests/sec, read:write ratio, concurrent users, cache hit rate).

The book's running example is Twitter circa November 2012: posting a tweet ran at about 4.6k requests/sec on average (spiking to 12k/sec), but reading a home timeline ran at roughly 300k requests/sec. The interesting problem wasn't tweet volume, it was fan-out: every user follows many people and is followed by many. Twitter tried two designs. Approach 1: store tweets globally, and compute each user's timeline on read by joining across everyone they follow. Approach 2: precompute a "mailbox" of tweets for every follower the moment someone tweets, so reads become cheap lookups. Since reads vastly outnumbered writes, Twitter moved from approach 1 to approach 2, then later blended in a hybrid for celebrity accounts with tens of millions of followers, whose tweets are too expensive to fan out to every follower on every post.

Once you can describe load, you need to describe performance, and the book pushes hard on percentiles over averages. An average response time hides how many users actually waited that long. If your median (p50) is 200ms, half your users waited under 200ms and half waited longer, that's useful. But it's the high percentiles (p95, p99) that catch the slow outliers, and those matter more than they sound like they should: the book notes Amazon has observed a 100ms increase in response time can cost about 1% of sales, and that Amazon deliberately targets its 99.9th percentile internally, even though it affects just 1 in 1,000 requests, because the slowest requests often belong to its highest-value, most-active customers. There's also tail latency amplification: if one user request fans out into several backend calls, it only takes one of those calls being slow to make the whole request feel slow, so the more backend calls a request makes, the worse your effective tail gets.

Response times are a distribution, not one number. The shaded tail past p95 is where "average looked fine" and "users complained" both live at once.

Maintainability: keep it workable for the next person

Most of the lifetime cost of software is maintenance, not the first version you shipped. The book frames three design goals that make maintenance less painful:

  • Operability: make life easy for whoever operates the system: good monitoring, good docs, predictable behavior, no dependency on any one machine being alive.
  • Simplicity: remove accidental complexity (complexity that comes from the implementation, not from the actual problem). The main tool here is abstraction: a good abstraction hides a mess of detail behind a clean interface. SQL is the book's example, it hides on-disk data structures and concurrent access entirely.
  • Evolvability: make future change easy, because requirements never sit still. This is closely tied to simplicity: the simpler and better-abstracted a system is, the easier it is to bend into a new shape later.

Why it matters

These three qualities are the lens the rest of the book uses to evaluate every technique it covers later (replication, partitioning, transactions, and so on). Every design choice trades off between them. A system that's brutally simple might not scale; a system built for extreme scale might be miserable to maintain. Learning to name these tradeoffs explicitly is what separates "we picked this database because it's popular" from actually designing a data system.

Remember this

  • A fault is one part breaking. A failure is the whole system breaking for the user. Good design keeps the first from becoming the second.
  • Hardware faults are random and independent; software bugs and human error tend to be correlated, which is why they're more dangerous.
  • Never trust an average response time. Look at percentiles, especially the high ones (p95, p99), because that's what your unluckiest users actually experience.
  • "Is it scalable?" is the wrong question. "What happens to this system if load grows this way?" is the right one.
  • Maintainability comes down to three things: is it easy to operate, easy to understand, and easy to change?

Try it yourself

  • Fault injector. Run 3 copies of a tiny worker script (each just prints a heartbeat every second). Write a second script that randomly kills one worker every 10-20 seconds. Watch whether your "system" (the surviving workers) keeps doing its job, or whether losing one worker breaks the whole thing.
  • Percentile meter. Write a script that makes 200 HTTP requests to any endpoint you control (or a public one that allows it) and records each response time. Compute the mean, p50, p95, and p99 yourself from the raw list (sort it, then index into it) instead of using a library, so you feel why the median and the mean disagree.
  • Tail latency amplifier. Build a tiny endpoint that's fast 95% of the time and randomly slow 5% of the time (sleep a couple seconds). Now have a "front door" endpoint that calls it 5 times in parallel and waits for all 5. Compare how often the front door is slow versus how often the backend alone is slow, that's tail latency amplification made visible.
  • Redundancy demo. Run two copies of the same tiny key-value service (even an in-memory dict behind an HTTP endpoint), one "primary" and one "backup." Write a client that talks to the primary, and manually kill it mid-run. Have the client retry against the backup and see what breaks (in-memory data that isn't shared is the obvious gap, which is the whole reason replication exists).
  • Load parameter log reader. Grab any log file you have access to (a web server access log, or generate synthetic timestamps) and compute requests/sec over rolling 1-minute windows. Find your own "peak vs average" number, the way the chapter does for Twitter's post-tweet rate.
  • Chaos toggle. Add a tiny middleware to any web app you already have that throws an error on a random X% of requests. Watch how your existing error handling (or lack of it) behaves, then improve it until a user genuinely can't tell the chaos is happening.

Go deeper

If one thing here is worth re-reading slowly, it's "Percentiles in Practice" (the tail latency amplification section). It's short, but it's the idea that explains why backend systems with many internal calls feel slower in production than any single component's benchmark would suggest, and it sets up why later chapters care so much about how you combine and monitor distributed calls.
