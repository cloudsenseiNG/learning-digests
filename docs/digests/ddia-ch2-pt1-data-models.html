Data Models and Query Languages: Relational vs. Document

The argument that's been running since the 1970s, and why it's really an argument about how connected your data is.

The big idea

Every app needs a way to shape its data before it can store or query it, and that shape is the data model. This chapter (I'm splitting it into two digests since it's a meaty one) covers the two most common shapes: relational tables and document trees, plus the languages we use to ask them questions. The short version: the "right" model isn't a matter of taste, it depends entirely on how tangled your data's relationships are.

Key terms

Impedance mismatch — the awkward gap between how you model data in your app's objects and how a relational database stores it in tables. ORMs (like ActiveRecord or Hibernate) paper over it, they don't remove it.

Normalization — pulling out anything that repeats (like a city name) into its own row, then pointing to it by ID everywhere else, so you only ever update it in one place.

One-to-many relationship — one thing owns many of another (one order has many line items). Nests cleanly into a tree.

Many-to-many relationship — both sides can point at many of each other (a recipe has many tags, a tag applies to many recipes). Doesn't nest, needs a proper link.

Schema-on-write vs. schema-on-read — does the database enforce the shape of your data before it's saved (write), or does your application code just figure out the shape when it reads it back (read)?

Declarative vs. imperative — do you describe what result you want and let the system figure out how (declarative), or do you spell out every step to get there (imperative)?

MapReduce — a way of querying by writing two small functions (map and reduce) that the database calls for you across many machines, instead of writing one query and one instruction to a single server.

One-to-many is easy. Many-to-many is where the fight starts.

Picture a food delivery app. One order has many line items: pizza, garlic bread, a drink. That's one-to-many, and it nests beautifully into a single JSON document:

{
  "order_id": 4821,
  "customer": "Jide A.",
  "items": [
    {"name": "Margherita pizza", "qty": 1},
    {"name": "Garlic bread", "qty": 2}
  ]
}

One document, one read, done. This is the document model's whole pitch: if your data is naturally a tree, storing it as a tree means you don't pay for joins you don't need.

Now add restaurant menus. A menu item (say, "Margherita pizza") belongs to one restaurant, but it can also appear in a "vegetarian" tag, a "under 15 minutes" tag, and a seasonal promotion, all of which touch other menu items too. That's many-to-many, and a tree can't represent it without either duplicating data everywhere or shoving in references that need a real lookup to resolve. This is exactly the problem relational databases were built to solve: normalize the shared stuff (restaurants, tags) into their own tables, reference them by ID, and let a join stitch it back together at query time.

Whether you store an ID or the actual text matters more than it sounds. Store "Vegetarian" as a string on every item, and renaming the tag means updating every item. Store a tag_id that points to one row, and you rename it once. That's normalization: it's not about tables for their own sake, it's about only ever having one place to update a fact.

Schema-on-write vs. schema-on-read

A relational table enforces its shape before anything gets saved: try to insert a row missing a required column and it's rejected on the spot (schema-on-write). A document store usually accepts whatever shape you hand it, and it's on your application code to make sense of it when reading it back (schema-on-read). Neither is free.

Worth knowing: most databases run ALTER TABLE in milliseconds because they just update metadata. MySQL used to be the exception, historically rewriting the whole table on certain schema changes, which could mean real downtime on a big table (modern MySQL and tools like gh-ost or pt-online-schema-change work around this). So "schema-on-write is always slow" is a myth, but it wasn't always a myth on every database.

Declarative wins because it hides the "how"

SQL is declarative: you say what rows you want, and the database's query optimizer decides how to get them (which index to use, which join order, whether to run parts in parallel). An imperative approach (a for-loop checking each row) locks in the "how" yourself, which means the database can never safely change its internals without risking your code's assumptions.

This isn't just a database thing. CSS is declarative ("style every selected item this way") and the browser keeps it correct automatically as the page changes. The equivalent hand-written DOM-walking JavaScript would need to be re-run every time something changes, and it's easy to leave stale styling behind. Same idea, different domain.

MapReduce: query logic as code, not as a query

Some databases (MongoDB is the classic example) let you query by writing two small JavaScript-like functions instead of a single query: a map function that emits a key and value for each matching document, and a reduce function that combines all the values sharing a key. It's neither fully declarative nor fully imperative, more like fill-in-the-blanks code that the database's engine calls repeatedly across many machines.

Google popularized this pattern for exactly that reason: it lets you crunch a huge dataset across a cluster without hand-coordinating the distribution yourself, you just write the two functions and the framework handles spreading the work out. It's genuinely powerful, but writing two coordinated functions for something like "average order value per month" is more ceremony than one SQL GROUP BY. MongoDB felt this pain too: by version 2.2 (released 2012) it shipped a declarative "aggregation pipeline" precisely so people didn't have to reach for MapReduce for routine aggregate queries anymore.

Remember this

  • One-to-many nests cleanly into a document. Many-to-many needs a real join, which is what relational databases are built around.
  • Storing an ID instead of a text value isn't bureaucracy, it's what lets you rename a fact in exactly one place.
  • Schema-on-write pays a cost up front (a migration); schema-on-read defers that cost to every reader. Neither is "no cost."
  • Declarative code (SQL, CSS) describes the result and leaves room for the system to optimize. Imperative code locks in the steps.
  • MapReduce sits between imperative and declarative. It's powerful for distributing work, but most teams reach for a declarative aggregation language when one exists.

Try it yourself

  1. Model the same data two ways. Take a small "blog with posts and comments" idea. Build it as normalized SQL tables (Postgres or even SQLite) and as a single JSON document per post. Write the query "all comments by user X across every post" both ways and count the lines of code.
  2. Race a migration. In SQLite or Postgres, add a column with ALTER TABLE to a table with a few hundred thousand rows and time it. Then simulate schema-on-read: store the same new field only on new rows and write a helper that fills in a default when reading an old row.
  3. Build a join emulator. Given two in-memory arrays (say, orders and customers), write plain code that joins them by ID without using your database. Then run the equivalent single SQL JOIN and compare effort.
  4. Declarative vs. imperative in the browser. Open dev tools on any page. Write a CSS rule that highlights .selected items, then toggle the class and watch it update itself. Now write the equivalent in raw JS DOM code and notice what breaks if you forget to re-run it.
  5. Write a tiny MapReduce. In Python or JS, write a map() and reduce() pair (no libraries) that counts orders per month from a list of dictionaries. Then write the same thing as one line of Pandas groupby or SQL GROUP BY and feel the difference.
  6. Normalize by hand. Take a spreadsheet of orders with a repeated "customer city" column. Pull city into its own table with an ID, rename one city, and see how many rows you touch in each version.
