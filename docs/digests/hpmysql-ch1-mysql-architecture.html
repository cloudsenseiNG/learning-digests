MYSQL · CH.1 DIGEST

MySQL Architecture

In one line: MySQL isn't really one program, it's a layered stack where the top layer handles connections and query smarts, and a swappable bottom layer actually stores your data on disk.

The big idea

Most databases you hear about are one solid block: you send it SQL, it stores your rows, done. MySQL is built differently. It splits the job into layers, and the layer that actually reads and writes data on disk, called a storage engine, is pluggable. You can swap it out, and different tables in the same database can even use different ones. That single design choice explains a lot of what makes MySQL behave the way it does: how locking works, how transactions work, why "REPEATABLE READ" means something specific in MySQL, all of it traces back to which storage engine is doing the actual work underneath.

This chapter is the map. It walks through the three layers of MySQL, how it juggles multiple people reading and writing at once (concurrency control), what a transaction actually guarantees (ACID), and why InnoDB, the default storage engine, is the one worth understanding deeply.

Key terms

  • Storage engine — the component that actually stores and fetches your data from disk. MySQL ships with several; InnoDB is the default and the one you'll use almost always.
  • Storage engine API — the fixed set of low-level operations (like "begin a transaction" or "fetch the row with this primary key") that MySQL's server layer uses to talk to whichever storage engine is plugged in. This is what makes engines swappable: the server doesn't need to know the engine's internals, just how to call this API.
  • InnoDB — MySQL's default storage engine. Supports transactions, row-level locking, and automatic crash recovery.
  • Transaction — a group of SQL statements bundled so they succeed or fail together, as one unit. No "half applied" state.
  • ACID — four properties (atomicity, consistency, isolation, durability) that a transaction system needs to be considered safe. Defined properly further down.
  • Concurrency control — the rules that stop multiple people reading and writing the same data at the same time from stepping on each other.
  • Lock — a claim a client places on some piece of data so others can't change it (or sometimes can't even read it) while the claim is held.
  • MVCC (multiversion concurrency control) — a technique where the database keeps multiple versions of a row around so readers can see a consistent snapshot without blocking writers. Explained in detail below.
  • Isolation level — a setting that controls how much one transaction is allowed to "see" of another transaction's in-progress changes.
  • Deadlock — when two transactions are each waiting on a lock the other one holds, so neither can ever proceed unless something intervenes.
  • Replication — MySQL's way of copying writes from one server (the source) to one or more other servers (replicas), so you have more than one copy of the data.

How it works

1. The three layers

Picture MySQL as three stacked layers. The top layer is the client/connection layer. It's the boring-but-necessary stuff every network service needs: accepting a connection, checking the username and password, deciding what that user is allowed to do. Nothing here is specific to databases.

The middle layer is the server layer, and this is where most of MySQL's "intelligence" lives. It parses the SQL text you send into an internal structure, figures out the smartest way to actually run your query (this is the query optimizer), and hosts anything that's shared across all storage engines, like stored procedures, triggers, and views. Built-in functions (date math, string handling, encryption) live here too.

The bottom layer is the storage engine layer. This is the part that actually touches disk: writing rows, reading rows, maintaining indexes. The server talks to whatever engine a table uses through the storage engine API mentioned above, which is basically a small, fixed menu of commands like "start a transaction" or "give me the row with this primary key." Because that menu is standardized, the server layer barely has to care which engine is underneath. It just calls the API and gets rows back. The engines themselves don't understand SQL at all; they just respond to these low-level requests. (One small exception noted in the book: InnoDB does parse foreign key definitions itself, because the server doesn't implement that piece.)

2. Connections and the optimizer

Each client connection gets its own thread inside the server, and MySQL keeps a pool of ready threads so it doesn't have to spin one up from scratch every time someone connects. Once you're authenticated (username, host, password, or a TLS certificate), the server checks your privileges on every single query, not just at login.

When a query arrives, MySQL parses it into a tree structure, then the optimizer decides things like which order to read tables in and which indexes to use. You can nudge the optimizer with hints, and you can ask it to explain its plan, which is a genuinely useful habit once you're debugging slow queries later in the book.

Simplification: older MySQL versions had a built-in query cache that could serve repeated queries straight from memory. The book notes it became a bottleneck under high concurrency and was removed entirely in MySQL 8.0. If you're on MySQL 8.0+, don't design around it, it's gone. Caching frequently-served results with something like Redis or Memcached, outside MySQL, is the modern equivalent.

3. Concurrency control: locks

Any time two people might change the same data at once, you need rules for who goes first. MySQL handles this at two levels: the server level and, more importantly, the storage engine level (since each engine can implement its own locking).

The classic mental model here is a shared spreadsheet. If you're the only editor, no problem. Add a second editor and you need rules, otherwise one person's edit can silently overwrite another's, or someone reading the sheet can see a half-finished change. The standard fix is two kinds of locks: read locks (shared: many people can hold one at once, since nobody's changing anything) and write locks (exclusive: only one at a time, and it blocks all reads too).

How much you lock at once is called lock granularity, and it's a genuine tradeoff. Lock less (say, one row instead of a whole table) and more people can work concurrently, but tracking lots of small locks costs more overhead. MySQL's storage engines differ here, which is exactly the kind of decision the pluggable-engine design lets MySQL sidestep: it doesn't need one universal answer.

  • Table locks — the simplest, lowest-overhead option. A write lock on a table blocks every other read and write on that whole table until it's released.
  • Row locks — the highest-concurrency option: only the specific rows being changed get locked, so different clients can edit different rows at the same time. It costs more to track (who holds what, for how long) but it's what InnoDB uses, which is why InnoDB tends to handle busy, concurrent workloads much better than table-locking engines.

4. Transactions and ACID

A transaction bundles multiple statements into one all-or-nothing unit. The book's example: moving $200 from checking to savings needs at least three steps (check the balance, subtract from checking, add to savings). If the server crashes between steps, you don't want the customer to just lose $200. Wrapping all three in a transaction means either every step lands, or none of them do.

START TRANSACTION; SELECT balance FROM checking WHERE customer_id = 10233276; UPDATE checking SET balance = balance - 200.00 WHERE customer_id = 10233276; UPDATE savings SET balance = balance + 200.00 WHERE customer_id = 10233276; COMMIT;

For a transaction system to actually be trustworthy, it needs to pass the ACID test:

  • Atomicity — the whole transaction happens, or none of it does. No partial transactions.
  • Consistency — the database moves from one valid state to the next; a crash mid-transaction can't leave money vanished from one account without landing in the other.
  • Isolation — one transaction's in-progress changes are normally invisible to other transactions until it commits (the word "usually" matters, more on that next).
  • Durability — once committed, the change survives a crash. The book is honest that "durable" is fuzzier than it sounds: no system is 100% durable (if it were, backups couldn't add anything).

Worth flagging as a beginner trap: MySQL doesn't manage transactions at the server level, each storage engine implements its own. That means you can't safely mix a transactional engine (InnoDB) and a non-transactional one (like MyISAM) in a single transaction. If a rollback is needed, only the InnoDB part can actually be undone, the other table's change sticks, and now your data is inconsistent with no error necessarily telling you so.

5. Isolation levels

Isolation isn't all-or-nothing, it's a dial with four ANSI SQL settings, each trading safety for concurrency:

SERIALIZABLENoNoNo

Quick definitions since these terms are easy to gloss over: a dirty read is seeing another transaction's uncommitted change. A nonrepeatable read is running the same SELECT twice in one transaction and getting different answers because someone else committed a change in between. A phantom read is re-running a range query and suddenly seeing a brand new row that another transaction inserted in between. Notably, most other databases default to READ COMMITTED, but MySQL defaults to the stricter REPEATABLE READ.

6. Deadlocks

A deadlock happens when two transactions each hold a lock the other one wants, so both wait forever. The book's example: two transactions updating the same two rows of a stock price table, but in opposite order, each grabs one row and then blocks waiting for the other's row. InnoDB detects this and immediately fails one of the transactions (rolling back whichever one holds fewer exclusive locks) rather than letting both hang indefinitely. Applications should expect this and just retry the failed transaction.

7. Transaction logging (write-ahead logging)

Writing straight to the table's on-disk data for every change would be slow, because it means scattered random I/O. Instead, storage engines like InnoDB update the in-memory copy of the data (fast) and append a record of the change to a transaction log on disk (also fast, since appending to a log is sequential I/O). The actual table on disk gets updated later. This is called write-ahead logging: log the intended change before you make it durable in the main data files, so if the server crashes in between, it can replay the log on restart and recover.

8. Multiversion concurrency control (MVCC)

MVCC is InnoDB's trick for letting readers and writers avoid blocking each other most of the time. Instead of a reader having to wait for a writer to finish, MVCC keeps multiple versions of a row around and gives each transaction a consistent "snapshot" of the data as it looked at some point in time.

Roughly how InnoDB does it: every transaction gets an ID. When a row is modified, InnoDB writes an "undo record" (how to reverse the change) before applying it, so an older transaction that shouldn't see the new version yet can walk backward through undo records until it finds the version it's allowed to see. This is also how rollback works: follow the undo trail back.

The upshot: most reads in InnoDB don't need to acquire a lock at all, they just read the right snapshot. The cost is extra bookkeeping (more data stored per row, more work maintaining old versions). MVCC only applies at the READ COMMITTED and REPEATABLE READ isolation levels; READ UNCOMMITTED always reads the newest version regardless, and SERIALIZABLE locks every row it reads, so neither one uses snapshots the same way.

9. Replication

MySQL is designed to take writes on one node at a time (the source) and replicate them out to one or more replicas. The source keeps a dedicated thread per connected replica; when a write happens, that thread wakes up and sends the new data along. The book recommends running at least three replicas for anything in production, ideally spread across different physical locations, purely for disaster recovery.

10. Why InnoDB specifically

InnoDB is the default and, per the book, "the recommended engine to use" for essentially everything, transactional or not, because its crash recovery and general performance are strong even outside heavy-transaction use cases. A few specifics worth knowing: InnoDB organizes each table around a clustered index (the table's rows are physically stored in primary-key order, covered in more depth later in the book), which makes primary-key lookups fast but means a large primary key makes every secondary index bigger too, since secondary indexes carry the primary key value along with them. InnoDB also defaults to REPEATABLE READ and uses "next-key locking" (locking gaps between index entries, not just matching rows) specifically to close the phantom-read hole at that isolation level. Since MySQL 5.7, InnoDB has native JSON column support with validation and functions to query into JSON documents. And since MySQL 8.0, table metadata itself moved into a transactional InnoDB-backed data dictionary (instead of separate file-based `.frm` files), which is also what enables atomic DDL: even schema-changing statements like `ALTER TABLE` can now be wholly applied or wholly rolled back, the same all-or-nothing guarantee transactions give you for data.

Why it matters

This architecture is why so much MySQL tuning advice boils down to "it depends on your storage engine." Locking behavior, isolation guarantees, crash recovery, even index structure all live in the storage engine layer, not the server. Understanding that split is what lets you read MySQL documentation (or this book's later chapters on indexing and replication) and know which layer a given piece of advice is actually about.

It also explains why the ecosystem around MySQL looks the way it does. The storage engine API is a genuine plugin interface, and companies have built entirely custom engines against it rather than forking MySQL itself. A well-documented real example: in 2016, Facebook's engineering team published MyRocks, a storage engine they built on top of RocksDB (an embedded key-value store originally developed at Facebook) and plugged into MySQL through exactly this storage engine API. The motivation was that InnoDB, while reliable, used more disk space and generated more write amplification (extra disk writes beyond the actual data change) than they wanted at Facebook's scale on flash storage. According to Facebook's engineering blog, MyRocks got roughly half the storage footprint of compressed InnoDB on their user-database tier, while keeping MySQL's SQL layer and replication machinery unchanged. That is the pluggable storage engine idea from this chapter, playing out at a scale most of us will never touch, but same mechanism.

Remember this
  • MySQL is layered: connection handling on top, query parsing and optimization in the middle, and a swappable storage engine on the bottom that actually touches disk.
  • The storage engine, not the server, decides how locking, transactions, and crash recovery actually behave. That's why "it depends on the storage engine" is a real answer, not a dodge.
  • InnoDB uses row-level locking plus MVCC (snapshotting old row versions) so most reads never need to wait on a lock at all.
  • A transaction only counts as safe if it's Atomic, Consistent, Isolated, and Durable (ACID). MySQL defaults to REPEATABLE READ, stricter than most databases' default of READ COMMITTED.
  • Never mix transactional and non-transactional storage engines inside one transaction, a rollback can't undo the non-transactional part, and MySQL often won't warn you.
Try it yourself
1. See the pluggable engines for yourself

Install MySQL locally or run it in Docker (docker run --name mysql-lab -e MYSQL_ROOT_PASSWORD=pass -d mysql:8.0 is a standard way to get a throwaway instance). Connect with the mysql client and run SHOW ENGINES;.

Done when: you can point to the row for InnoDB and explain, in your own words, what "DEFAULT" in the Support column means.
2. Watch a transaction actually roll back

Create a small InnoDB table with a numeric balance column and one row. In one session, run START TRANSACTION;, update the balance, then SELECT it back (you'll see your own change). Now run ROLLBACK; and select again.

Done when: you've seen the value change after the UPDATE and then revert after ROLLBACK, proving atomicity isn't just a word in a book.
3. Reproduce isolation with two sessions

Open two separate mysql client sessions (two terminal windows) against the same table. In session A, start a transaction and update a row but don't commit yet. In session B, try to SELECT that row.

Done when: you can describe what session B sees under MySQL's default isolation level, and explain why it isn't a dirty read.
4. Force a real deadlock

Using the two-session setup above and a table with at least two rows, have session A lock row 1 then try to lock row 2, while session B locks row 2 then tries to lock row 1 (mirroring the book's stock price example), all inside open transactions with no commit yet.

Done when: one session gets an error mentioning deadlock, and you can say which transaction MySQL chose to roll back and why (hint: fewest locks held).
5. Compare table locks vs row locks under load

Create one InnoDB table and, if your MySQL build still supports it, one MyISAM table with identical data (MyISAM is deprecated in modern MySQL, so InnoDB with explicit LOCK TABLES is a fine substitute if MyISAM isn't available). Write a tiny script (any language) that opens several connections and fires concurrent UPDATEs against different rows in each table, timing how long the whole batch takes.

Done when: you have a rough number showing the row-locked table finished faster under concurrent writes than the table-locked one.
6. Read your own EXPLAIN output

Take any query against a table with a few hundred rows and run EXPLAIN in front of it.

Done when: you can identify, from the output, whether the optimizer used an index or scanned the whole table.
