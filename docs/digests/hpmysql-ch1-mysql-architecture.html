MySQL Architecture

How MySQL is actually put together underneath the SQL you type, and why almost every "why does MySQL behave this way" question traces back to its layered design.

The big idea

MySQL isn't one monolithic block of code. It's a stack of layers: a top layer that handles connections and security, a middle layer that parses and optimizes your SQL, and a bottom layer, the storage engine, that actually reads and writes data on disk. That bottom layer is swappable, which is unusual for a database. Once you can picture those layers, most of MySQL's quirks (locking behavior, how transactions work, why some features exist only for some tables) stop feeling random and start feeling like consequences of that design.

Key terms

Storage engine — the component that actually stores and retrieves data on disk for a table. MySQL lets you pick a different one per table (InnoDB is the default and the one almost everyone should use).
Storage engine API — the fixed set of low-level operations (like "begin a transaction" or "fetch the row with this primary key") that the server uses to talk to whichever storage engine a table uses, so the rest of MySQL doesn't need to know the engine's internals.
Parser / parse tree — the part of MySQL that reads your raw SQL text and turns it into an internal structure it can reason about, before it decides how to run it.
Optimizer — the part of MySQL that decides how to actually run a query: which table to read first, which index to use, and so on.
Table lock — a lock on an entire table. Cheap to manage, but it blocks every other writer (and sometimes reader) on that table while held.
Row lock — a lock on a single row. More overhead to track, but lets many clients write to different rows of the same table at once.
Transaction — a group of SQL statements treated as one all-or-nothing unit. Either every statement in the group takes effect, or none of them do.
ACID — the four properties (atomicity, consistency, isolation, durability) that describe a "safe" transaction system. Defined in full below.
Isolation level — a setting that controls how much of one transaction's in-progress work another transaction is allowed to see.
Deadlock — a standoff where two transactions each hold a lock the other one needs, so neither can proceed.
Write-ahead logging — writing a record of a change to an on-disk log before (or instead of, in the short term) updating the actual data files, so a crash doesn't lose the change.
MVCC (multiversion concurrency control) — a technique where the database keeps multiple versions of a row around so that readers can see a consistent snapshot without locking anything.
Replication — MySQL's built-in way of copying writes made on one server to one or more other servers.

How it works

Think of MySQL less as "a database" and more as a small pipeline: a request comes in, gets authenticated, gets parsed and planned, and finally gets handed to whichever storage engine owns the data. Each stage is a separate layer with a separate job, and the layers only talk to each other through narrow, defined interfaces. That separation is the whole reason MySQL can support several genuinely different storage engines (each with its own tradeoffs) while still speaking one SQL dialect on top. The rest of this section walks through each layer and the mechanics (locking, transactions, MVCC) that live inside it.

MySQL's logical architecture

MySQL's architecture is described as three logical layers stacked on top of each other:

  1. The client/connection layer handles things that aren't specific to MySQL at all: accepting network connections, authenticating who's connecting, and checking security/privileges. Most network client-server tools need something like this.
  2. The server layer is where "the brains" of MySQL live: parsing your SQL, analyzing and optimizing it, and running built-in functions (dates, math, encryption, and so on). Anything that works the same way no matter which storage engine a table uses, like stored procedures, triggers, and views, also lives here.
  3. The storage engine layer is responsible for actually storing and fetching the data. Each storage engine has its own strengths and weaknesses, similar to how different filesystems on Linux trade off differently.

The server talks to storage engines through the storage engine API, a set of maybe a couple dozen low-level operations like "begin a transaction" or "fetch the row with this primary key." This is what makes storage engines swappable: the server doesn't need to know how InnoDB stores a row on disk versus how any other engine would. Storage engines don't parse SQL themselves (InnoDB is a partial exception: it does parse foreign key definitions, because the server doesn't implement that itself) and they don't talk to each other directly. They just answer requests from the server.

Connection handling and security

By default, every client connection gets its own thread inside the server process, and that connection's queries all run on that one thread (which sits on one CPU core). Spinning up a thread from scratch for every connection would be wasteful, so the server keeps a cache of ready-to-use threads instead of creating and destroying one for each new connection.

When a client connects, MySQL authenticates it using username, originating host, and password (or, over a TLS connection, an X.509 certificate). After that, before running each query, the server checks whether that client actually has permission to do it, like whether it's allowed to run a SELECT against a particular table.

Optimization and execution: the SQL layer

Once a query gets past the connection layer, MySQL parses it into an internal structure called a parse tree and then runs a set of optimizations on it: rewriting the query, deciding what order to read tables in, choosing which indexes to use, and more. You can nudge the optimizer with hints in the query itself, and you can ask MySQL to explain its own decisions, which is useful when you're trying to work out why a query is slow.

Here's a subtlety worth internalizing: the optimizer doesn't care what storage engine a table uses, but the storage engine still shapes the outcome, because the optimizer asks the storage engine what it's capable of, roughly how expensive certain operations are, and for statistics about the table's data. Some storage engines support index types that help specific kinds of queries, which is one reason the choice of storage engine per table actually matters.

A simplification worth flagging: older MySQL versions had a built-in query cache that could serve some results without rerunning a query. As of MySQL 5.7.20 it was deprecated, and by MySQL 8.0 it's removed entirely, so don't assume a query cache exists inside current MySQL. Caching query results is still a good idea in general, it's just typically done outside the server now (a separate cache like memcached or Redis, for example) rather than built in.

The pluggable storage engine layer, and InnoDB specifically

The storage engine is the piece that actually decides how data lives on disk (or doesn't, at all, for a given moment) and how it's read back. Because MySQL treats this as a swappable layer behind a shared API, different tables in the same database can even use different storage engines, though in practice almost everyone standardizes on one.

InnoDB is the default storage engine and, per the chapter, "the most important and broadly useful engine overall." It's built for lots of short transactions that usually succeed rather than get rolled back, and its automatic crash recovery makes it a common default even for data that doesn't strictly need transactions. A few concrete things worth knowing about it:

  • InnoDB stores its data in datafiles collectively called a tablespace, which InnoDB manages internally, essentially a black box from the outside.
  • InnoDB tables are built on a clustered index (the table data itself is physically organized by the primary key), which makes primary-key lookups fast, but means secondary indexes also carry a copy of the primary key columns. A large primary key makes every other index on that table larger too.
  • It implements all four ANSI SQL isolation levels and defaults to REPEATABLE READ (more on isolation levels below), using a "next-key locking" strategy that locks gaps between index entries, not just matching rows, to prevent certain anomalies.
  • Starting in MySQL 5.7, InnoDB gained native JSON document support with validation and optimized storage, an upgrade over the older pattern of stuffing JSON into a plain BLOB column.
  • MySQL 8.0 introduced atomic DDL: schema-changing statements (like adding a column) can now either fully succeed or fully roll back, the same all-or-nothing guarantee transactions give you for data changes.

Simplification: MySQL does ship other storage engines (MyISAM is the other one mentioned in this chapter), and the general "pick the right engine per table" idea is real, but in practice this and later material treats InnoDB as the engine to reach for almost every time, so that's the mental default worth keeping.

Concurrency control: locks

Concurrency control is the general problem of what happens when more than one query wants to change the same data at the same time. The chapter's own analogy: imagine a spreadsheet that only you can edit versus one on a shared server that a whole team edits at once. Reading isn't too dangerous since multiple people reading at once doesn't corrupt anything, but if someone deletes a cell while others are reading, a reader can end up with an inconsistent view. A database table has exactly the same problem.

The classic fix is two kinds of locks. Read (shared) locks don't block each other: many clients can read the same resource at once with no conflict. Write (exclusive) locks block everything else, both other writes and reads, because the only safe policy is one writer at a time with no one reading mid-change.

How much you lock at once (the lock granularity) is a tradeoff. Lock less and more operations can run concurrently, but locks themselves cost overhead to acquire, check, and release, so locking too finely can hurt performance too. MySQL doesn't force one answer here; different storage engines implement their own locking policy and granularity, which is one more reason the engine you pick per table matters.

  • Table locks are the simplest and cheapest: a write lock on a table blocks all other reads and writes to it; once nobody's writing, readers can take read locks freely. (MySQL has variants like READ LOCAL table locks that allow some concurrent writes, and write requests queue ahead of read requests.)
  • Row locks give the most concurrency (multiple clients can edit different rows of the same table at once) at the cost of more bookkeeping overhead: tracking who holds which row lock, for how long, and cleaning them up afterward. Row locks are implemented inside the storage engine itself, not the server, so the server is largely unaware of them.

Transactions and ACID

A transaction groups SQL statements into one all-or-nothing unit: either the whole group applies, or, if something goes wrong partway (a crash, for instance), none of it applies. The chapter's example is moving $200 from checking to savings, which takes at least three steps (check the balance, subtract from checking, add to savings). Wrap all three in a transaction and a failure partway through rolls everything back instead of leaving the money in limbo.

You start one with START TRANSACTION, then either make it permanent with COMMIT or discard it with ROLLBACK.

A transaction system that's actually safe needs to satisfy ACID:

  • Atomicity — the transaction is one indivisible unit: it's either fully applied or never committed, never partially done.
  • Consistency — the database always moves from one consistent state to the next; an interrupted transaction shouldn't leave, say, money subtracted from one account but never added to the other.
  • Isolation — a transaction's results are usually invisible to other transactions until it finishes (more nuance on the word "usually" below).
  • Durability — once committed, a transaction's changes survive a crash. The chapter is upfront that durability is "a slightly fuzzy concept": some durability strategies are stronger than others, and nothing is ever 100% durable in an absolute sense (if it were, backups couldn't add anything).

In MySQL specifically, ACID transactions live at the storage-engine level, and InnoDB's implementation of them is one of its strongest, most mature features.

Isolation levels and deadlocks

Isolation turns out to be more complicated than "one transaction can't see another's unfinished work." The ANSI SQL standard defines four levels, and lower levels generally allow more concurrency at the cost of stricter guarantees:

Isolation levelDirty reads?Nonrepeatable reads?Phantom reads?
READ UNCOMMITTEDYesYesYes
READ COMMITTEDNoYesYes
REPEATABLE READ (MySQL's default)NoNoYes*
SERIALIZABLENoNoNo

*In InnoDB specifically, REPEATABLE READ also closes off phantom reads in practice, via MVCC and gap locking, covered in the next section.

A dirty read is reading data another transaction hasn't committed yet. A nonrepeatable read is running the same query twice in one transaction and getting different results because another transaction committed a change in between. A phantom read is selecting a range of rows, having another transaction insert a new row into that range, then re-running the same selection and seeing a new "phantom" row that wasn't there before.

Bounded claim: READ COMMITTED is the default isolation level for most database systems, but not MySQL, whose default is REPEATABLE READ. It's a detail worth remembering if you're used to another database.

A deadlock happens when two or more transactions each hold a lock the other one is waiting for, forming a cycle nobody can escape without help. This is a real risk whenever transactions lock the same resources in a different order. More sophisticated engines like InnoDB detect the circular wait and immediately fail one of the transactions with an error rather than let both hang forever; InnoDB specifically rolls back whichever transaction is holding the fewest exclusive row locks, as a rough approximation of "easiest to undo." Once a deadlock happens it can't be resolved without rolling back at least one of the transactions involved, so applications that use transactions should expect deadlocks occasionally and be built to retry.

Under the hood, most storage engines make transactions efficient with write-ahead logging: instead of updating the on-disk table immediately, the engine updates its in-memory copy and appends a record of the change to a transaction log on disk (fast, because it's sequential I/O in one spot rather than scattered random I/O). The actual table gets updated from that log later. If the server crashes after the log write but before the table update, the engine can still recover using the log. The cost is that most data technically gets written to disk twice.

Multiversion concurrency control (MVCC)

Most of MySQL's transactional storage engines don't rely on plain row locking alone. They pair row-level locking with MVCC, which keeps snapshots of data as it existed at a point in time so a transaction can see a consistent view of the data no matter how long it runs, and so different transactions can legitimately see different versions of the same row at the same moment. (MVCC isn't MySQL-specific; Oracle and PostgreSQL use it too, though there's no formal standard, so implementations genuinely differ.)

Here's roughly how InnoDB does it: every transaction gets a transaction ID, assigned the first time it reads any data. When a row changes inside a transaction, InnoDB writes an undo record (a note on how to reverse that change) to the undo log, and points the transaction's rollback pointer at it, which is how a rollback knows what to undo. When a different session reads a row, InnoDB compares that row's transaction ID against the reading session's own view. If the row's current version shouldn't be visible yet (the transaction that changed it hasn't committed), InnoDB walks backward through the undo log until it finds a version that is visible, potentially all the way back to "this row didn't exist yet." Rows aren't immediately physically deleted either, a "deleted" flag gets set first and tracked in the undo log.

The payoff: most read queries in InnoDB never need to take a lock at all, they just read data as fast as they can. The cost: the storage engine stores more data per row, does more work examining rows, and has extra bookkeeping to manage.

Bounded claim: MVCC works with the REPEATABLE READ and READ COMMITTED isolation levels. READ UNCOMMITTED isn't MVCC-compatible (it just always reads the newest version, snapshot or not), and neither is SERIALIZABLE (it locks every row it reads instead). So "reads don't take locks" is a property of MVCC at those two specific isolation levels, not an absolute rule for every setting.

Transactions in MySQL: AUTOCOMMIT, mixed engines, and explicit locks

By default MySQL runs in AUTOCOMMIT mode: a single INSERT, UPDATE, or DELETE is implicitly wrapped in its own transaction and committed right away. Turning AUTOCOMMIT off lets you run a series of statements as one transaction, ending with an explicit COMMIT or ROLLBACK. Some commands (DDL statements like ALTER TABLE, plus LOCK TABLES and a few others) force MySQL to commit whatever transaction is currently open before they run.

Because MySQL doesn't manage transactions at the server level, only inside each storage engine, you can't reliably mix engines within one transaction. Mixing a transactional table (InnoDB) with a nontransactional one in the same transaction mostly works fine until you need to roll back: the nontransactional table's changes can't be undone, which can leave the database inconsistent. Simplification worth naming: this is specifically about mixing engines within a single transaction, not a claim that you can never use more than one storage engine in the same database.

InnoDB uses a two-phase locking protocol: it can acquire locks at any point during a transaction, but doesn't release any of them until COMMIT or ROLLBACK, releasing them all at once. Most of that locking is implicit, handled automatically based on your isolation level, but InnoDB also supports explicit locking reads with SELECT ... FOR SHARE and SELECT ... FOR UPDATE. MySQL separately supports LOCK TABLES / UNLOCK TABLES, implemented at the server level rather than inside a storage engine, though the general guidance is to avoid it if you're using a transactional engine like InnoDB, since row-level locking already gives you what you need.

Replication and datafiles, briefly

MySQL is designed so that, in a common setup, one node accepts writes at any given time, and it offers built-in replication to copy those writes out to other servers (the source server runs a thread per replica that wakes up and ships new data whenever a write happens). Bounded claim: that's describing a common source-and-replica topology, not a universal law: MySQL also supports other configurations. For anything running in production, the chapter's own guidance is to use replication with at least three replicas, ideally spread across different locations, for disaster-recovery purposes.

On the storage side, MySQL 8.0 changed how it keeps track of table metadata (column definitions, and so on): instead of separate .frm files, table structure now lives in a transactional data dictionary backed by InnoDB, with a small in-memory LRU cache in front of it and a per-table .sdi file (serialized dictionary information) replacing the old files. Practically, this means metadata changes get the same crash-recovery and atomicity guarantees regular data changes get.

Why it matters

Nearly everything else you'll learn about MySQL performance and behavior sits on top of this chapter. Slow query? You're probably diagnosing the SQL layer (the optimizer's choices) or the storage engine layer (how InnoDB is reading rows). Confusing "why did my SELECT return different results than I expected" moment? That's isolation levels and MVCC. An app that hangs under load? Very possibly a lock, and now you know there's a real difference between a cheap table lock and a more concurrent but heavier row lock. Once these layers and mechanisms are visible to you, most MySQL troubleshooting turns into "which layer is this actually happening in," instead of guessing.

Remember this

  • MySQL is layered: connection handling on top, the SQL brain (parser + optimizer) in the middle, swappable storage engines at the bottom, all talking through one storage engine API.
  • InnoDB is the default storage engine and the one to reach for almost every time; it's what gives you transactions, row-level locking, and crash recovery.
  • Table locks are cheap but block everything; row locks allow far more concurrency at the cost of extra bookkeeping.
  • MySQL's default isolation level is REPEATABLE READ, not READ COMMITTED like most other databases.
  • MVCC is why most reads in InnoDB never need a lock: they follow the undo log back to the version of a row they're allowed to see instead of waiting for a writer to finish.

Try it yourself

  • See the storage engines available. Install MySQL locally (or run the official Docker image) and run SHOW ENGINES;. Note which engine is marked "DEFAULT" and confirm it's InnoDB.
  • Check a table's engine. Create a small table, then run SHOW TABLE STATUS LIKE 'yourtable'; (or query information_schema.tables) to see which storage engine it's actually using. Try ALTER TABLE yourtable ENGINE=MyISAM; and check again, then switch it back.
  • Watch a write lock make a reader wait. Open two terminal sessions connected to the same database. In session 1, run START TRANSACTION; then update a row without committing. In session 2, try to update that same row and watch it hang until session 1 commits or rolls back.
  • Reproduce a deadlock on purpose. Create a table with at least two rows. In session 1, start a transaction and update row A, then (before committing) try to update row B. In session 2, start a transaction and update row B, then try to update row A. One session should get a deadlock error almost immediately. Read the error message.
  • Compare isolation levels. In one session, run SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;, start a transaction, and run the same SELECT twice with a gap in between. In another session, commit a change to that row in between the two selects. Repeat the whole experiment with REPEATABLE READ and compare what each session sees the second time.
  • Turn off AUTOCOMMIT and roll back on purpose. Run SET autocommit=0;, make a few changes, then run ROLLBACK; instead of COMMIT;. Confirm none of your changes persisted.
