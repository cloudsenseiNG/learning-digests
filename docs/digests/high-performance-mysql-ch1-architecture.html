MySQL Architecture

Why MySQL is really two databases stacked on top of each other, and why almost everything about how it behaves under load traces back to that split.

In one line

MySQL hands your query to the same query-processing brain no matter what, but the actual reading and writing of data is done by a swappable "storage engine" underneath, and that engine is what decides how locking, transactions, and crash recovery actually behave.

The big idea

Most databases you've heard of are one solid block: one way of storing data, one way of locking rows, one way of handling a crash. MySQL isn't built that way. It's split into a top half that's the same for every query (parsing your SQL, deciding how to run it) and a bottom half, the storage engine, that's pluggable. InnoDB is the one you'll use almost all the time today, but the server itself doesn't know or care that it's talking to InnoDB specifically, it just calls a generic interface. Once that separation clicks, a lot of "why does MySQL do it that way" questions answer themselves.

Key terms

Storage engine — the component actually responsible for storing and retrieving your data on disk. e.g. InnoDB is a storage engine; MySQL itself is not.
Storage engine API — the fixed, low-level interface the server uses to talk to whatever storage engine a table uses (things like "begin a transaction" or "fetch the row with this primary key"). This is what makes engines swappable.
Transaction — a group of SQL statements treated as a single all-or-nothing unit of work.
ACID — Atomicity, Consistency, Isolation, Durability. The four properties a transaction system needs before you can call it "safe."
Isolation level — a setting that controls how much of one transaction's in-progress work another concurrent transaction is allowed to see.
Dirty read — reading a change another transaction made but hasn't committed yet (and might still roll back).
Nonrepeatable read — running the exact same SELECT twice inside one transaction and getting two different answers, because someone else committed a change in between.
Phantom read — you SELECT a range of rows, someone else inserts a new row that fits that range and commits, you run the same SELECT again and a "phantom" row has appeared.
Deadlock — two transactions each hold a lock the other one is waiting for, so neither can ever finish.
MVCC (multiversion concurrency control) — a technique where the engine keeps older versions of a row around so a reader can see a consistent snapshot without ever needing to lock anything.
Write-ahead logging — writing a record of a change to a sequential on-disk log first (fast), and updating the actual table data on disk later (slower), instead of doing the slow part on every single change.
AUTOCOMMIT — MySQL's default mode, where every single statement is automatically its own tiny transaction that commits immediately.
Source / replica — the modern MySQL terms for the one server writes originate from, and the servers that receive a streamed copy of those writes.
Clustered index — how InnoDB physically stores a table: the rows themselves are sorted and stored by primary key, not stored separately from the index.

How it works

Before any of the individual pieces make sense, it helps to see the whole shape at once: a query comes in from a client, gets handled by server-side code that's identical regardless of what's storing your data, and only right at the very end does it touch a specific storage engine that actually reads or writes bytes. Everything below is really an elaboration of that one idea, plus what happens once more than one query wants to touch the same data at the same time.

The three-layer architecture

MySQL is organized in three layers stacked on top of each other. The top layer is clients: this is generic network server stuff, not specific to databases at all, connection handling, authentication, security. The middle layer is the server itself, and this is where most of MySQL's "brains" live: parsing your SQL into an internal structure, deciding how to optimize and run it, and all the built-in functions (dates, math, encryption). Anything that works the same no matter which storage engine you're using, like stored procedures, triggers, and views, lives here too. The bottom layer is the storage engines, responsible for actually storing and retrieving the data. The server talks to whichever engine a table uses through the storage engine API, a set of low-level operations like "begin a transaction" or "fetch the row with this primary key." Storage engines don't parse SQL and don't talk to each other, they just respond to what the server asks of them.

The "why" here matters: because the top two layers don't care which engine is underneath, MySQL can offer very different storage engines (each with different trade-offs) without you having to relearn SQL or query behavior every time you pick one.

Connections, threads, and security

Each client connection gets its own thread inside the server process, and that connection's queries all run on that one thread. Creating and destroying a thread for every new connection would be wasteful, so the server keeps a cache of ready-to-use threads instead. When a client connects, MySQL authenticates it based on username, the host it's connecting from, and its password (or an X.509 certificate over TLS). After that, every single query the client sends gets checked against that client's privileges, for example, whether it's even allowed to SELECT from a particular table.

The optimizer, and the query cache that disappeared

MySQL parses your query into an internal "parse tree" and then applies a set of optimizations: rewriting the query, deciding what order to read tables in, choosing which indexes to use. You can nudge this with optimizer hints in the query itself, and you can ask the server to explain its decisions, which gives you a starting point for reworking a slow query. The optimizer doesn't care which storage engine a table uses, but the choice of engine absolutely affects the optimizer's decisions, because it asks the engine what it's capable of and roughly how expensive certain operations will be. One historical note worth knowing: older MySQL versions had a built-in query cache that tried to serve repeat queries straight from memory, but it became a bottleneck as concurrency grew, was deprecated as of MySQL 5.7.20, and was fully removed in 8.0. If you need that kind of caching today, it's typically done outside MySQL entirely (an external cache like Redis or memcached), which the chapter mentions but doesn't cover in depth.

Why concurrency needs locks at all

Picture a spreadsheet that only you can open: no conflicts possible, because you're the only writer. Now put that same spreadsheet on a shared server with a whole team editing it. Reading isn't really a problem, multiple people can look at it at once with nothing going wrong. But if someone deletes a cell while others are reading, a reader could end up seeing a corrupted or half-changed view. A database table has exactly this same problem. The standard fix is two kinds of locks: read (shared) locks, which don't block each other, so many clients can read at once, and write (exclusive) locks, which block everything else, both other writes and other reads, because the only safe way to change something is to be the only one touching it at that moment.

Lock granularity: table locks vs. row locks

Locking has a "how much do you lock at once" dial, and where you set it is a genuine trade-off. A table lock locks the entire table: cheap to manage (low overhead), but it means only one write can happen anywhere in that table at a time. A row lock locks just the specific rows being changed, so unrelated writes to other rows can happen at the same time, but the engine now has to track who holds which row lock, for how long, and clean it all up, which is real overhead. Locks aren't free: every lock, check, and release costs something, so if a system spends more time managing locks than actually doing work, performance suffers. MySQL's different storage engines can each pick their own locking strategy and granularity here, which is exactly why the choice of engine matters so much for how a table behaves under concurrent load.

Transactions and the ACID properties

A transaction groups several SQL statements into one all-or-nothing unit: either the whole group succeeds, or none of it does. The classic example is moving money between two accounts: check the source balance is high enough, subtract from it, then add to the destination. If your app did those as three separate, un-grouped statements and the process crashed between step two and step three, money would simply vanish. Wrapping it in a transaction (START TRANSACTION ... COMMIT, or ROLLBACK to cancel) means either all three steps land or none of them do. For a transaction system to actually be trustworthy, it needs to satisfy four properties, ACID:

  • Atomicity — the whole transaction is one indivisible unit; there's no such thing as "half applied."
  • Consistency — the database moves from one valid state straight to the next valid state; a crash mid-transaction can't leave the money missing.
  • Isolation — other transactions can't see this one's in-progress changes until it commits (with some nuance, see isolation levels below).
  • Durability — once committed, the change survives, even a crash right afterward. The chapter is honest that "durable" is fuzzy in practice: no storage is ever 100% durable, which is exactly why backups exist on top of durability guarantees.

Isolation levels: how much can transactions see of each other

Isolation sounds simple until you ask "invisible to what degree, exactly." The ANSI SQL standard defines four levels, and lower levels generally mean more concurrency but weaker guarantees. MySQL's default is REPEATABLE READ, which is worth noting because plenty of other database systems default to READ COMMITTED instead, so don't assume MySQL behaves like whatever you used last.

*InnoDB specifically closes this phantom-read gap at REPEATABLE READ using MVCC plus "next-key locking" (more on both below), so in practice InnoDB avoids phantoms too, even though the ANSI standard says REPEATABLE READ alone doesn't guarantee that.

READ UNCOMMITTED lets you see another transaction's changes before it even commits (a "dirty read") and is rarely used because the performance upside isn't big enough to justify the risk. SERIALIZABLE is the strictest: it locks every row it reads, which can cause a lot of lock waits and timeouts, and the book notes it's rarely used in practice. REPEATABLE READ sits in the middle and is what you'll actually be running by default with InnoDB.

Deadlocks

A deadlock happens when two transactions each hold a lock the other one needs, so both wait forever unless something intervenes. Picture two transactions touching the same two rows in opposite order: Transaction 1 updates row A then tries to update row B; Transaction 2 has already updated row B and now tries to update row A. Each is waiting on a lock the other is holding, a cycle with no way out on its own. InnoDB detects this and breaks the cycle immediately by rolling back whichever transaction is holding the fewest exclusive row locks (a rough proxy for "cheapest to undo"), rather than letting both hang forever. Some other systems just wait for a timeout instead, which is worse because a real deadlock then looks like a slow query instead of a clear error. The practical takeaway from the chapter: deadlocks are a fact of life in any system with concurrent writes, and applications should be built to simply retry a failed transaction from the top.

Transaction logging (write-ahead logging)

Updating a table's on-disk data on every single change would mean constant, scattered ("random") disk I/O, which is slow. Instead, the storage engine changes its in-memory copy of the data (fast) and appends a record of that change to a transaction log on disk (also fast, because it's sequential writing to one small area, not scattered writing across the whole table). The actual table data on disk gets updated later. This means most engines using this approach, known as write-ahead logging, end up writing every change twice: once to the log, once eventually to the table. The payoff is crash safety: if the server crashes after a change hits the log but before the table itself is updated, the engine can replay the log on restart and recover the change. The exact recovery process differs by storage engine.

Transactions in MySQL: AUTOCOMMIT, and never mixing engines mid-transaction

By default MySQL runs in AUTOCOMMIT mode: every single INSERT, UPDATE, or DELETE is implicitly its own transaction, committed immediately. Turning AUTOCOMMIT off (or explicitly starting with START TRANSACTION) lets you group several statements into one transaction that you commit or roll back yourself. A few DDL-style commands (like ALTER TABLE, or LOCK TABLES) will force-commit whatever transaction is currently open before they run, so it's worth checking your version's docs for the full list. You can also set the isolation level per session or server-wide with SET SESSION TRANSACTION ISOLATION LEVEL ....

One sharp edge worth calling out clearly: MySQL doesn't manage transactions at the server level at all, each storage engine manages its own. That means if you mix a transactional table (InnoDB) and a non-transactional table (like MyISAM) inside one transaction, everything looks fine as long as nothing goes wrong. But if a rollback is ever needed, the non-transactional table's changes simply can't be undone, leaving your data in an inconsistent, hard-to-recover state, often with no warning at all. The chapter is blunt about this: don't mix storage engines within a single transaction.

MVCC: how InnoDB lets most reads skip locking entirely

Row-level locking alone still means a reader can end up waiting on a writer. MVCC (multiversion concurrency control) is a technique for avoiding that in most cases by keeping multiple historical versions of a row around, so a reader can see a consistent snapshot from a specific point in time, without taking a lock at all. InnoDB does this by giving every transaction a transaction ID, assigned the first time it reads anything. When a row is modified, InnoDB writes an "undo record" (basically: instructions for how to reverse this exact change) to an undo log, and points the transaction at that record. When a different session reads a row, InnoDB compares the row's transaction ID to that session's own view: if the row's current version isn't supposed to be visible yet (because the transaction that changed it hasn't committed), InnoDB walks backward through the undo log until it finds a version that is visible to that reader. The result is that most reads never need to acquire a lock at all, they just read as fast as they can. The cost is real too: more data stored per row, more work when scanning, and extra bookkeeping. One important limit: MVCC only works with REPEATABLE READ and READ COMMITTED. READ UNCOMMITTED always reads the newest version regardless of version history, and SERIALIZABLE locks every row it reads, so neither one is compatible with MVCC's whole premise.

Replication

MySQL only accepts writes on one node at a time. That node is called the source, and it can distribute those writes out to one or more replicas. The source runs a dedicated thread per connected replica; that thread wakes up when a write happens and sends the new data along. This is the backbone of how MySQL scales reads and survives a server going down: the book recommends that any production setup use replication with at least three replicas, ideally spread across different physical locations, for disaster recovery.

Inside InnoDB

InnoDB is now MySQL's default, general-purpose storage engine, and the book is explicit that it's worth understanding deeply rather than spreading your attention across every engine equally. A few specifics worth knowing: InnoDB stores its data in files collectively called a tablespace, which it manages internally. Tables are built on a clustered index, meaning the rows themselves are physically stored sorted by primary key, which makes primary-key lookups fast, but it also means every secondary (non-primary-key) index has to carry a copy of the primary key value, so a large primary key makes all your other indexes larger too. InnoDB defaults to REPEATABLE READ and uses "next-key locking" to close the phantom-read gap at that level: instead of only locking the rows a query actually touched, it also locks the gaps between index entries, so a new row can't sneak into a range that was just queried. As of MySQL 8.0, table metadata itself moved into an InnoDB-backed data dictionary (replacing the old separate .frm files with serialized .sdi files per table), which brings the same crash-safety and transactional guarantees to schema changes themselves, this is also what enables atomic DDL: a schema change (like adding a column) now either fully completes or fully rolls back, backed by its own dedicated undo and redo log, instead of possibly leaving a table half-altered after a crash. Separately, since MySQL 5.7, InnoDB also supports a native JSON column type with built-in validation and storage optimized for fast reads (an upgrade over storing JSON as a plain BLOB), and as of 8.0.7 you can even build multivalued indexes directly on values inside a JSON array.

Why it matters

The single-writer replication model in this chapter is the exact design behind a real, dated incident: on October 21, 2018, a 43-second network blip inside GitHub's data centers caused their automated failover tooling to promote a West Coast MySQL replica to become a second writer, while the original East Coast source kept accepting writes too. For a short window, two servers both believed they were "the" source, and both took writes the other never saw. GitHub chose to prioritize not losing any data over getting back online fast, and full recovery took just over 24 hours. That's the practical stakes behind "MySQL only accepts writes on one node at a time": the moment tooling around it can be fooled into believing two nodes are both writable, you get exactly this kind of split-brain mess. Beyond replication, everything else in this chapter (locking, isolation levels, MVCC, transaction logging) is the machinery that makes a single MySQL server safe to write to concurrently in the first place, and it's exactly what you'll be tuning later in the book when performance problems show up as lock waits, replication lag, or a query that "used to be fast."

Remember this

  • MySQL is layered: a client layer handles connections and auth, a server "brain" parses and optimizes every query the same way, and a swappable storage engine underneath actually reads and writes the bytes.
  • InnoDB, the default engine, combines row-level locking with MVCC, which is why most reads never have to wait on a lock at all.
  • MySQL's default isolation level is REPEATABLE READ, not READ COMMITTED like a lot of other databases assume by default, so don't carry assumptions over from another database blindly.
  • A transaction isn't automatically safe just because you wrapped it in START TRANSACTION / COMMIT: mixing a transactional table with a non-transactional one in the same transaction can leave you half-committed with no warning at all.
  • Deadlocks are an expected outcome of concurrent writes, not a bug. Design your app to retry a failed transaction from the top instead of assuming it'll never happen.

Try it yourself

  • Reproduce the transfer example. Install MySQL or MariaDB locally (or via Docker). Create a table with a checking and a savings balance, run START TRANSACTION, subtract from one, add to the other, COMMIT. Then run it again and kill the client mid-transaction before COMMIT, reconnect, and confirm nothing changed. That's atomicity, not a promise, an observable behavior.
  • Cause a deadlock on purpose. Open two terminal sessions against a table with at least two rows. In each session, without committing, update the two rows in opposite order. Watch InnoDB detect the cycle and kill one of your transactions with a deadlock error.
  • Compare table locks vs. row locks. Create the same table twice, once as MyISAM and once as InnoDB. From two sessions, write to different rows of each at the same time and time how long the second write waits in each engine.
  • Watch a nonrepeatable read happen. Open two sessions. Set one to READ COMMITTED, SELECT a row, have the other session UPDATE and COMMIT that same row, then SELECT again in the first session and watch the value change mid-transaction. Repeat at REPEATABLE READ and watch it stay put instead.
  • See MVCC give you a nonlocking read. Start a transaction, UPDATE a row, don't commit yet. From a second session, run a plain SELECT on that row. Confirm it returns immediately (no blocking) and shows the old, committed value, not your uncommitted change.
  • Build a two-node replication pair. Two MySQL containers work fine: one source, one replica. Write a row on the source and measure how long it takes to show up on the replica. Then disconnect the replica, write more rows on the source, reconnect, and watch it catch up, that's replication lag made visible.
