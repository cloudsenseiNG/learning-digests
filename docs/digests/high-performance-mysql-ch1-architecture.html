MYSQL · CH.1 DIGEST

MySQL Architecture

In one line: MySQL isn't one program that stores data one way, it's a layered server that hands your query through connection handling, then a query brain, then a swappable storage engine, before a single byte hits disk.

The big idea

Most people think of "MySQL" as one thing: the database. This chapter's whole point is that it's really three things stacked on top of each other, and the bottom one (the part that actually stores your rows) is interchangeable per table. That's not a trivia fact, it's the reason MySQL can be fast for a tiny blog and also survive a banking workload: you don't get one storage strategy, you get to pick.

The rest of the chapter follows what happens to a query and a transaction as they move through that stack: how connections get authenticated, how locks keep concurrent writers from clobbering each other, what a transaction actually promises you (ACID), and how MySQL keeps a copy of your data flowing to other servers (replication).

Key terms

  • Thread: a lightweight worker inside the MySQL process. Each client connection gets its own thread, pulled from a pool MySQL keeps ready so it doesn't have to spin one up from scratch every time.
  • Storage engine: the component that actually stores and retrieves your rows. MySQL ships with several (InnoDB, MyISAM, and others), and you choose one per table.
  • Storage engine API: the fixed set of low-level operations (things like "begin a transaction" or "fetch the row with this primary key") the server uses to talk to whatever engine a table uses. It's why the query layer above never needs to know which engine is underneath.
  • Parse tree: the structured version of your SQL statement MySQL builds before it can plan or run anything, similar to diagramming a sentence before translating it.
  • Optimizer: the part of the server that decides how to actually execute a query: which index to use, which table to read first, and so on.
  • Lock: a flag placed on a piece of data meaning "in use, wait your turn." Locks are how a system with many simultaneous users avoids two people corrupting the same data at once.
  • Transaction: a group of statements the database treats as one all-or-nothing unit.
  • ACID: the four guarantees a real transaction system provides: atomicity, consistency, isolation, durability (each defined below with an example).
  • Isolation level: a setting that controls how much of another, still-unfinished transaction's changes your transaction is allowed to see.
  • Deadlock: when two transactions each hold a lock the other one needs, so both wait forever unless the database steps in.
  • Write-ahead logging (WAL): writing a record of a change to a sequential log file first (cheap, fast), then updating the real data file later, so a crash can't silently lose a committed change.
  • AUTOCOMMIT: a connection setting that, when on (MySQL's default), wraps every single statement in its own tiny transaction and commits it immediately.
  • MVCC (multiversion concurrency control): a technique where the engine keeps multiple versions of a row around so a reader can see a consistent snapshot without blocking a writer, and vice versa.
  • Replication: automatically and continuously copying every write from one MySQL server to one or more others.
  • Source / replica: the source is the one server allowed to accept writes; replicas copy from it and stay a little behind in real time.

How it works

The three layers a query passes through

Picture MySQL as three stacked layers. The top layer handles connections: opening a connection, authenticating you, and checking permissions. This part isn't really MySQL-specific, it's the same kind of plumbing any network client/server tool needs.

The middle layer is where MySQL's actual "brains" live: parsing your SQL, optimizing it, and running any logic that's shared across every table regardless of storage engine (built-in functions, stored procedures, triggers, views).

The bottom layer is the storage engines, the part that actually stores and fetches rows, talking to the middle layer only through the storage engine API described above. Because that API is the only contact point, the middle layer never has to special-case a particular engine, and you can put different engines on different tables in the same database.

Getting a connection: threads, auth, privileges

Every client connection gets its own thread, and that thread's queries all run on it (which in turn runs on one CPU core). MySQL keeps a small cache of ready threads so it isn't constantly creating and destroying them. Example: open three terminal windows and connect to the same server three times, and you've got three threads doing independent work.

Once connected, MySQL authenticates you using your username, the host you're connecting from, and your password (optionally with a TLS certificate). After that, it doesn't just check permissions once: every single query gets its own privilege check, for example confirming you're actually allowed to SELECT from a given table before it runs your query.

Turning SQL into a plan: parsing and the optimizer

Before MySQL can run your query, it parses it into a parse tree, then the optimizer decides how to execute it: what order to read tables in, which index (if any) to use, whether to rewrite the query into an equivalent but cheaper form. You can nudge this with optimizer hints, and you can ask MySQL to show you its plan with EXPLAIN (more on that in "Try it yourself" below).

One thing worth flagging so you don't go looking for it: older MySQL versions had a built-in query cache that stored full result sets. It became a bottleneck under real concurrency, was deprecated as of MySQL 5.7.20, and is fully gone in 8.0. If you want result caching today, that's typically handled by an external layer like memcached or Redis in front of the app, not by MySQL itself.

Concurrency control: why locks exist

Any time more than one connection might change the same data at the same time, you need a rule for who goes first. Imagine a shared grocery list app: multiple people reading the list at once is harmless, nobody's changing anything. But if one person is actively editing the quantity on "eggs" while someone else reads it, that reader could see a half-written value. So even reading needs some care when writes are happening concurrently.

The classic fix is two kinds of locks: read (shared) locks, which many clients can hold on the same thing at once because nobody's changing it, and write (exclusive) locks, which block every other read and write on that resource until released.

How much you lock at once (the granularity) is a trade-off. A table lock locks the whole table, cheap to manage but it blocks everyone else from touching any row in that table while it's held. A row lock locks just the specific row being changed, which lets unrelated rows keep getting written concurrently, at the cost of more bookkeeping (tracking who holds which row lock, for how long, and cleaning them up). This is a storage-engine decision, not a server-wide one: different engines implement locking differently, which is part of why picking the right engine per table matters.

Transactions and the ACID guarantees

A transaction bundles several statements into one all-or-nothing unit. Say an app is booking a concert ticket: (1) confirm a seat is still free, (2) mark that seat sold, (3) charge the card. If the charge fails right after step 2, you don't want a seat marked sold with nobody actually charged. Wrapping all three steps in a transaction means if any step fails, the earlier steps roll back too. In MySQL you'd write START TRANSACTION; ... COMMIT; (or ROLLBACK; to discard everything).

ACID is the four-part promise a proper transaction system makes, using that same ticket example:

  • Atomicity: all three steps happen, or none of them do. No "seat marked sold but card never charged."
  • Consistency: the database moves from one valid state to the next; a crash between steps 2 and 3 shouldn't leave a half-applied booking visible to anyone.
  • Isolation: while the booking transaction is mid-flight, another customer checking seat availability shouldn't see the seat as "sold" until the transaction actually commits (with a caveat, covered next).
  • Durability: once the transaction commits, the booking survives a crash a second later. (Durability isn't a single absolute thing in practice, more on that below.)

Isolation levels: how much of "in progress" can you see

The ANSI SQL standard defines four isolation levels, each trading some consistency for concurrency:

  • READ UNCOMMITTED: you can see another transaction's changes before it even commits (a "dirty read"). Rarely used; the risk usually isn't worth the small performance gain.
  • READ COMMITTED: you only ever see committed changes, but running the same query twice inside one transaction can return different results if someone else committed in between (a "nonrepeatable read"). This is the default in a lot of other database systems, but not MySQL.
  • REPEATABLE READ: rows you've already read will "look the same" if you read them again in the same transaction. It can still theoretically allow a "phantom read" (you select a range, someone else inserts a new row into that range, you select the same range again and see a new row that wasn't there before). This is MySQL's default, and InnoDB specifically closes the phantom-read gap at this level using MVCC and a locking technique that also locks the gaps between index entries, not just the rows.
  • SERIALIZABLE: the strictest level: it locks every row a transaction reads, so transactions effectively run one at a time on the same data. Safe, but expect more waiting and lock contention. Used rarely in practice.

Simplification: the table above is the standard-defined behavior. Real engines (InnoDB included) implement these levels with their own mechanics, and the exact guarantees can differ slightly between storage engines and even between database products, so treat this as the mental model, not the last word for a specific engine.

Deadlocks: when two transactions wait on each other forever

A deadlock happens when two transactions each hold a lock the other one needs. Picture an Orders table: Transaction A updates order #101, then tries to update order #205. Transaction B, running at the same time, updates order #205 first, then tries to update order #101. Now A is waiting on the lock B holds, and B is waiting on the lock A holds. Neither can move.

MySQL (specifically InnoDB) detects this automatically and picks a victim to roll back, generally the transaction holding fewer exclusive row locks, since it's the cheaper one to undo, and returns an error to that connection so it can retry. This is a normal, expected event in a busy transactional system, not a bug, and applications that touch the same rows from multiple places should be written to catch that error and retry.

Write-ahead logging: how commits survive a crash

Rather than updating the on-disk data file every time something changes, many storage engines change an in-memory copy first (fast) and append a record of that change to a transaction log on disk (also fast, because it's sequential writing to one spot rather than scattered writes across the data file). The actual data file gets updated later. If the server crashes after the log entry is written but before the data file catches up, the engine can replay the log on restart and recover. The cost: most storage engines using this approach end up writing every change to disk twice, once to the log and once (eventually) to the data file.

AUTOCOMMIT: is every statement its own transaction?

By default, MySQL runs with AUTOCOMMIT on, meaning a single INSERT, UPDATE, or DELETE is automatically wrapped in its own transaction and committed right away. Turn it off (SET autocommit = 0;) and you stay inside one open transaction until you explicitly COMMIT or ROLLBACK. Certain statements, notably DDL like ALTER TABLE, force an implicit commit of whatever transaction was already open, so you can't casually mix schema changes into the middle of a data transaction.

MVCC: how readers avoid blocking writers

Most of MySQL's transactional storage engines (InnoDB included) don't rely purely on locking reads. They use multiversion concurrency control (MVCC): each transaction gets a snapshot of the data as it looked at a specific point in time, so it can keep reading consistently without needing a lock, while writers keep working on the current version underneath. Practically, this means two transactions can legitimately see different data in the same table at the same moment, which feels strange the first time you encounter it. InnoDB implements this by tagging each transaction with an ID and, when a row is changed, writing an "undo record" describing how to reverse that change; older versions of a row get reconstructed by walking that undo trail. MVCC only makes sense at READ COMMITTED and REPEATABLE READ; it doesn't apply at READ UNCOMMITTED (which always reads the newest version regardless) or SERIALIZABLE (which locks everything it reads instead).

Replication: one writer, many copies

MySQL is built around accepting writes on exactly one server at a time, called the source. Every other server, a replica, watches the source's stream of changes and replays them to stay in sync, running a little behind in real time rather than instantly. The source keeps one thread per connected replica to ship it new data as writes happen. Arranged visually, a source with several replicas is usually drawn as a simple tree.

For anything you'd call production, the chapter's own rule of thumb is to run at least three more replicas beyond the source, ideally spread across different physical locations, specifically for disaster recovery. Replication has picked up a lot of sophistication over the years too (global transaction identifiers, replicating from multiple sources, parallel replay on replicas, semisynchronous replication), which this chapter only names in passing as topics the book returns to later.

Under the hood of the default engine: InnoDB

InnoDB is MySQL's default and generally recommended storage engine, built for lots of short-lived transactions that usually complete successfully rather than roll back. It stores data in a tablespace (a set of datafiles InnoDB manages internally) and organizes each table as a clustered index built on the primary key, which makes primary-key lookups fast. The trade-off: every secondary index also carries a copy of the primary key internally, so a large primary key makes every other index on that table larger too, worth keeping in mind when you pick a primary key.

A few other things changed specifically in MySQL 8.0 that this chapter calls out: table metadata moved out of separate .frm files and an information_schema-only view into a proper data dictionary backed by InnoDB itself (with an in-memory LRU cache of table/partition definitions to cut down on repeated disk reads), and DDL statements (schema changes) became atomic, meaning a schema change either fully applies or fully rolls back rather than potentially leaving things half-changed. InnoDB also gained native JSON support in 5.7, with automatic validation and storage that's more efficient than stuffing JSON into a BLOB column, plus multivalued indexes on JSON arrays as of 8.0.7 for faster lookups into array contents.

Why it matters

This chapter is the map you'll keep referring back to. When a query is slow, you're usually looking at the optimizer's choices (is it using the index you expect?) or lock contention (is something else holding a row you need?). When behavior looks "wrong" under load, like a report showing numbers that don't add up, isolation levels are usually the reason, not a bug in your code. When you're picking a storage engine for a new table, this chapter is why the answer is almost always InnoDB unless you have a specific, well-understood reason not to.

Replication design decisions here have real stakes too. GitHub's own public postmortem describes an incident on October 21–22, 2018, where a network issue briefly cut connectivity between their primary MySQL database and its replicas; their automated failover promoted a replica that hadn't yet received a few seconds' worth of writes, and reconciling that safely took over 24 hours of degraded service. Nobody designed for that gap to matter, until the day it did.

Remember this

  • MySQL is a stack: connection handling, then query brains (parser/optimizer), then a pluggable storage engine, in that order.
  • The storage engine changes how a table actually behaves (transactions or not, locking style, crash recovery); InnoDB is the safe default for nearly everything.
  • Locks trade overhead for concurrency: table locks are cheap but block everyone, row locks let more happen at once but cost more bookkeeping.
  • MySQL's default isolation level is REPEATABLE READ (not READ COMMITTED like a lot of other databases), and MVCC is what lets reads stay fast without blocking writers.
  • Replication has exactly one writer (the source) at any moment, every replica is a little behind, and that gap is exactly where real incidents happen.

Try it yourself

  • Install MySQL and inspect its engines. Install it locally (your OS package manager) or run docker run --name mysql-lab -e MYSQL_ROOT_PASSWORD=lab -p 3306:3306 -d mysql:8.0, connect with the mysql client, and run SHOW ENGINES;. Done when: you can say which engine is marked DEFAULT and name one difference between InnoDB and MyISAM using the Transactions/XA/Savepoints columns in the output.
  • Watch an uncommitted transaction stay invisible. Open two sessions to the same server. In session A: START TRANSACTION; then insert a row, but don't commit. In session B: SELECT from that table and confirm the new row isn't there. Back in A, run COMMIT;, then re-run the SELECT in B. Done when: you've captured the before/after output showing the row appear only after commit.
  • Trigger a real deadlock. Create a small table with at least two rows. In two sessions, start transactions that update those same two rows in opposite order (session A: row 1 then row 2; session B: row 2 then row 1) without committing in between. Done when: one session returns MySQL's deadlock error (error 1213, "Deadlock found when trying to get lock; try restarting transaction") and you can explain in your own words why that session was the one rolled back.
  • Reproduce the storage-engine mixing trap on purpose. Create two identical tables, one ENGINE=InnoDB and one ENGINE=MyISAM. Inside one transaction, insert a row into each table, then ROLLBACK;. Done when: you've confirmed the InnoDB insert was undone but the MyISAM row is still sitting there, which is exactly why the chapter says not to mix transactional and nontransactional engines in one transaction.
  • Watch the optimizer make a decision. Create a table with a few hundred rows and an index on one column. Run EXPLAIN SELECT * FROM yourtable WHERE indexed_col = 'x'; and then the same query against an unindexed column. Done when: you can point at the type and key columns in each output and explain which query used the index and which one had to scan every row.
  • Sketch the architecture from memory. No setup needed: on paper or in a drawing app, draw MySQL's three layers (connections, server brains, storage engines) and mark where a SELECT crosses the storage engine API boundary. Done when: your sketch has all three layers labeled and you can explain out loud why swapping a table's storage engine doesn't change the SQL you write against it.

Each of these is doable in an hour or two. If you don't want to install anything locally, a free-tier cloud MySQL instance or the Docker option above works just as well.
