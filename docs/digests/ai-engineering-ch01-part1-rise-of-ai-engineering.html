The Rise of AI Engineering

This is Part 1 of a 3-part digest on Chapter 1 of AI Engineering by Chip Huyen (pages 25–39). It covers the section "The Rise of AI Engineering": what a language model actually is, how self-supervision let language models scale into LLMs, how LLMs stretched beyond text into foundation models, and why the author lands on the term "AI engineering" instead of ML engineering, MLOps, or LLMOps. Part 2 will cover "Foundation Model Use Cases" (what these models are good and not-yet-good at), and Part 3 covers planning AI applications and the AI engineering stack.

The big idea

Here's the throughline for this whole chapter: a language model is just a system that guesses what token comes next. That idea is decades old. What changed after 2020 is scale, models trained on far more data than before. The reason they could scale is a training trick called self-supervision, which lets a model learn from raw text without anyone hand-labeling it. Once language models got big enough, and started handling more than just text, people started calling them foundation models instead. And once foundation models became something you could rent through an API instead of having to build yourself, a new job showed up: adapting those models to real problems. The author calls that job AI engineering.

Key terms

  • Token — the basic unit a language model works with. Can be a whole short word, part of a word, or a character. Example: "cooking" can split into "cook" + "ing".
  • Tokenization — the process of chopping raw text into tokens before a model can use it.
  • Vocabulary — the full set of tokens a model knows. GPT-4's vocabulary has 100,256 tokens; the smaller Mixtral 8x7B has 32,000.
  • Autoregressive language model — predicts the next token using only the tokens that came before it. This is what "language model" means almost all the time today.
  • Masked language model — predicts a missing token using context from both before and after the gap (fill-in-the-blank, not write-forward). BERT is the classic example.
  • Self-supervision — a way of training where the data supplies its own correct answers, so nothing needs to be hand-labeled. Not the same as unsupervised learning, where there are no labels at all; self-supervision still has labels, they're just pulled from the input itself.
  • Supervision — training where a human (or a labeling service) tags each example with the "right answer" first, like marking a transaction "fraud" or "not fraud".
  • Parameter — one of the internal values a model adjusts while learning (loosely, a "dial"). More parameters usually means more capacity to learn, not automatically more accuracy.
  • Foundation model — a large model, language or multi-modality, general enough to be adapted to many different tasks instead of being built for one.
  • Model as a service — using a powerful model through an API call instead of hosting and training your own.

How it works

What a language model actually does

At its core, a language model encodes statistical information about a language: how likely a given word is to show up next, given the words before it. Feed it "My favorite color is __" and a model trained on English should rate "blue" as far more likely than "car". This isn't a new idea; people noticed language has statistical patterns long before computers existed. What's new is what we can build on top of it.

A simple way to picture a language model: it's a completion machine. Give it some text, it tries to continue it. "To be or not to be" in, ", that is the question" out. Simplification: this "completion machine" framing explains raw text generation well, but it doesn't explain why a chatbot answers your question instead of just continuing your sentence with another question. That extra behavior comes from a separate training step (post-training) that's outside this section.

Tokens: the units a model actually thinks in

A model doesn't see letters or whole words, it sees tokens. A token might be a whole short word or a meaningful piece of a longer one. "cooking" can split into "cook" + "ing", and a made-up word like "chatgpting" can split into "chatgpt" + "ing". That's deliberate: splitting words into pieces lets a model keep some meaning from each piece, keep its total vocabulary far smaller than "one entry per possible word" would need, and still handle brand-new or made-up words it's never seen whole. As a rough rule of thumb, an average token is about ¾ of a word, so 100 tokens is roughly 75 words.

Two ways to predict: masked vs. autoregressive

There are two main flavors of language model, and they differ in what information they're allowed to use to predict a token. An autoregressive model only looks at what came before, and predicts the next token moving forward, one at a time. A masked model can look at what's both before and after a gap, and predicts what belongs in the middle. Autoregressive models are what you interact with when something writes text for you. Masked models (BERT is the classic example) show up more in tasks like classifying a chunk of text or spotting an error, where understanding the whole surrounding context matters more than generating new text. Unless stated otherwise, "language model" by itself usually means the autoregressive kind.

Self-supervision: how models train without a human labeler

This is the part that explains why these models could get so big. Most machine learning needs supervision: you show a model lots of examples labeled with the right answer (this transaction: fraud; this transaction: not fraud), and it learns the pattern. Labeling is slow and expensive. The chapter's own example: if it costs 5 cents to label one image, labeling a million images (roughly the size of ImageNet, the dataset behind the deep-learning model AlexNet) runs $50,000. Push that same approach to a million categories and you're at $50 million, just for labels.

Self-supervision sidesteps this. The input data supplies its own labels. Take any sentence and you already have a built-in set of practice questions: given the words so far, what's the next word? Nobody has to sit down and tag it by hand. Because usable text already exists everywhere (books, articles, comments, code), models could train on far more data than any hand-labeled dataset could realistically provide. More data plus more parameters is what turned language models into LLMs.

Worth keeping straight: self-supervised is not the same as unsupervised. In self-supervised learning, labels are inferred from the input itself (as above). In unsupervised learning, there are no labels at all, of any kind.

From language model to LLM: what changed was scale

What actually makes something an "LLM" instead of just a "language model"? Mostly size, and the book is upfront that this isn't a precise scientific line. Size is usually measured in parameters, the internal values a model adjusts while training. When OpenAI's first GPT model shipped in June 2018, its 117 million parameters counted as large. By February 2019, GPT-2 arrived with 1.5 billion parameters, and 117 million suddenly looked small. At the time this book was written, around 100 billion parameters was the rough bar for "large". Why do bigger models need more data? Because a bigger model has more capacity to learn, so it needs more examples to actually use that capacity well; train a huge model on a tiny dataset and you're mostly wasting compute a smaller model could've used to reach similar results.

Flag: those parameter-count benchmarks are already a few years old as of the book's writing, and it's mid-2026 as you're reading this. Treat "100 billion is large" as a snapshot of a moving target, not a number to memorize.

From LLM to foundation model: beyond text

Language models are stuck with text, but humans understand the world through more than just language: images, sound, and more. So the same core idea (predict the next token) got extended to other kinds of data. A multimodal model takes in more than one type of input, for example both text tokens and image tokens, and still predicts what comes next. A generative multimodal model (one that produces new content, not just classifies) is also called a large multimodal model, or LMM.

Once a model works across modalities and is genuinely general-purpose, the book's term for it is foundation model: something you build applications on top of, the way a building sits on a foundation. This book (and this digest) uses "foundation models" to mean both large language models and large multimodal models. Not every foundation model generates content, either: the chapter's example is CLIP, trained on 400 million (image, text) pairs found naturally co-occurring on the internet (a self-supervision variant the book calls natural language supervision, avoiding manual labeling entirely). CLIP doesn't generate text or images; it produces embeddings, numeric vectors that capture meaning, and those embedding models power a lot of newer generative multimodal systems under the hood.

The other shift is that foundation models are general-purpose. Older models were usually built for exactly one task; a sentiment classifier couldn't do translation and vice versa. A foundation model can do a wide range of tasks out of the box, and can be pushed further toward a specific task with prompt engineering (writing better instructions), retrieval-augmented generation or RAG (giving the model a database to pull facts from), or finetuning (further training it on examples of what you want). A task-specific model built from scratch can still win in narrow cases, and it can be smaller, faster, and cheaper to run, but adapting an existing foundation model is almost always faster to get started with.

From foundation model to AI engineering: three factors, one new job

People have shipped ML-powered products for over a decade already (fraud detection, recommendations, churn prediction), usually called ML engineering or MLOps. So why does the author give "AI engineering" its own name? Three conditions lined up at once:

  1. General-purpose capability. Because foundation models can do so many different tasks reasonably well out of the box, way more applications became possible, pulling in way more people who want to build them.
  2. A jump in investment. The success of ChatGPT triggered a wave of venture and enterprise spending on AI; the book cites a Goldman Sachs Research estimate that AI investment could approach $100 billion in the US and $200 billion globally by 2025 (a projection from the book's 2024 writing, worth treating as dated context rather than a current figure).
  3. A much lower barrier to entry. "Model as a service" means you can call a powerful model through an API instead of standing up your own training infrastructure, and you can often steer it in plain English instead of writing a training pipeline.

So why not just call it "ML engineering"? The book's answer: working with a foundation model is different enough from building your own model from scratch (you're adapting, not training from zero) that the old term undersells the difference, even though the two disciplines share a lot underneath. Why not one of the "Ops" terms (MLOps, LLMOps)? Because those emphasize operations, and the day-to-day work here is more about engineering: tweaking prompts, retrieval setups, and finetuning to get an existing model to behave the way you want. In the end, the author asked 20 people who were actually building on foundation models what they called what they did. Most said "AI engineering." That's the term the book goes with.

Why it matters

This isn't just vocabulary. It explains the shape of the job you'd be doing if you got into this field today. You're mostly not training models from scratch (that's still a small, well-funded club: big tech companies, a few governments, well-funded startups). You're adapting existing, rentable models to a specific problem, using prompt engineering, RAG, and finetuning, the three techniques this section names and the rest of the book digs into. It also explains why the field feels like it's moving so fast: when the barrier to trying something is "write a prompt" instead of "assemble a training pipeline and a GPU cluster," a lot more people can experiment.

Remember this

  • A language model just predicts the next likely token, given the tokens before it. Everything else in this chapter builds on that.
  • Tokens aren't words, they're often word-pieces, which is how a model handles text it's never seen before.
  • Self-supervision is the real unlock: the training data grades itself, so nothing needs hand-labeling, which is why models could scale up so much.
  • "LLM" isn't a precise threshold, it just means "big," and what counts as big keeps moving.
  • AI engineering means adapting a model you didn't train (prompting, retrieval, finetuning), not building one from scratch.

Try it yourself

  1. Count tokens vs. words. Use a public tokenizer tool or the tiktoken library to tokenize 3 sentences you write. Record token count vs. word count for each. Done when: you can point to one word that got split into multiple tokens, and you've checked whether the "100 tokens ≈ 75 words" rule of thumb roughly holds.
  2. Build a tiny bigram language model. In any language, take a short text file, count how often each word follows each other word, then write a function that starts from a word and keeps picking a next word from those counts. Done when: your script generates a 10-word chain from the counts, even if it reads like nonsense. That's next-token prediction at its smallest scale.
  3. Compare autoregressive vs. masked completion. Using a library like transformers, load a small autoregressive model and a small masked model, then have each complete the same sentence: one at the end, one with a gap in the middle. Done when: you have two completions side by side and can explain in your own words which model did which job.
  4. Recreate a self-supervision table by hand. Pick any short sentence and write out every (context → next token) pair it generates for training, the way this digest's diagram does. Done when: your table has one row per token (plus start/end markers) and you can explain why it needed zero human labeling.
  5. Turn a hosted model into a mini classifier. Using any model-as-a-service API or playground you have access to, prompt it to answer "spam or not spam" for 5 short test messages you write yourself. Done when: you get a clean one-word answer for all 5, and can explain why this is "completion" rather than a purpose-built classifier.
  6. Sketch the parameter-count timeline. Using the figures in this digest (117M in 2018, 1.5B in 2019, 100B+ by the book's writing), plot a simple growth chart, then add today's date as a question mark. Done when: you have a sketch or chart showing the growth curve, with a note on what "large" might mean by the time you read this.
