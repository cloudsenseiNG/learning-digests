MYSQL · CH.1 DIGEST

MySQL Architecture

In one line: MySQL splits itself into layers, a client layer, a server layer that parses and optimizes your query, and a swappable storage engine layer underneath, and that separation is what lets it juggle concurrent readers and writers safely through locking, transactions, and a clever trick called MVCC.

The big idea

MySQL isn't one monolithic block of code. It's built in layers, and the bottom layer, the part that actually stores your bytes on disk, is pluggable. You can swap it out per table without touching the SQL you write. That single design choice explains almost everything else in this chapter: why the server has to negotiate with the storage engine about locking, why transactions behave differently depending on which engine you pick, and why the book spends the rest of itself mostly talking about one specific engine, InnoDB, because that's the one almost everyone should use.

The second big idea is concurrency control: what happens when two things want to touch the same row at the same time. MySQL (through InnoDB) solves most of this with a technique called multiversion concurrency control (MVCC), which lets readers avoid locking almost entirely by handing them a consistent "snapshot" of the data instead of making them wait.

Key terms
  • Storage engine — the component responsible for actually storing and retrieving rows on disk for a given table. MySQL ships with several (InnoDB, MyISAM, and others); you pick one per table.
  • Storage engine API — the fixed set of low-level operations (like "begin a transaction" or "fetch the row with this primary key") that the server uses to talk to whichever storage engine a table uses. It's the reason the server doesn't need to know or care which engine it's talking to.
  • Thread — in MySQL, each client connection gets handed its own worker thread inside the server process; that thread runs all of that connection's queries.
  • Shared lock (read lock) — a lock that lets many clients read the same data at once, none of them block each other.
  • Exclusive lock (write lock) — a lock that only one client can hold; it blocks every other read and write on that resource until it's released.
  • Lock granularity — how much of the data a single lock covers: a whole table (coarse) or a single row (fine). Coarser locks are cheaper to manage but block more people; finer locks allow more concurrency but cost more overhead to track.
  • Transaction — a group of SQL statements that must succeed or fail together, as one indivisible unit.
  • ACID — atomicity, consistency, isolation, durability: the four properties a transaction system needs to be considered "safe" (defined one by one below).
  • Isolation level — a setting that controls how much of one transaction's in-progress changes another concurrent transaction is allowed to see.
  • Dirty read — reading data written by a transaction that hasn't committed yet (and might still roll back).
  • Nonrepeatable read — running the exact same read twice inside one transaction and getting two different answers, because another transaction committed a change in between.
  • Phantom read — running the same range query twice inside one transaction and having new rows show up the second time, because another transaction inserted them in between.
  • Deadlock — two transactions each holding a lock the other one needs, so both wait forever unless something intervenes.
  • Write-ahead logging — writing a durable record of a change to an on-disk log before (or as) you apply the change, so a crash can still be recovered from.
  • MVCC (multiversion concurrency control) — a technique where the storage engine keeps multiple versions of a row around, so a read can get a consistent snapshot without locking anything, while writers keep working.
How it works

The three layers

Picture MySQL as three stacked layers. The client layer at the top handles the stuff any network client/server tool needs: accepting connections, authenticating who you are, checking permissions. The server layer in the middle is where "MySQL's brains" live: it parses your SQL into an internal structure, decides how to run it efficiently, and hosts the functionality that's shared across every table no matter which engine stores it (built-in functions like date math, plus stored procedures, triggers, and views). The storage engine layer at the bottom is where rows actually get written to and read from disk.

The server never touches disk itself for table data. It talks to whichever storage engine a table uses through the storage engine API, a small set of low-level operations like "begin a transaction" or "give me the row with this primary key." Storage engines don't parse SQL and don't talk to each other; they just answer requests from the server layer. That's what makes them swappable: from the server's point of view, InnoDB and any other engine look the same shape, just with different tradeoffs underneath, much like different filesystems on Linux all expose "read this file, write that file" even though they store bytes on disk completely differently.

Connection handling and the optimizer

Every client connection gets its own thread on the server, and MySQL keeps a cache of ready-made threads so it doesn't have to spin one up from scratch each time you connect. Once you're connected, the server authenticates you (username, host, password, optionally an X.509 certificate over TLS) and then checks your privileges before running each query.

When a query comes in, the server parses it into an internal structure called a parse tree, then optimizes it: it might rewrite the query, decide the order to read tables in, or pick which index to use. You can nudge the optimizer with hints in the query, and you can ask it to explain its decisions, which gives you a way to figure out why a query is slow. The chapter is careful to note the optimizer doesn't care which storage engine a table uses, but the storage engine does affect the optimizer's decisions, because the optimizer asks the engine about its capabilities, the cost of certain operations, and statistics on the table before deciding how to run the query.

One detail worth flagging: older MySQL versions had a built-in query cache that could serve repeated results instantly. It became a bottleneck under high concurrency, got deprecated in MySQL 5.7.20, and was fully removed in 8.0. The chapter notes that caching results externally, in something like memcached or Redis, is still a reasonable pattern, just not a built-in MySQL feature anymore.

Concurrency control and locking

The chapter's mental model here: think of a shared spreadsheet. If only you have it open, there's no conflict. Add a coworker editing it at the same time, and now you need rules for who can read and write what, and when. Database tables have exactly the same problem.

The classic fix is two kinds of locks. A shared (read) lock lets many clients read at once, they don't block each other. An exclusive (write) lock is solo: it blocks every other read and write, because the only safe way to change something is to make sure nobody else is reading or writing it at the same instant.

How much data a single lock covers is called its granularity, and it's a real tradeoff. Lock the whole table (a table lock) and the bookkeeping is cheap, but a writer blocks every other reader and writer on that table, even if they wanted completely different rows. Lock just the row you're touching (a row lock) and far more people can work at once, but now the engine has to track who holds what, for how long, and clean it all up, which adds overhead. The chapter is explicit that this is a per-storage-engine decision: MySQL doesn't force one locking strategy on every table, different engines can (and do) pick different granularities, which is one more reason the storage engine you choose matters. Also worth knowing: table locks and row locks are implemented in different places. Table locks (with a variant called READ LOCAL that allows some concurrent writes) exist at a fairly basic level; row locks are implemented inside the storage engine itself, the server layer is mostly unaware of them.

Transactions and ACID

A transaction bundles several SQL statements into one all-or-nothing unit. The chapter's example: moving $200 from checking to savings needs at least three steps (check the balance, subtract from checking, add to savings). Wrap all three in a transaction with START TRANSACTION, and either finish with COMMIT to make it permanent, or ROLLBACK to undo everything if something goes wrong partway through, like a crash after step two but before step three.

For a transaction system to actually be trustworthy, it needs to pass what's called the ACID test:

  • Atomicity — the whole transaction happens, or none of it does. No partial transactions.
  • Consistency — the database moves from one valid state to the next; an interrupted transaction that never commits leaves no partial trace behind.
  • Isolation — one transaction's in-progress changes are usually invisible to others until it commits (the word "usually" matters, more on that next).
  • Durability — once committed, a change survives a crash. The chapter is honest that durability is fuzzy in practice: no system is ever 100% durable, since even backups exist to add a layer of durability on top of "durable" storage.

Isolation levels

Isolation controls how much of one transaction's unfinished work another transaction is allowed to see, and the ANSI SQL standard defines four levels, each trading some data safety for concurrency:

  • READ UNCOMMITTED — the loosest. Transactions can see each other's uncommitted changes (a dirty read). Rarely used in practice.
  • READ COMMITTED — you only ever see committed changes, but running the same read twice in one transaction can return different results (a nonrepeatable read) if someone else committed in between. This is the default for most database systems, but notably not for MySQL.
  • REPEATABLE READ — guarantees the same read returns the same rows within one transaction. In theory it still allows phantom reads (a repeated range query returning new rows someone else inserted), but InnoDB closes that gap using MVCC. This is MySQL's default isolation level.
  • SERIALIZABLE — the strictest. It forces transactions to run as if one at a time, by locking every row a transaction reads. Safest, but the chapter notes it's rarely used because of how much concurrency it gives up.

You can set the isolation level per session with SET SESSION TRANSACTION ISOLATION LEVEL ..., and the chapter's advice is to set whatever you use most often at the server level and only override it in specific cases.

Deadlocks

A deadlock happens when two transactions each hold a lock the other one is waiting for, so both sit there forever unless something breaks the cycle. The chapter's illustration: two transactions each update two rows, but in opposite order, transaction 1 locks row A then wants row B, transaction 2 has already locked row B and now wants row A. Neither can proceed.

InnoDB detects this automatically and picks a transaction to roll back (it tries to pick the one holding the fewest exclusive row locks, since that's roughly the cheapest one to undo), returning an error right away instead of letting the query hang forever. Some other systems instead rely on a lock wait timeout, just giving up after a set time. Because deadlocks are partly a fact of how a particular storage engine implements locking, the chapter's advice is that applications should be built to expect them and simply retry the transaction from the start when one occurs.

Transaction logging (write-ahead logging)

Rather than updating the actual table on disk on every single change, a storage engine can update its in-memory copy of the data (fast) and write a record of the change to a transaction log on disk (also fast, because appending to a log is sequential I/O in one spot, versus scattered random I/O across the table). The real table gets updated from that log later. This means most storage engines that use this technique end up writing every change to disk twice. If the server crashes after the log write but before the table itself is updated, the engine can still recover by replaying the log on restart.

AUTOCOMMIT and mixing storage engines

By default MySQL runs in AUTOCOMMIT mode: every single INSERT, UPDATE, or DELETE is implicitly its own transaction, committed immediately. Turn AUTOCOMMIT off and you stay inside one open transaction until you explicitly COMMIT or ROLLBACK, at which point MySQL starts the next one automatically. Some commands, mostly DDL statements like ALTER TABLE, force an implicit commit of whatever transaction is currently open.

Because MySQL doesn't manage transactions at the server level, each storage engine implements its own transaction support, you genuinely cannot rely on mixing engines inside one transaction. If you mix a transactional table (InnoDB) with a nontransactional one (MyISAM) in the same transaction and a rollback is needed, the nontransactional table's changes simply can't be undone. MySQL often won't warn you clearly when this happens, so the chapter's advice is blunt: don't mix storage engines within a single application's transactions.

Multiversion concurrency control (MVCC)

Most of MySQL's transactional engines don't rely purely on row locks, they add MVCC on top. The idea: instead of making a reader wait for a lock, hand it a consistent snapshot of the data as it looked at some point in time. Different transactions can legitimately see different versions of the same row at the same moment, which sounds strange at first but is exactly what lets most reads happen without acquiring any lock at all.

The chapter walks through roughly how InnoDB does this: every transaction gets a transaction ID the first time it reads data. When a row is modified, InnoDB writes an undo record (basically "here's how to undo this change") to an undo log, and points the transaction's rollback pointer at it. When another session reads that row, InnoDB compares the row's transaction ID against that session's own read view. If the current version of the row shouldn't be visible yet (its writer hasn't committed), InnoDB walks backward through the undo log until it finds a version that is visible, potentially all the way back to "this row didn't exist yet." Deletes are handled the same way, as a flag plus an undo log entry, not an immediate physical removal.

The upshot: most reads never take a lock, they just walk the version chain as needed. The cost is that the engine stores more data per row and does more bookkeeping. One more detail worth knowing: MVCC only works with the READ COMMITTED and REPEATABLE READ isolation levels. READ UNCOMMITTED always reads the newest version regardless of your transaction (so there's no "snapshot" to speak of), and SERIALIZABLE locks every row it reads, which defeats the point of not locking.

The InnoDB engine, briefly

InnoDB is MySQL's default and, per the chapter, the storage engine almost everyone should use. A few specifics the chapter calls out: it stores data in a tablespace (a set of datafiles it manages internally), it supports all four ANSI isolation levels and defaults to REPEATABLE READ, and it uses a "next-key locking" strategy that locks gaps between index entries (not just existing rows) to fully prevent phantom reads at that isolation level. Its tables are built on a clustered index (covered in more detail later in the book), which makes primary key lookups fast, but means secondary indexes also carry a copy of the primary key columns, so a large primary key makes every other index on the table larger too. InnoDB also does predictive read-ahead, keeps an adaptive hash index in memory for fast lookups, and uses an insert buffer to speed up inserts.

A few MySQL 8.0-specific changes the chapter flags: InnoDB introduced the JSON data type back in 5.7 (with validation and more efficient storage than stuffing JSON into a BLOB), and 8.0.7 added multivalued indexes on JSON arrays. Also in 8.0, MySQL moved table metadata out of separate files and into an InnoDB-backed data dictionary, which is what makes DDL (schema-changing) statements atomic, they now fully succeed or fully roll back, the same all-or-nothing guarantee transactions give you for data.

Replication, briefly

MySQL is designed to accept writes on one node (the "source") at a time. That source has a thread per replica that wakes up and streams new data whenever a write happens, and this fans out into what the chapter calls a topology tree. For anything running in production, the chapter recommends at least three replicas, ideally spread across different locations, for disaster recovery. This chapter only introduces the concept; it says replication gets a full chapter of its own later in the book.

Why it matters

Once you see MySQL as "server layer talks to a swappable storage engine through a small API," a lot of confusing behavior stops being confusing. Table exists but transactions silently don't roll back fully? Check which engine that table uses. Query runs, but locking behavior looks different from what you'd expect in another database? Locking is implemented per storage engine, not by the server. Getting more concurrent writers than you expected without explicit locking headaches? That's MVCC doing its job quietly in the background. Every later chapter about performance tuning, indexing, or backups is really just a deeper dive into decisions this chapter's architecture already set up.

Remember this
  • MySQL has three layers: client (connections, auth), server (parsing, optimizing, shared features), storage engine (actually storing rows). The storage engine layer is pluggable per table.
  • Locking is a tradeoff between granularity and overhead: table locks are cheap but block everyone, row locks allow more concurrency but cost more to track. Row locks live in the storage engine, not the server.
  • A transaction needs ACID (atomicity, consistency, isolation, durability) to actually be trustworthy. MySQL's default isolation level is REPEATABLE READ, not READ COMMITTED like many other databases.
  • MVCC lets InnoDB give most readers a consistent snapshot without locking anything, by keeping old row versions around in an undo log until nothing needs them anymore.
  • InnoDB is the engine to default to. It's transactional, uses MVCC, and (as of MySQL 8.0) even backs atomic schema changes.
Try it yourself
  1. Find out what engine a table actually uses. Install MySQL locally (or run it in Docker), create a plain table, and inspect it with SHOW CREATE TABLE or SHOW TABLE STATUS. Completion: you can point at the exact line that names the storage engine.
  2. Compare a transactional engine to a nontransactional one. Create the same simple table twice, once with ENGINE=InnoDB and once with ENGINE=MyISAM (check current MySQL docs to confirm MyISAM is still available in your version). Insert a row in a transaction, then ROLLBACK, on each table. Completion: you can explain, from what you observed, why the chapter says never to mix engines inside one transaction.
  3. Watch a row lock in action. Open two MySQL client sessions. In session 1, start a transaction and update one row without committing. In session 2, try to update that same row (it should wait), then try a different row in the same table (it should succeed immediately). Completion: you've watched, live, that a row lock only blocks the specific row, not the table.
  4. Trigger a real deadlock. Using two sessions, have each start a transaction and update two rows in opposite order (session 1 locks row A then tries row B, session 2 locks row B then tries row A). Completion: one session gets a deadlock error and rolls back automatically. Read the exact error message MySQL returns.
  5. See a nonrepeatable read happen (or not). Set one session to READ COMMITTED and another to REPEATABLE READ (via SET SESSION TRANSACTION ISOLATION LEVEL ..., check the docs for exact syntax on your version). In each, start a transaction, read a row, have a different session update and commit a change to that row, then read it again in the original transaction. Completion: you can describe the difference you saw between the two isolation levels.
  6. Toggle AUTOCOMMIT. Turn AUTOCOMMIT off for a session, run two or three statements, then roll back. Confirm nothing persisted. Turn AUTOCOMMIT back on and repeat, confirming each statement commits on its own. Completion: you can state, in your own words, what AUTOCOMMIT actually controls.
