MySQL Architecture

What a "storage engine" actually is, why MySQL is built as a stack of swappable layers, and how that one design choice explains almost everything else in this book.

The big idea

MySQL isn't one program that reads and writes a file. It's three layers stacked on top of each other: a layer that handles who's connecting, a layer that figures out how to run your query, and a layer that actually touches the data on disk. That bottom layer, the storage engine, is pluggable: MySQL ships with several of them, and you pick per table which one you want. Most of the time today that choice is InnoDB, and a huge amount of what makes MySQL behave the way it does (locking, crash recovery, whether a rolled-back transaction actually rolls back) comes down to which engine is underneath a given table.

Key terms

  • Storage engine — the component that physically stores and retrieves a table's data. InnoDB and MyISAM are two examples; they make very different tradeoffs behind the same SQL.
  • Storage engine API — the internal interface the SQL layer uses to talk to whichever engine a table uses ("begin a transaction," "fetch the row with this primary key"). It's what lets the query layer stay mostly the same no matter which engine is underneath.
  • Thread — a lightweight worker inside the MySQL server process. By default, each client connection gets its own thread to run its queries in, pulled from a cache of ready-to-use threads.
  • Transaction — a group of SQL statements that succeed or fail as a single unit. If any part can't be applied, none of it is kept.
  • ACID — atomicity, consistency, isolation, durability: the four properties a transaction system needs to be trustworthy when things crash or run at the same time.
  • Isolation level — a setting that controls how much of one transaction's in-progress work another transaction is allowed to see. Lower isolation buys more concurrency and more surprises.
  • Lock — a mechanism that stops two clients from stepping on each other's changes to the same data at the same time. A read (shared) lock lets many readers in at once; a write (exclusive) lock lets in exactly one client and blocks everyone else.
  • Lock granularity — how much data one lock covers: an entire table (table lock) or a single row (row lock).
  • Deadlock — two transactions each holding a lock the other one needs, so both wait forever unless something intervenes.
  • MVCC (multiversion concurrency control) — a technique for keeping old versions of a row around so a reader can see a consistent snapshot of the data without blocking a writer, and vice versa.
  • Replication — copying the changes made on one MySQL server to one or more other servers.
  • Write-ahead logging — writing a durable record of a change to a sequential log before the change is fully applied to the actual data files on disk, so a crash mid-write can still be recovered from.
  • Tablespace — the set of files InnoDB uses to physically hold the data for its tables.

How it works

Think about what a "storage engine" is for a second, because it's the idea the rest of the chapter hangs off of. A database server has to do two very different jobs: understand SQL (parse it, figure out the fastest way to run it, check your permissions) and actually manage bytes on disk (find the right page, lock the right row, recover cleanly after a crash). MySQL's answer is to split those jobs into separate layers, with a formal internal API between the query-handling part and the storage part. Because that boundary exists, MySQL can swap in different storage engines under the same SQL front end, the way different filesystems can sit under the same operating system. That's why understanding the architecture starts with understanding the layers.

The three layers

At the top sits the stuff that isn't really MySQL-specific: accepting network connections, authenticating who you are, and basic security. In the middle is the SQL layer: this is where query parsing, the optimizer, and built-in functions (dates, math, string handling) live, along with anything that has to work the same no matter which engine a table uses, like stored procedures, triggers, and views. At the bottom are the storage engines themselves, each responsible for storing and retrieving the actual rows. The SQL layer never touches disk directly; it always goes through the storage engine API, a set of maybe a couple dozen low-level operations like "begin a transaction" or "fetch the row with this primary key." Storage engines don't parse SQL and don't talk to each other, they just answer requests handed to them from above.

Connection management and security

By default, every client connection gets its own thread inside the server process, and that thread's queries all run on a single CPU core. The server keeps a cache of ready threads so it doesn't have to spin one up from scratch for every new connection. When a client connects, the server checks their username, host, and password (optionally over TLS with a certificate), and from then on it checks their privileges on every single query, for example whether they're allowed to SELECT from a particular table.

Parsing and query optimization

Every query gets parsed into an internal structure (a parse tree), and the optimizer then decides how to actually run it: what order to read tables in, which indexes to use, whether to rewrite the query into an equivalent but faster form. You can nudge the optimizer with hints in the query itself, and you can ask the server to explain its decisions, which is how you diagnose a slow query later. The optimizer doesn't care which storage engine a table uses, but it does ask the storage engine what it's capable of and roughly how expensive an operation will be, since different engines support different index types and have different costs for the same operation. One thing worth flagging as a simplification for anyone reading an older tutorial: older MySQL versions had a built-in query cache that could serve a repeated query straight from memory, but it became a well-known bottleneck under concurrent load. It was deprecated as of MySQL 5.7.20 and fully removed in 8.0, so if you're on a current version, don't design around it; caching frequently-served results in something like an external cache is the common substitute today.

Concurrency control: locks and granularity

Any time more than one client wants to change the same data at once, something has to referee. Picture a shared spreadsheet: if it's just you, there's no conflict, but the moment a coworker can edit it too, you need rules for who gets to change what and when. Databases solve this with locks. A read lock is shared, meaning any number of clients can hold one on the same resource at the same time, because reading doesn't change anything. A write lock is exclusive: it blocks every other read and write lock, because the only safe way to change something is to have exactly one client doing it at a time.

How much data a single lock covers is called its granularity, and it's a genuine tradeoff, not a free upgrade. Locking less (say, one row instead of a whole table) lets more clients work at once, but every lock still costs something to acquire, track, and release, so finer-grained locking means more total overhead. A table lock is the cheapest, simplest option: a write to a table takes an exclusive lock on the whole table, blocking every other reader and writer until it's done. A row lock only blocks the specific row being changed, so different clients can edit different rows of the same table at the same time, at the cost of the server having to track who holds what and clean it all up. Because MySQL's storage engines can each implement their own locking policy, this isn't a single global setting: it's part of what makes one engine different from another. Row locks in particular are implemented inside the storage engine itself, not the server, which is part of why the choice of engine matters so much for how a table behaves under concurrent writes.

Transactions and ACID

A transaction is a group of statements treated as one indivisible unit: either the whole group applies, or none of it does. The classic example is moving money between two accounts: check the balance, subtract from checking, add to savings. If the server crashes between the subtract and the add, you need a way to guarantee you don't end up with money that vanished or appeared from nowhere. You start one with START TRANSACTION, then either make it permanent with COMMIT or throw it away with ROLLBACK:

START TRANSACTION;
SELECT balance FROM checking WHERE customer_id = 10233276;
UPDATE checking SET balance = balance - 200.00 WHERE customer_id = 10233276;
UPDATE savings  SET balance = balance + 200.00 WHERE customer_id = 10233276;
COMMIT;

That alone isn't enough to be trustworthy; you also need the transaction system to pass what's called the ACID test:

  • Atomicity — the transaction is all or nothing. There's no such thing as a partially-applied transaction.
  • Consistency — the database moves from one valid state to the next; an interrupted transaction never leaves half its changes behind.
  • Isolation — other transactions usually can't see a transaction's changes until it commits (the word "usually" matters here, and the isolation level controls exactly how much they can see).
  • Durability — once committed, a transaction's changes survive a crash. Durability comes in degrees rather than being an absolute guarantee.

In InnoDB specifically, ACID transactions are one of the most mature parts of MySQL, and using them well can save you from re-implementing a lot of that safety logic yourself in application code.

Isolation levels

Isolation level controls exactly how much of one in-progress transaction another transaction is allowed to see, and it's a knob, not a fixed rule: lower isolation generally means higher concurrency and more edge cases to think about. The ANSI SQL standard defines four levels, and each storage engine is free to implement them slightly differently, so don't assume they behave identically to whatever database you've used before.

  • READ UNCOMMITTED — transactions can see other transactions' uncommitted changes (a "dirty read"). Rarely used in practice; the concurrency gain over the other levels usually isn't worth the risk.
  • READ COMMITTED — you only ever see committed changes, but running the same query twice inside one transaction can return different rows if someone else committed in between (a "nonrepeatable read").
  • REPEATABLE READ — rows you've already read look the same for the rest of your transaction, but in theory a new row can still appear in a repeated range query (a "phantom read"). InnoDB closes this particular gap with MVCC and next-key locking, which we get to below.
  • SERIALIZABLE — the strictest level: transactions are ordered so they can't conflict, effectively by locking every row a transaction reads. Strong guarantees, but expect more lock waits and timeouts.

REPEATABLE READ is MySQL's default (this is really InnoDB's default behavior, since isolation is implemented per storage engine), which is worth knowing because it's not the default in a lot of other database products, where READ COMMITTED is more common. You can change it per session with SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; or set it once at the server level if that's the behavior you want most of the time.

Deadlocks

A deadlock happens when two transactions each hold a lock the other one is waiting for, so both sit there forever unless something steps in. It's easy to trigger by accident just by updating the same two rows in opposite order in two different transactions: transaction 1 locks row A then wants row B, while transaction 2 has already locked row B and wants row A. Neither can proceed. More sophisticated engines, InnoDB included, detect this cycle and immediately fail one of the transactions with an error rather than letting it hang; InnoDB specifically rolls back whichever transaction is holding the fewest exclusive row locks, since that's the cheapest one to undo. Deadlocks aren't a sign something is broken, they're a normal fact of life in transactional systems, and the usual fix on the application side is simply to retry the failed transaction from the start.

Transaction logging (write-ahead logging)

Writing straight to the actual data files on every single change would be slow, since that means scattered, random disk I/O. Instead, a transactional storage engine changes its in-memory copy of the data (fast), then appends a record of that change to a transaction log on disk (also fast, because appending to a log is sequential I/O in one small area, not random I/O all over the place). The actual data files get updated later, in the background. This means most transactional engines end up writing every change to disk twice: once to the log, once eventually to the real data. The payoff is that if the server crashes after the log write but before the data file is updated, it can recover by replaying the log on restart. This general approach is called write-ahead logging.

AUTOCOMMIT mode

By default, every single INSERT, UPDATE, or DELETE is implicitly wrapped in its own transaction and committed right away; this is AUTOCOMMIT mode, and it's on by default. You can turn it off for a session with SET autocommit = 0; (or 1/ON to turn it back on), at which point you're always inside an open transaction until you explicitly COMMIT or ROLLBACK, and a new one starts right after. Note that a handful of commands, mostly schema-changing DDL like ALTER TABLE, will silently commit whatever transaction is currently open before they run, so check your version's docs if you're relying on a long-running transaction around DDL.

Don't mix storage engines inside one transaction

MySQL doesn't manage transactions at the server level; each storage engine implements its own transaction support. That means if you mix a transactional table (InnoDB) and a nontransactional table (MyISAM) in the same transaction, everything looks fine as long as nothing goes wrong. But if you ever need to roll back, the changes to the nontransactional table can't be undone, which leaves the database in an inconsistent state and defeats the entire point of wrapping things in a transaction. MySQL often won't warn you when this happens. The practical takeaway is simple: pick one appropriate engine per table and avoid mixing transactional and nontransactional tables in the same piece of application logic.

Multiversion concurrency control (MVCC)

Most of MySQL's transactional engines don't rely on simple row locking alone, they layer MVCC on top of it. Instead of a reader blocking behind a writer's lock, MVCC keeps a consistent snapshot of the data as it looked at some point in time, so a long-running read can proceed without holding locks, and a write doesn't have to wait on a reader either. The tradeoff is that different transactions can genuinely see different versions of the same row at the same moment, which feels strange the first time you notice it. In InnoDB specifically, every transaction gets a transaction ID the first time it reads data, and every change to a row writes an "undo record" describing how to reverse that change to an undo log; a rollback pointer chains each row version back to the one before it. When another session reads a row, InnoDB compares that row's transaction ID against what the reading session is allowed to see, and if the current version isn't visible yet, it walks backward through the undo log until it finds a version that is. This is exactly why most reads in InnoDB never need to take a lock at all: they just read the right version off this chain. MVCC only makes sense at the READ COMMITTED and REPEATABLE READ isolation levels; READ UNCOMMITTED always reads the newest version regardless of visibility, and SERIALIZABLE locks every row it reads, so neither one uses the version chain the same way.

Replication

In a common MySQL setup, one server (the source) is designed to accept writes, and it distributes those changes out to one or more replicas. The source keeps a dedicated thread per connected replica, logged in like a replication client, which wakes up whenever a write happens and streams the new data over. This "source and replica(s)" arrangement is usually drawn as a topology tree. Note this is describing the common, traditional single-writer setup the chapter focuses on; MySQL's replication features have grown well beyond this over the years (global transaction identifiers, multi-source replication, parallel replication, semisynchronous replication), so treat "one writer" as the default mental model, not an absolute rule for every possible topology.

InnoDB, the default engine

InnoDB is the default general-purpose storage engine in current MySQL, and it's the engine the rest of this book leans on almost exclusively. It was built for lots of short-lived transactions that usually succeed rather than roll back, and its combination of performance and automatic crash recovery has made it the practical default even for workloads that don't strictly need transactions. It stores its data in a set of files called a tablespace, uses MVCC for concurrency (as covered above), implements all four ANSI isolation levels, and defaults to REPEATABLE READ with a locking technique called next-key locking, which locks gaps in the index as well as the rows themselves so a phantom row can't sneak into a range you've already read. InnoDB tables are built around a clustered index on the primary key, which makes primary-key lookups fast, but every secondary index also stores the primary key's columns internally, so a large primary key quietly makes every other index on that table larger too. Beyond the core engine, a few version-specific additions are worth knowing: JSON support arrived in MySQL 5.7 with validated storage that's more efficient than stashing JSON in a BLOB, and 8.0.7 added multi-valued indexes over JSON arrays; MySQL 8.0 also moved table metadata into an InnoDB-backed data dictionary (replacing the old .frm files with .sdi files plus an in-memory LRU cache), and introduced atomic DDL, meaning a schema change either fully completes or fully rolls back rather than leaving things half-changed.

Why it matters

Nearly everything you'll tune later, indexing strategy, backup approach, how replication is configured, how a query gets optimized, depends on which storage engine is under the table you're looking at. If you've ever been confused about why a rollback "didn't work," why two sessions of the same query returned different rows, or why a replica read stale data right after a write landed on the source, the answer traces straight back to the concepts in this architecture: which engine, which isolation level, and how far behind the replica actually is. Getting this mental model right first makes every later, more specific topic (indexes, schema design, replication tuning) much easier to reason about instead of memorizing as isolated facts.

Remember this

  • MySQL is layered: connection/auth on top, the SQL layer (parser + optimizer) in the middle, storage engines at the bottom, talking through the storage engine API.
  • The storage engine is chosen per table and it's pluggable. InnoDB is the default and the one to reach for unless you have a specific reason not to.
  • Locking is a tradeoff between overhead and concurrency: table locks are cheap but block everyone, row locks let more work happen at once but cost more to manage.
  • InnoDB uses MVCC, not just plain locking, so two transactions can legitimately see two different versions of the same row at the same time.
  • In a common source-replica setup, one server takes the writes and replicas copy asynchronously, which means a replica can genuinely lag behind the source.

Try it yourself

  • See your engines. On any MySQL or MariaDB install (a free Docker container works fine: docker run --name mysqltest -e MYSQL_ROOT_PASSWORD=test -d mysql:8), run SHOW ENGINES; and SHOW VARIABLES LIKE 'default_storage_engine';. Done when you can name your server's default engine and at least two others it supports.
  • Table lock vs. row lock, live. Create two tables with the same columns, one ENGINE=InnoDB and one ENGINE=MyISAM. Open two terminal sessions (two mysql clients). In each engine's table, start a transaction in session 1 that updates one row and don't commit yet; from session 2, try to update a different row in the same table. Done when you can describe, in your own words, why MyISAM's session 2 has to wait but InnoDB's doesn't.
  • Commit vs. rollback. Pick any InnoDB table, note a value, then run START TRANSACTION;, change it, and ROLLBACK;. Confirm the value is unchanged. Repeat with COMMIT; instead and confirm the value stuck. Done when both outcomes match what you predicted before running them.
  • Watch a nonrepeatable read happen (or not). Open two sessions on an InnoDB table. In session A, run SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;, start a transaction, and SELECT a row. In session B, update and commit that same row. Back in session A, run the same SELECT again inside the still-open transaction. Done when you've seen the value change mid-transaction, then repeated the whole test with REPEATABLE READ and seen it not change.
  • Trigger a real deadlock. Using two rows in one InnoDB table, open two sessions. In session 1: START TRANSACTION; update row A, then try to update row B. In session 2 (started right after session 1 locks row A): START TRANSACTION; update row B, then try to update row A. Done when one session gets a deadlock error and you can identify, from InnoDB's message, which transaction got rolled back.
  • AUTOCOMMIT, on and off. In session 1, run SET autocommit = 0;, then insert a row without committing. From session 2, query the table and confirm the new row isn't visible yet. Back in session 1, run COMMIT;, then confirm session 2 can now see it. Done when you can explain in your own words why the row was invisible before the commit.

Go deeper

The MVCC walkthrough above is worth re-reading slowly with a pen and paper: pick a single row, sketch its undo-log chain by hand across three or four changes, and trace which version two different concurrently-running transactions would each land on. A natural next question once that clicks: if row locks and MVCC both live inside the storage engine, what actually changes for a table that uses a non-transactional engine like MyISAM instead of InnoDB, given that MyISAM has no undo log or transactions to fall back on at all?
